# hxaxd-research

一个可以直接在 Codex 中使用的本地计算机科学研究工作区，覆盖领域学习、Idea 审查、
实验、论文写作、审稿和投稿准备。

本项目不是 Codex Skill，也不需要安装或注册为 Skill。仓库中的 Markdown 文档就是公开、可修改的执行规范；用 Codex 打开仓库并引用 `@AGENTS.md` 即可开始。

## 快速开始

1. 克隆仓库，并用 Codex 打开仓库根目录。
2. 在请求中引用 `@AGENTS.md`。
3. 描述一个新任务，或指定已有的中文领域、论文目录或研究项目目录。
4. 根据 Codex 给出的影响说明，确认方向、范围、阶段跳过和首次落盘。

例如：

```text
@AGENTS.md
我想系统学习 Coding Agent Evaluation。请帮我划定范围，
生成一条阅读路线，并逐步维护领域知识地图。
```

```text
@AGENTS.md
我有一个关于长程 Agent 失败恢复的研究想法。先不要替我包装，
请检查本质假设、最相近工作和最强反例，判断是否值得立项。
```

```text
@AGENTS.md
请继续 workspace/长程智能体恢复机制。读取已有文档，
判断当前阶段、状态、已跳过项和未决确认，再开展下一步。
```

```text
@AGENTS.md
请独立审查 workspace/上下文压缩评测 中的论文草稿。
这次只做综合审稿，不要求补齐完整研究生命周期。
```

```text
@AGENTS.md
请归档并翻译 2605.22763v1.pdf。先检查现有领域目录；
如果没有明确归属，请给出建议并询问我应该放入哪个领域。
```

Codex 会读取 [`WORKFLOW.md`](WORKFLOW.md)，从已有文档判断任务主线，再读取
[`指导/使用说明.md`](指导/使用说明.md) 和对应专项指导。关键输入缺失、方向改变或到达确认点时，它会先与你对齐；确认后逐步写入真实产物，不会一次创建整套空模板。

## 工作方式

仓库区分两种工作：

- **生命周期阶段**表示任务主线发生了什么变化。推荐路线是学习规划、知识地图、
  Idea 立项、Story、实验、写作和投稿准备。
- **按需能力**像可重复调用的函数，包括文献与全文定位、论文归档与处理、Idea 自动发现、
  知识更新、绘图、翻译、修订、审稿、投稿规则、引用核验、Tex 验收和匿名检查。

阶段之间的箭头是推荐路线，不是门禁。用户可以直接进入任意可执行阶段，也可以明确跳过阶段。跳过会记录原因和证据影响，但不会伪装成完成。翻译、审稿、引用核验等能力也可以作为一次性任务独立使用。

`workspace/` 顶层保持扁平，直接放中文领域目录和中文研究项目目录：

```text
workspace/
├─ <中文领域名>/
│  ├─ 学习目标.md
│  ├─ 最终论文列表.md
│  ├─ 领域全景地图.md
│  ├─ 资源获取记录.md
│  └─ <论文中文短标题>/
│     ├─ 论文信息.md
│     ├─ 原文.pdf
│     ├─ 中文译文.pdf
│     ├─ 双语对照.pdf
│     ├─ 论文处理记录.md
│     ├─ 阅读笔记.md
│     └─ 论文评议.md
│
└─ <中文研究项目名>/
   ├─ 研究想法.md
   ├─ 叙事定稿.md
   ├─ 核心主张.md
   ├─ 文献证据库.md
   ├─ 实验记录/
   └─ 稿件与其他研究产物
```

论文目录中的文件按实际需要逐步创建，不要求一次生成全部内容。一个论文档案集中保存原始 PDF、中文 PDF、双语对照 PDF、版本元数据、记录和评议，不再使用 `论文资料/`中间层。研究项目通过路径和精确证据位置引用领域论文档案，不重复复制整套材料。

如果用户只提供一篇尚未归类的 PDF，Codex 会先检查已有领域，提出可能归属，并询问应加入哪个已有领域或创建哪个中文领域目录。领域归属不会由 Agent 静默决定。

任务状态不保存在隐藏字段或额外状态文件中，而由真实文档、用户确认、跳过记录和未决问题共同推导。

“归档并翻译”这类组合请求默认作为一个整体处理。路径、环境或正文传输条件缺失时，
任务整体阻塞；只有用户明确同意拆分，才会先落盘可执行部分。

论文处理时，PDF2zh 可以在独立进程中读取全文并生成中文与双语 PDF，但正文默认不会进入 Codex 的模型上下文。Codex 只接收文件状态、大小、哈希和错误摘要；只有你请求摘要、问答、审稿、修订或内容质量检查时，它才读取完成任务所需的局部内容。

## 实验仓库

实验代码不放入中文研究项目目录。需要实验工程时，由用户把独立代码仓库手动准备到根目录的 `labs/<实际仓库目录>/`，任务文档记录该路径、上游仓库、分支和基线 commit。

`labs/` 被本仓库忽略。Codex 不会替用户创建、初始化或拉取实验仓库，也不会规定独立仓库内部的代码、日志和结果布局；它只在研究文档中维护实验目标、证据和可追溯交接。

## 可选：PDF 翻译工具

仓库调用外部开源项目
[`PDFMathTranslate-next`](https://github.com/PDFMathTranslate/PDFMathTranslate-next)
处理 PDF，固定使用 `pdf2zh-next==2.9.0`。仓库不复制或修改其 AGPL-3.0 源码，也不再维护项目级 Python 文档处理管线。

可以让 Agent 主导安装：

```text
@AGENTS.md
请安装并验证仓库使用的 PDF2zh 环境。首次安装或下载大型模型资源前先告诉我。
```

也可以直接运行：

```powershell
.\scripts\install-pdf2zh.ps1
```

翻译一篇已归档的 PDF：

```powershell
.\scripts\translate-pdf.ps1 `
  -InputPdf "workspace\<中文领域名>\<论文中文短标题>\原文.pdf" `
  -OutputDir "workspace\<中文领域名>\<论文中文短标题>"
```

脚本默认生成并直接替换 `中文译文.pdf` 和 `双语对照.pdf`，使用 DeepSeek V4 Flash、
4 QPS 和 4 个工作线程。可通过 `-Pages`、`-Glossary`、`-Qps` 和 `-Workers` 调整。

处理前遵循 [`指导/学习/论文处理工具使用.md`](指导/学习/论文处理工具使用.md)。
翻译会把正文发送到配置的 API，执行前必须得到用户许可。

脚本依次读取当前进程、Windows 用户级和机器级的
`PDF2ZH_DEEPSEEK_API_KEY`；若未找到，再读取 `DEEPSEEK_API_KEY`。密钥只传给 PDF2zh
子进程，不写入仓库或命令输出。PDF2zh 只接受 PDF；TeX 阅读材料应先用论文项目自身工具链编译成 PDF。仓库不承诺 PDF→Markdown 或通用 Markdown、HTML、TeX 翻译。

仓库维护者可阅读 [`开发指南.md`](开发指南.md)，了解文档分层、路由设计和未来修改某类规则时应更新哪些文件。
