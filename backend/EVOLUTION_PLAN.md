## 1. 已确认的产品决策

1. 候选收录后进入文献页，但不自动下载、编译或翻译资源。
2. 四类智能体任务全部保留，不隐藏；在本轮内为它们补齐真实领域能力：
   - 文献检索
   - 元数据补全
   - 资源获取
   - 冲突分析
3. 智能体不能直接修改数据库、文件或用户筛选状态。
4. 智能体写入分为两类：
   - 暂存候选属于安全写入，可以直接执行。
   - 元数据、资源、项目判断信息和 Zotero 冲突只能形成结构化变更建议，由用户批准后交给确定性代码执行。
5. 元数据修正更新当前书目条目并保留字段级修订历史；只有预印本、会议版、期刊版等真实出版版本才创建新的书目条目。
6. 阅读数据分层：
   - 原文锚点属于具体附件和页码。
   - 阅读解释属于具体项目。
   - 可复用的事实性书目注释属于文献条目。
7. 主要阅读体验改为“原始 PDF + 结构化双语内容”；重新排版的翻译 PDF 是可选导出产物。
8. 普通论文默认把全部结构化段落放入一次大模型请求；超过安全预算、输出截断或校验失败时自动按章节降级。
9. DeepSeek 是第一版翻译提供者，但翻译接口必须可替换。
10. PDF2zh 继续作为受管理依赖受到安装和状态检测支持，但不再拥有核心翻译策略；后续主要承担兼容性输出或 PDF 导出适配。
11. TeX 存在时优先从 TeX 获取章节和段落结构；否则从 PDF 版面恢复结构。
12. 优先复用 PDF2zh/BabelDOC 已有的版面识别、阅读顺序、段落切分和可用的 OCR 中间结果，不重复建设质量更差的解析器。
13. 优先通过上游公开扩展点或稳定 Python API 注入本系统生成的整篇结构化译文，再让上游完成版面回填与 PDF 渲染。
14. 前端与后端是同等重要的交付物。所有新领域能力必须同时具备现代、清晰、可发现的前端操作界面。
15. 前端必须以触屏和平板完整使用为基线，不得把桌面鼠标悬停、右键或精细拖拽作为唯一操作方式。
16. 默认部署形态是电脑运行后端，平板通过受保护的局域网浏览器或 PWA 访问；不默认把未认证接口暴露到公网。

- 内容哈希、任务幂等、取消、失败恢复、审计事件和原子附件登记。
- Codex App Server 运行时适配器，以及按运行签发和撤销的 MCP 能力令牌。
- PDF 阅读器、候选收件箱、资源操作、任务中心和 Zotero 迁移界面。

本计划形成前完成并已提交的前置重构：

该前置重构已作为独立提交落盘；后续实现继续保持提示词组装与领域执行边界分离。

## 3. 目标工作流

```text
建立项目
  → 发起文献检索
  → 实时查看智能体检索、核验和暂存候选
  → 审阅来源证据与重复项
  → 明确收录或排除
  → 进入文献页
  → 请求元数据补全或资源查找
  → 审阅结构化变更建议
  → 后端执行已批准命令
  → 提取论文结构
  → 整篇语义翻译
  → 在结构化双语阅读器中阅读、摘录和记录
```

任何一步失败，都必须能从任务中心看到失败阶段、输入对象、可操作原因和下一步入口。

## 4. 不变的架构边界

- 数据库和文件系统只由后端领域服务访问。
- 前端、智能体、翻译提供者和 Zotero 适配器不得直接接触存储。
- 提示词只描述任务，不授予权限；实际权限来自后端签发的作用域和工具白名单。
- 用户筛选决定不能由智能体代替。
- 外部网页、PDF 文本、元数据和模型输出全部视为不可信数据。
- 所有写命令具有版本前置条件、幂等键、审计主体和关联 ID。
- 多输出操作先写入任务暂存区，完整校验后一次登记；失败不得留下部分记录。
- 不为旧接口和旧前端保留兼容层，但必须保留现有文献、项目关系、附件、哈希和筛选决定。
- 数据结构升级先迁移影子数据库并做完整性验证，再原子切换活动数据库。

## 5. 第一阶段：完成日常主工作流

本阶段每个后端能力都必须在同一交付批次中提供对应前端入口、状态反馈、错误恢复和触屏交互；不接受“接口已经完成、前端以后再补”的阶段性完成定义。

### 5.1 智能体任务入口按领域组织

保留四种任务，但不再仅靠前端硬编码名称。后端发布每种任务的：

- 标识符和用户名称。
- 所需作用域：工作区、项目、文献或 Zotero 预览。
- 是否允许网页搜索。
- 可用工具。
- 当前是否就绪及缺失原因。

任务策略：

| 任务 | 必要作用域 | 网页搜索 | 允许产生的结果 |
| --- | --- | --- | --- |
| 文献检索 | 项目 | 是 | 暂存候选 |
| 元数据补全 | 文献，必要时附带项目 | 是 | 元数据变更建议 |
| 资源获取 | 文献 | 是 | HTTPS 资源获取建议 |
| 冲突分析 | Zotero 迁移预览或文献 | 否 | 冲突处理建议 |

首页可以发起项目级检索；文献页发起元数据补全和资源获取；Zotero 冲突页发起冲突分析。全局启动器仍显示全部能力，但会先要求选择合法作用域。

### 5.2 候选审阅闭环

- 收录成功后直接进入对应文献页。
- 排除后自动选择下一条候选。
- 增加键盘操作和批量判断，但每条决定仍独立记录理由。
- 对匹配文献显示关键字段差异，而不只显示“可能重复”。
- 显示来源提供者、原始地址、抓取时间、标识符和智能体推荐理由。
- 收录匹配项时复用已有 Work，不静默制造重复书目条目。

### 5.3 项目文献判断编辑

在文献页提供项目级字段编辑：

- 状态。
- 角色。
- 项目摘要。
- 相关性。
- 主要贡献。
- 阅读重点。

`included`、`excluded` 和 `archived` 只能由用户在这个界面或候选决定命令中产生。

### 5.4 资源与任务衔接

- 资源建议批准后创建持久任务，而不是由智能体下载文件。
- 下载、编译和翻译任务完成后，文献页通过事件流刷新附件。
- 任务成功事件包含公开的结果附件 ID 和可打开的产品内链接，不暴露文件路径。
- 任务失败显示错误类别、是否可重试、建议修正入口。
- 取消后不得登记输出；提交点后取消则回读已提交结果并明确报告。

### 5.5 第一阶段验收

- 从首页完成一次检索到候选暂存。
- 从候选收录后无手工刷新进入文献。
- 在文献页完成项目字段修改并回读。
- 从文献页发起资源建议、批准、执行和打开附件。
- 浏览器端到端测试覆盖上述完整路径。
- 前端不出现没有真实后端能力的可执行按钮。
- 全部主流程在 768×1024 和 1024×1366 的触屏视口中无需鼠标完成。
- 所有重要操作具有进行中、成功、失败、取消和重试反馈。

## 6. 第二阶段：结构化变更集与智能体领域工具

逻辑上这是第二阶段；实际编码时先建立本阶段的后端基础，再完成第一阶段依赖它的前端入口。

### 6.1 新增 ChangeSet 领域

变更集与 Codex 运行时审批分离：

- 运行时审批控制智能体进程提出的危险操作。
- 变更集表示产品领域中的可审阅建议。

建议的核心实体：

#### ChangeSet

- `id`
- `kind`
- `status`: `draft`、`submitted`、`partially_applied`、`applied`、`rejected`、`stale`、`failed`
- `agent_run_id`
- `project_id`
- `item_id`
- `source_version`
- `content_hash`
- `summary`
- `created_at`
- `reviewed_at`
- `applied_at`

#### ChangeItem

- `id`
- `change_set_id`
- `operation`
- `target_type`
- `target_id`
- `base_revision`
- 类型化载荷
- 来源证据
- 建议理由
- `status`
- 校验或执行错误

数据库可以用 JSON 保存类型化载荷，但进入和离开领域层时必须经过具体 Pydantic 模型，不提供任意命令执行入口。

### 6.2 第一批变更类型

1. `metadata.patch`
   - 字段级旧值、新值和来源证据。
   - 应用时校验条目修订版本。
   - 成功后写入字段级修订历史。
2. `resource.acquire`
   - 目标文献、HTTPS 地址、资源类型、来源和建议文件名。
   - 批准后创建附件下载任务。
3. `project.insight.patch`
   - 只能建议项目摘要、相关性、贡献和阅读重点。
   - 不允许携带筛选状态修改。
4. `zotero.conflict.resolve`
   - 绑定固定的预览 ID、预览哈希和冲突 ID。
   - 执行时若预览版本变化则标记为 `stale`。

### 6.3 用户审批模型

- 前端展示每个字段或资源动作的差异与证据。
- 用户可以逐项勾选。
- 一次提交原子应用所有被选中的纯数据库变更。
- 涉及外部进程或网络的变更只原子创建持久任务，不在 HTTP 请求中执行。
- 被取消选择的项目记录为拒绝，不能被同一变更集再次偷偷执行。
- 智能体不能读取批准接口的写能力，也不能批准自己的建议。

### 6.4 MCP 领域工具

新增受作用域限制的工具：

- `propose_metadata_patch`
- `propose_resource_acquisition`
- `propose_project_insights`
- `propose_zotero_conflict_resolution`
- `get_change_set`

原有 `stage_candidate` 保持直接暂存语义。所有工具调用都记录：

- 智能体运行 ID。
- 工具名。
- 参数和结果的哈希。
- 目标对象。
- 产生的变更集或候选 ID。
- 完成、失败或拒绝状态。

### 6.5 元数据修订模型

为 BibliographicItem 增加单调递增修订号，并记录：

- 修改前后字段值。
- 修改主体：用户、Zotero 或已批准变更集。
- 来源证据。
- 关联运行、变更集或同步 ID。
- 时间戳。

书目版本关系仍由 Work 与多个 BibliographicItem 表达，不能用修订历史代替真实出版版本。

### 6.6 第二阶段失败与恢复测试

- 重复提交同一变更集不重复执行。
- 基础修订号变化时拒绝陈旧变更。
- 部分选择只应用选中项。
- 数据库事务失败时全部回滚。
- 资源任务入队失败时变更项保持可解释的失败状态。
- 服务重启后待审批变更仍可审阅。
- 智能体令牌撤销后不能再提交建议。
- 智能体不能通过变更集修改筛选状态。

## 7. 第三阶段：语义文档、整篇翻译与结构化阅读器

### 7.1 新增 Document 领域

附件是文件事实；Document 是从附件提取出的可阅读语义结构。建议实体：

#### Document

- `id`
- `item_id`
- `source_attachment_id`
- `source_sha256`
- `extractor`
- `extractor_version`
- `structure_version`
- `status`
- `language`
- 页数和块数
- 创建与完成时间

#### DocumentBlock

- `id`：稳定 ID，不使用数据库行号或模型输出位置。
- `document_id`
- `parent_id`
- `ordinal`
- `kind`: 标题、章节、段落、列表、公式、表格、图片、脚注、参考文献。
- `semantic_role`: 背景、问题、方法、证据、结果、局限、结论等。
- 原始文本和规范化哈希。
- 起止页码。
- 页面坐标和阅读顺序。
- 章节路径。

#### BlockTranslation

- `block_id`
- 目标语言。
- 译文。
- 源文本哈希。
- 提供者、模型和提示词版本。
- 翻译批次 ID。
- 校验状态。

#### Annotation

- 项目、文献、附件和块的关联。
- 页码、引用文本、坐标锚点和源哈希。
- 类型：高亮、摘录、问题、主张、方法、结果、局限。
- 用户正文与自由标签。
- 锚点失效状态。

#### ReadingState

- 项目和文献。
- 当前附件、页码和块。
- 阅读进度、书签和最近阅读时间。

### 7.2 文档提取流水线

任务类型：`document.extract`

1. 校验源附件哈希。
2. 如果存在匹配的 TeX 源码，解析章节、段落、公式、引用和图表关系。
3. 否则优先通过 PDF2zh/BabelDOC 适配器取得版面、阅读顺序、段落、坐标和可用 OCR 结果。
4. 只有上游无法提供必要中间结果时，才补充本系统自己的 PDF 提取器；不得无依据重复实现整套段落切分。
5. 生成稳定块 ID 和结构哈希。
6. 验证：
   - 块不越页或越界。
   - 阅读顺序无循环。
   - 父子关系有效。
   - 文本覆盖率达到阈值。
7. 在一次事务中登记 Document 和全部块。

第一版不自行研发另一套 OCR。固定版本的上游工具链已验证能对扫描 PDF 返回 OCR 文本与置信度，系统直接复用并记录引擎与版本；任何未验证的版面或语义能力仍明确报告为能力缺口，不生成虚假结构。

#### PDF2zh/BabelDOC 适配能力探针

实施第三阶段前先完成一个有测试证据的技术探针：

1. 检查固定依赖版本的公开 Python API、翻译器回调和中间表示，不依赖终端输出解析。
2. 分别用原生文本 PDF、复杂双栏 PDF 和扫描 PDF 验证实际能力。
3. 证明能否稳定取得：
   - 页面与坐标。
   - 阅读顺序。
   - 章节和段落边界。
   - 公式、表格、图片和脚注类型。
   - OCR 文本及置信度。
4. 禁用上游自身的零散大模型翻译，验证能否将带块 ID 的外部译文回注渲染流程。
5. 如果公开接口不足，优先实现固定版本的进程内适配器或类型化子进程桥接；不得解析 TUI、日志或临时文件名来推断业务状态。
6. 记录上游版本、适配能力和失败降级路径，并以契约测试防止升级依赖后静默失效。

### 7.3 整篇语义翻译流水线

任务类型：`document.translate`

翻译是确定性任务调度加模型提供者调用，不通过通用智能体运行。

#### 请求准备

- 只选择需要翻译的自然语言块。
- 公式、引用键、URL 和结构 ID 使用占位符保护。
- 输入包含标题、摘要、目录、全部段落、章节路径和稳定块 ID。
- 请求模型返回严格结构化结果：
  - 块 ID。
  - 译文。
  - 语义角色。
  - 全局术语表。
  - 必要的翻译警告。

#### 调度策略

1. 提供者报告可用上下文与最大输出预算。
2. 估算整篇输入和预期中文输出。
3. 安全预算内默认一次请求翻译整篇。
4. 如果超过预算，按章节切分；每一批都附带全局目录、术语表和相邻上下文。
5. 如果整篇请求出现长度截断、缺失块或结构错误，自动降级为章节批次。

#### 返回校验

- 输入块 ID 与输出块 ID 集合完全一致。
- 不允许重复、未知或缺失块。
- 公式和引用占位符必须完整返回。
- 每块译文记录对应源文本哈希。
- 全部批次成功后一次登记翻译；失败不留下半套可见译文。
- 取消、重试和服务重启后可以根据任务暂存清单恢复，不重复已验证的提供者请求。

#### 提供者接口

定义可替换的 `TranslationProvider`：

- 能力与预算查询。
- 结构化翻译请求。
- 流式进度。
- 使用量和结束原因。
- 可重试错误分类。
- 提供者请求 ID 的内部审计。

第一版实现 DeepSeek 适配器。密钥只通过受控运行环境读取，不写入数据库、任务事件或前端响应。

### 7.4 PDF2zh 与翻译 PDF

- 继续支持 PDF2zh 自动安装、版本检查和独立任务。
- 当前直接翻译整份 PDF 的实现标记为可替换适配器。
- 结构化译文是权威翻译数据。
- 如果 PDF2zh/BabelDOC 能接受外部块译文，则将其段落中间表示映射为稳定块，并将本系统优化后的整篇翻译结果按块回注，由其继续完成字体、版式和 PDF 渲染。
- 如果不能可靠注入译文，第一版不强行生成新版翻译 PDF；保留现有翻译附件，同时优先交付结构化双语阅读器。

### 7.5 结构化双语阅读器

主要界面分为：

- 原始 PDF 区域：保留原排版、页码、图表和公式。
- 文档导航：目录、章节、搜索结果、书签。
- 双语内容区：按结构块显示原文与译文，并与 PDF 页码联动。
- 阅读记录区：摘录、问题、主张、方法、结果和局限。

第一版功能：

- 点击结构块跳转 PDF 页码。
- 在 PDF 选中文字时创建带引用文本、页码和坐标的锚点。
- 原文与译文按块切换或并排显示。
- 全文搜索同时匹配原文和译文。
- 保存阅读进度与书签。
- 创建六种结构化阅读记录和自由标签。
- 译文重新生成后，利用源块 ID 和源哈希保持锚点稳定；无法重定位时明确标记失效。

暂不包含：

- 自行研发或训练 OCR 引擎；扫描件识别复用固定版本的受管理工具链。
- 原文和翻译 PDF 的像素级同步滚动。
- 自动跨论文知识图谱。
- 智能体自动生成用户笔记。
- 研究综述和写作层。

### 7.6 现代触屏与平板 UX

前端不是后端功能的薄壳，而是独立设计和验收的产品层。所有页面遵循以下要求：

#### 响应式结构

- 桌面宽屏使用多栏工作台；平板横屏使用可折叠双栏；平板竖屏切换为主内容加抽屉或底部面板。
- 阅读器中的 PDF、结构化双语内容、目录和注释面板可以切换、并排或全屏，不强迫小屏同时容纳全部区域。
- 分栏比例可以拖动，但必须同时提供点按预设，不能把精细拖拽作为唯一方式。
- 支持安全区域、虚拟键盘、浏览器工具栏高度变化和横竖屏切换。

#### 触屏交互

- 主要触控目标不小于 44×44 CSS 像素。
- 不依赖悬停显示关键按钮或解释。
- 列表滑动、长按、文本选择和上下文操作之间不得冲突。
- PDF 文本选择后显示触屏友好的浮动操作条，用于高亮、摘录和创建结构化记录。
- 所有拖拽操作都有点按替代方案。
- 危险操作使用明确确认，不使用容易误触的边缘手势。
- 支持键盘和触控笔，但二者都不是完成主流程的必要条件。

#### 视觉与状态

- 使用一致的设计令牌、间距、层级、颜色、圆角、阴影和动效节奏。
- 动效解释状态变化，不阻塞操作，并尊重减少动态效果设置。
- 每个异步操作都有局部加载状态，避免整页闪烁。
- 空状态说明下一步，而不是只显示“没有数据”。
- 错误信息提供修复动作和任务入口。
- 保证深色、浅色、柔和阅读模式以及足够对比度。

#### 丰富设置

设置页至少覆盖：

- 阅读：字体、字号、行距、内容宽度、翻页或连续滚动、单双栏和默认面板。
- 双语：原译文上下或左右布局、默认显示语言、术语高亮、段落联动。
- PDF：色彩模式、默认缩放、工具栏密度和恢复阅读位置。
- 翻译：提供者、模型、翻译风格档案、整篇/自动降级策略、术语表和重新翻译范围。
- 智能体：默认任务模型、推理强度、可用能力和上下文摘要。
- 任务：完成通知、失败通知、自动打开结果和并发限制。
- 系统：工具安装、Zotero、快照、完整性检查、局域网访问和设备配对。

设置必须保存为后端管理的单用户偏好或明确的设备本地偏好，不得把两者混在无法迁移的零散浏览器状态中。

#### 平板访问与安全

- 前端可安装为 PWA，并提供应用图标、启动页和独立窗口模式。
- 后端局域网监听必须显式启用。
- 首次平板访问通过短期配对码建立设备会话。
- 会话凭据使用安全 Cookie，不放在 URL、本地日志或任务事件中。
- 限制允许来源并防止跨站请求伪造。
- 默认只服务局域网或受控 VPN，不实现公网账户系统。
- 设备可以在设置页查看并撤销。

#### 前端验证

- 为领域组件编写交互测试，不只测试工具函数。
- 使用浏览器端到端测试覆盖候选、变更审批、任务、阅读和设置。
- 固定检查桌面、平板横屏和平板竖屏视口。
- 在触屏模拟模式下完成所有主流程。
- 对关键页面建立视觉回归截图，防止布局在功能迭代中退化。
- 验证大候选列表、长论文、虚拟键盘和网络短暂断开场景。

### 7.7 第三阶段失败与恢复测试

- 同一附件和提取器版本重复执行只产生一个 Document。
- 源附件哈希变化后旧结构不会被误用。
- 翻译缺块、重复块、公式丢失和长度截断均拒绝提交。
- 整篇翻译失败可以自动降级为章节翻译。
- 中途取消不暴露部分译文。
- 服务重启后任务可以恢复或明确失败。
- 阅读锚点在相同源哈希下稳定，在源变化时明确失效。
- 多输出登记、审计事件和使用量记录具有幂等性。

## 8. 数据库升级与现有数据保护

本轮新增结构预计需要新的数据库基线版本。执行步骤：

1. 通过公开快照服务创建迁移前快照。
2. 复制活动数据库和必要文件到影子目录。
3. 在影子数据库中建立 ChangeSet、修订、Document、翻译、注释和阅读状态结构。
4. 迁移并验证现有项目、文献、关系、附件、哈希和筛选决定数量。
5. 对全部现有附件执行路径与内容哈希核验。
6. 运行领域测试、接口测试和深度完整性检查。
7. 使用激活日志原子切换。
8. 保留带时间戳的迁移前副本。
9. 启动后再次通过公开完整性接口验证。

不得直接原地修改活动数据库，也不得通过人工 SQL 修补生产数据。

## 9. API 与前端契约

- 后端 OpenAPI 是唯一字段契约。
- 新类型集中在 `frontend/src/shared/api/contracts.ts`，并由契约测试与 OpenAPI 对照。
- 智能体提示词、内部审计载荷、密钥、文件路径和提供者会话 ID 不进入公开模型。
- 公开任务事件只返回产品语义和安全对象链接。
- 变更集、文档块、翻译和注释均使用独立资源接口，不提供通用数据库查询或任意 JSON 写接口。
- 用户偏好、设备会话和能力状态使用后端公开契约；仅瞬时视觉状态留在浏览器。
- 前端路由可以深链接到项目、文献、任务、变更集、文档块和阅读位置。

## 12. 已接受并将纳入执行的默认假设

以下内容已经按当前讨论写成默认方案；如需调整，应在实施前修改：

1. 元数据补全和资源获取任务允许网页搜索，但只能提交建议，不能直接执行外部写入或下载。
2. 默认翻译风格是忠实、克制的学术中文，术语全篇统一，不擅自总结、解释或删减；后续可以增加翻译配置档案。
3. 结构化双语阅读器优先于新版翻译 PDF；PDF 导出不阻塞第三阶段主体交付。
4. PDF2zh/BabelDOC 的段落结构和 OCR 能力先验证、后复用；如果术语“识别”实际只代表版面和文本提取，系统会如实区分。
5. 平板是受信任的浏览器客户端，后端继续运行在电脑；局域网访问采用显式开启与设备配对。
