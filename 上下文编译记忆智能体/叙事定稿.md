# 叙事定稿

## 叙事

Agent Skill 已经逐渐成为冻结模型积累和复用程序性经验的主要载体, 因而当前研究的关键问题不再只是“Skill 能不能从轨迹中学出来”。[SkillOpt](https://arxiv.org/abs/2605.23904)已经表明, 单份自然语言 Skill 可以作为冻结 Agent 的外部可训练状态, 通过受限文本编辑、验证门和跨轮更新稳定优化; [Trace2Skill](https://arxiv.org/abs/2603.25158)进一步从大量执行轨迹中归纳可迁移流程; [SkillGrad](https://arxiv.org/abs/2605.27760)则把待优化对象扩展为带元数据、主体和条件资源的多层 Skill 包, 让更新器同时决定改什么以及放在哪一层。[MUSE-Autoskill](https://arxiv.org/abs/2605.27366)又把 Skill 创建、记忆、管理、评估和改进组织成统一生命周期, 并为每个 Skill 保留跨任务经验。轨迹驱动的 Skill 形成、分层更新和长期维护都已经是现实研究对象。

研究前沿也已经超越完全无结构的提示词。[Skill-Pro](https://arxiv.org/abs/2602.01869)把 Skill 形式化为激活、执行和终止条件; [Anything2Skill](https://arxiv.org/abs/2606.09316)用结构化契约描述调用条件、禁用条件、动作、流程、约束、输出、支持证据和置信度, 并在任务时同时检索声明性证据与程序性 Skill; [SkillRAE](https://arxiv.org/abs/2605.10114)把检索到的 Skill、细粒度子单元和输出约束编译为紧凑的任务上下文; [SkVM](https://arxiv.org/abs/2604.03088)针对模型能力、Harness、软件包和工具依赖生成目标制品; [SkillSmith](https://arxiv.org/abs/2605.15215)则把给定 Skill 降级为带输入输出、依赖、来源、风险、缓存和回退路径的运行接口。因此, “结构化 Skill”“分层 Skill”“上下文编译”“依赖绑定”“运行接口”或“跨模型执行”都不能单独承担本文的新颖性。

与此同时, 代码 Agent 的项目记忆和运行基座也已经成为明确的系统组件。[PROJECTMEM](https://arxiv.org/abs/2606.12329)用带类型的事件日志保存项目中的问题、尝试、修复、决策和注记, 并在动作发生前依据失败历史给出确定性警告; [AI Harness Engineering](https://arxiv.org/abs/2605.13357)把上下文选择、项目记忆、任务状态、失败归因、验证、权限和人类干预记录都列为开发 Harness 的责任。现有系统因此并不缺少一个可以存放项目历史的位置。本文把项目记忆视为外部后端, 但不是只读后端: 存储与索引由外部系统负责, Supervisor 仍可通过受治理接口新增、修正、失效、取代或保留冲突记录。

这些进展留下的缺口不只是“流程记忆没有声明自己还需要哪些项目答案”, 还包括一个更基础的写入问题: **产生任务证据的执行上下文, 是否应该同时拥有解释证据并修改未来任务所复用状态的权限。** 现有结构可以描述 Skill 已经有什么内容、怎样被选择或怎样执行; 项目记忆可以描述已经发生什么; 优化器可以根据一次结果提出补丁。但如果执行、归因和持久写入在同一长上下文中完成, 局部事实、模型自我解释和一次性成功就可能直接进入项目记忆或跨项目流程, 而系统没有独立证据链说明为什么改、改动适用于哪些仓库、后来是否被验证或回滚。

这一缺口不是纯粹的接口美学。[SWE-Skills-Bench](https://arxiv.org/abs/2603.15401)观察到, 与目标项目不兼容的版本化指导和近似模板会造成表面锚定、概念串扰和上下文干扰, 使注入 Skill 后的结果反而退化。[SkillEvolBench](https://arxiv.org/abs/2605.24117)进一步发现, 当前 Agent 往往能在经验获取或回放任务上局部适应, 却难以在 Skill 冻结后稳定应对上下文变化、对抗性捷径和组合任务; 原始轨迹还经常优于抽象后的 Skill, 而增加更新次数或资源文件会带来 episode-specific drift 和 procedural clutter。这些结果并不直接证明本文的因果解释, 但它们排除了一个简单答案: Skill 的问题不是只要写得更多、更细或更频繁更新就会自然消失。

本文把失败解释为**作用域边界与写入治理同时缺失**。一条成功轨迹包含流程不变量、当时环境的见证值以及执行模型对结果的局部解释。例如“先确认真实测试入口再验证”是可复用程序, “运行 `pytest tests/api`”只可能是某个仓库的答案, “因为这个命令成功所以以后都这样做”则是尚未通过跨仓库验证的归纳。如果三者在同一对象和同一写入上下文中出现, 系统既可能把项目值过早绑定进流程, 也可能把一次执行解释直接提升为跨项目规则。

四种直观补救都不充分。把所有具体事实留在 Skill 中会造成冻结迁移污染; 删除所有事实会迫使执行 Agent 重复调查; 将项目记忆或整库文本直接拼进执行上下文会降低判定相关证据密度; 给每种记忆简单增加一个永久独立窗口也不能自动获得“全部上下文”或保证不过拟合。隔离窗口只能减少某类上下文干扰, 不能替代作用域、来源、版本、跨仓库复现、验证与回滚。真正需要隔离的是会改变持久状态的**治理事务**, 而不是把每一份记忆包装成一个拥有无限历史的角色。

本研究据此提出两个核心模块。

第一是**结构化记忆层**。可进化流程不再是一份可直接部署的最终文本, 而是尚未绑定项目答案的流程语义源 `S=<A,K,P,O,V>`: `A` 定义适用条件与目标, `K` 定义未绑定知识依赖及证据/时效/冲突策略, `P` 定义通用流程与分支, `O` 定义可追踪执行检查, `V` 定义验证与完成条件。具体项目答案保存在外部、项目作用域、可读写的 `M_project` 中。结构化的意义不只是组织文本, 而是让系统能够判断一条内容是流程规则、项目值、执行检查、验证条件还是目标适配, 并约束它可以被写到哪里。

第二是**带 Meta 的生命周期 Supervisor**。Supervisor 不只是任务开始时的展开器, 而是治理流程记忆从创建与版本化、选择、绑定、编译、执行观察、失败归因、项目写入、流程候选更新、跨仓库验证到提交、拒绝和回滚的完整生命周期。执行 Agent 可以消费编译结果和产生证据, 但不能直接修改任何持久记忆。所有记忆读取、绑定和写入决定都发生在与任务执行分离的事务上下文中; 每个上下文获得完整的记忆版本、作用域和来源关系, 以及做出当前决定所需的最小充分任务证据, 而不是自动继承完整对话和全部轨迹。

Supervisor 维护一份名为 `Meta` 的跨仓库时序演化记录。`Meta` 不是直接指导当前任务的另一份自由文本 Skill, 而是只追加的记忆变更账本。每个事件记录修改前流程版本、触发修改的仓库与任务情形、根因归属、候选修改、验证仓库与结果、接受/拒绝/回滚决定以及修改后版本。原始事件不能被覆盖; 对旧判断的纠正通过追加取代或回滚事件完成。`Meta` 因而回答“过去遇到什么情况时为什么修改了这份 Skill、修改在哪里验证过、后来是否出现回归”, 而不是“当前任务具体执行什么命令”。

项目记忆和流程记忆采用不同写入规则。项目中新发现或修正的事实可以在隔离写事务中新增、修正、失效、取代或保留冲突版本, 但仍停留在 `M_project`。只有当归因指向跨项目程序缺陷时, Supervisor 才生成流程候选; 它随后查询与该失败模式、流程位置和候选补丁相关的 `Meta` 事件, 检查是否只是单仓库偶然现象、是否重复提出过已失败方案、是否已有跨仓库复现, 并执行预注册的迁移或回归验证。单仓库证据默认不能直接提交到 `S`; 具体提升门槛由后续实验协议确定。

这里的独立上下文不是目的本身。绑定事务需要看见流程依赖、相关项目记忆和必要环境证据; 项目写事务需要看见当前项目的旧值、作用域、写入证据与冲突; 流程进化事务需要看见流程版本、归因证据、相关 `Meta` 事件和跨仓库验证; 执行上下文只接收编译后的流程、已采用项目值和检查项。这样得到的是“对当前记忆决定所需状态的完整可见性”, 不是对全部原始上下文的无差别占有。

执行检查仍然只承担可观察性。读取规范、请求批准、运行测试或静态检查被展开为带触发条件、预期证据和状态的检查项, 状态至少包括未触发、待完成、已完成、失败和有理由跳过。Harness 可以据此判定检查是否完成并向归因事务提供证据, 但不据此拦截每次工具调用, 也不声称获得形式执行保证。

这一方法不替代 SkillOpt、Trace2Skill 或 SkillGrad 等候选更新后端, 也不重新发明项目记忆存储、检索器、目标编译器或开发 Harness。`Meta` 作为修改日志、Supervisor 作为生命周期角色、独立上下文或项目记忆可读写, 单独看都不是本文声称首次提出的能力。本文要检验的是它们形成的共同写入边界: 结构化作用域决定信息属于哪里; 独立治理事务决定谁能写; `Meta` 保存的跨仓库证据和修改后果决定局部经验是否可以提升为流程规则。

因此, 本研究的中心判断是: **可迁移流程记忆既需要结构化的作用域边界, 也需要与执行分离、由 Meta 支持的生命周期写入治理。** 如果判断成立, 应出现四类可分离效果: 结构化表示本身减少项目值污染; 独立治理事务相对执行 Agent 直接读写提高绑定、项目记忆修正和责任路由的正确性; 在相同模型、后端、证据与预算下, 结构化 `Meta` 相对无修改历史或扁平历史减少单仓库经验误提升、重复无效补丁和跨轮回归; 两个模块共同使冻结流程在跨项目、模型与 Harness 迁移时更稳定。若隔离写事务不优于直接自修改, `Meta` 不优于普通版本日志或简单跨仓库验证, 或收益只来自更多信息、更强模型和更多调用, 则相应叙事必须被推翻或收窄。

## 论文事实

### 研究对象、任务与目标

- **研究对象**: 面向长程代码任务的可进化流程记忆, 研究结构化作用域与独立生命周期写入治理如何共同决定项目经验能否进入未来任务复用的状态。
- **研究单位**: 一份带版本的流程语义源 `S_t`、多个仓库各自的外部可读写项目记忆 `M_project,t`、跨仓库时序演化记录 `Meta_t`, 以及 Supervisor 对这些状态执行的绑定、写入、提升、验证与回滚事务。
- **进化阶段输入与输出**: 输入流程版本、绑定清单、检查项状态、验证反馈、必要轨迹切片、相关 `Meta` 事件和跨仓库验证集合; 输出项目记忆写入、流程候选版本、目标适配更新或不写回决定。候选流程通过验证后才成为 `S_{t+1}`, 无论接受、拒绝或回滚都向 `Meta` 追加事件。
- **运行阶段输入与输出**: 绑定事务输入当前任务、`S_t`、按依赖读取的 `M_project,t`、必要环境证据和目标模型/Harness 配置; 输出运行制品、完整绑定清单和检查项清单。执行 Agent 只输入编译结果, 输出任务产物、执行记录、检查项状态和验证结果, 不直接写持久记忆。
- **真实场景**: 冻结模型需要跨项目积累调查、修改和验证等程序性经验, 但每次执行又依赖代码库版本、接口、历史决策、用户授权和实时状态。
- **主要评价目标**: 在初始记忆快照、后端读写能力、信息访问、模型能力、进化轮次和总成本公平的条件下, 是否减少项目局部值污染、错误项目记忆写入、跨层误写、单仓库经验误提升、重复无效补丁和跨轮回归, 同时提高绑定证据密度、项目记忆质量、演化可追溯性与冻结迁移稳定性。
- **次要评价目标**: 分别报告各治理事务与执行 Agent 的上下文 token、调用数、端到端时延、货币成本, `Meta` 存储与查询开销、项目记忆读写放大, 以及结构与提升门造成的表达损失、漏改或更新延迟。
- **适用范围**: 主要面向存在可复用程序、可执行验证和项目局部状态的长程代码任务。纯一次性任务、事实几乎不变化的封闭任务、无法获得任何验证信号或根本不需要项目状态的任务不是主要收益场景。

### 操作性定义与状态责任

- **流程不变量**: 在预先声明的目标任务族内, 不依赖某个具体项目取值仍应成立的决策规则、步骤关系、约束或验证原则。它是否真为不变量必须通过跨项目证据检验, 不能由作者命名决定。
- **项目知识**: 只在特定项目、模块、版本、用户关系或时间范围内成立的事实或决策。其记录至少包含值、作用域、来源、观测时间或版本和有效状态。
- **外部项目记忆后端**: 由本文方法边界之外的存储、索引或项目状态系统维护, 向 Supervisor 提供项目知识及元数据的读写接口。允许的写操作至少包括新增、修正、失效、取代和保留冲突版本。实验固定后端实现、初始快照、访问权限与可用写操作, 此后允许各方法在相同能力下演化内容; 本文不把后端建设本身计作贡献。
- **知识依赖**: 流程在选择分支或执行步骤前需要满足的外部信息条件。它描述需要回答的问题、合格证据和失败策略, 不包含某个项目的预填答案。
- **过早绑定**: 在流程进化或源制品编写阶段, 将项目知识的具体值持久化为默认流程内容, 使其在目标项目尚未知时已经影响执行。
- **可信晚绑定**: 在任务与项目状态已知后, 按知识依赖声明取得、核验和裁决具体值, 并只在其有效范围内实例化流程。
- **持久记忆治理事务**: 任何能够改变项目记忆、流程语义源、目标适配或 `Meta` 的决定。事务具有明确输入证据、目标状态、允许操作、验证条件、决定和版本结果; 执行 Agent 无权直接提交这些变化。
- **独立角色上下文隔离**: 绑定、执行、归因、项目写入、流程进化和验证分别发生在独立调用中; 每个角色只获得完成其责任所需的结构化输入, 不自动继承其他角色的完整对话、轨迹或候选信息。它提供的是对当前决定所需记忆状态的完整可见性, 不是对全部原始上下文的占有, 也不单独保证解耦或不过拟合。
- **Meta 演化记录**: Supervisor 使用的跨仓库、按时间排序、只追加的流程记忆变更账本。每条事件至少包含修改前版本、仓库与任务情形、触发证据、根因归属、候选修改、验证集合与结果、接受/拒绝/回滚决定和修改后版本。它不是执行规则, 默认不进入任务执行上下文。
- **项目经验提升**: 将一个或多个项目中的观察从 `M_project` 或任务证据转化为 `S` 中跨项目规则的过程。单仓库证据默认不足以提交; 候选必须经过预注册的跨仓库复现或迁移/回归验证。具体证据门槛是实验变量, 不能在方法描述中假定有效。
- **有效上下文信噪比**: 对某个角色的正确输出具有判定相关性的输入证据量记为 `S_ctx`, 与当前角色责任和激活依赖无关的输入量记为 `N_ctx`。信号不仅包括最终采用的值, 也包括正确判断过期、冲突、缺失或越权所必需的反证与状态信息; 某条候选被拒绝不自动意味着它是噪声。实验依据预先标注的依赖—证据关系、受控干扰注入和删除消融确定相关性, 同时报告 `S_ctx/(N_ctx+ε)`、信号密度 `S_ctx/(S_ctx+N_ctx)`、绝对信号/噪声 token 与事实数以及任务结果, 避免用事后“模型是否引用”循环定义信号, 也避免仅靠删短输入抬高比例。
- **可追踪执行检查项**: 由 `O` 展开的运行记录 `o = <id, trigger, expected_evidence, status, reason>`。状态按 `未触发 -> 待完成 -> {已完成, 失败, 有理由跳过}` 转移; 跳过必须留下理由及允许跳过的规则。任务结束时, 系统可以依据已触发检查项是否已完成或被允许跳过给出“检查完成/检查未完成”的确定判断, 但该判断只作为任务状态和失败归因输入, 不构成工具调用硬门或执行正确性保证。
- **项目局部事实污染**: 项目特定值或只由单个局部 episode 支持的结论进入通用流程语义源, 且没有被表示为带来源和作用域的依赖答案。
- **跨层误写**: 失败根因属于一种状态责任, 更新却落到另一持久化层, 例如依赖版本过期却修改通用流程, 单仓库偶然失败却提升为跨项目规则, 或模型能力不足却新增项目事实。
- **冻结迁移**: 流程语义源的内容、版本和哈希以及 `Meta` 的训练截止时间保持不变; 目标项目的 `M_project` 可以在项目作用域内按相同治理规则继续读写, 实时环境输入和预先声明的目标适配可以变化。若目标任务反馈被隐式写回 `S`、用于改写训练期 `Meta` 决定或泄露给其他目标项目, 不属于冻结迁移。

| 状态对象 | 应持久化的内容 | 典型变化尺度 | 不应承担的内容 |
|---|---|---|---|
| 流程语义源 `S_t` | 适用条件、知识依赖、通用流程、执行检查、验证契约 | 跨仓库证据和验证支持后形成新版本 | 具体项目路径、当前依赖版本、一次性用户决定 |
| 外部项目记忆 `M_project,t` | 项目事实、历史决策、已尝试方案及来源、版本与有效范围 | 由治理事务随项目和时间新增、修正、失效或取代 | 被默认视为跨项目成立的程序规则; 本文内部复制一份影子长期记忆 |
| `Meta_t` | 流程修改的触发情形、根因、候选补丁、跨仓库验证与接受/拒绝/回滚结果 | 每次流程候选决定后只追加 | 当前任务的具体执行指令; 可覆盖改写的“更高层 Skill” |
| 目标适配 | 模型能力、Harness 工具语法、上下文与权限接口的降级规则 | 目标平台变化时更新 | 项目业务事实或通用流程知识 |
| 运行制品、绑定清单与检查项清单 | 当前任务需要的流程分支、绑定值、可追踪检查项及来源 | 每次任务生成, 可按有效指纹缓存 | 作为新的权威流程或项目记忆直接长期保存 |
| 轨迹与验证证据 | 实际动作、观察、检查项状态、结果和归因依据 | 每次执行产生 | 未经治理事务直接升级为任何持久状态 |

### 关键失败与技术解释

#### F1 项目局部值污染通用流程

- **发生条件**: 从一个或少数项目的成功/失败轨迹直接更新统一 Skill, 或优化器只按结果分数和文本幅度选择补丁。
- **可观察表现**: 源制品出现项目路径、版本、命令、具体模板、工具偏好或局部接口假设; 源项目效果改善, 目标项目出现锚定、错误命令、额外资源或回归。
- **当前技术解释**: 表示没有把知识需求和知识值分开, 更新器只能在自由文本或文件层级上猜测内容责任。
- **外部证据状态**: SWE-Skills-Bench 为版本失配、模板锚定和上下文干扰提供观察; SkillEvolBench 为局部适应、冻结部署不稳定和 episode-specific drift 提供观察。它们不单独证明污染是唯一原因。

#### F2 为保持通用而导致运行时重复探索

- **发生条件**: 通用 Skill 删除所有项目细节, 又没有声明执行所需的项目知识接口。
- **可观察表现**: 每次任务重复读取相同规范、搜索测试入口、探测依赖和重新询问历史决策; 外部项目记忆已经存在却未被精确消费。
- **当前技术解释**: 流程没有把外部信息需求与具体步骤连接起来, 执行器只能从任务文本临时推断检索目标。
- **替代解释**: 重复探索也可能来自外部项目记忆覆盖不足、检索器弱或执行模型不信任记忆, 实验必须分别控制。

#### F3 绑定角色与执行上下文耦合造成低信噪比和错误采用

- **发生条件**: 执行 Agent 在同一上下文中自行检索、判断和执行, 或虽然设置 Supervisor 角色, 却把完整执行对话、整库文本、全部记忆候选和无关轨迹一并交给它; 通用检索又只优化相关性, 没有依赖级来源、时效和冲突要求。
- **可观察表现**: 与激活依赖有关的证据只占输入的一小部分; 召回到“相关”事实却使用错误版本; 多个来源冲突时随机选择; 应向用户确认的决策被环境推断替代; 上下文持续增长但绑定正确率不升, 执行 Agent 还被未采用候选干扰。
- **当前技术解释**: 角色责任和信息可见范围同时缺少边界。“被召回”被误当成“可绑定”, 绑定判断又与执行目标和长历史竞争注意力。独立调用隔开证据裁决与任务执行, 依赖过滤的隔离输入进一步提高有效证据密度。完整方法同时使用两者, 但各自效果必须分别验证。
- **替代解释**: 收益可能只来自额外一次推理、输入更短或 Supervisor 模型更强, 错误采用也可能来自执行器忽略正确编译结果。因此实验必须分别测量候选召回、绑定裁决、编译后遵循和最终成功, 并做同模型、同信息、同总预算的角色与上下文消融。

#### F4 执行上下文直接写入持久记忆

- **发生条件**: 执行 Agent 在完成任务后依据同一上下文中的自我解释直接新增或改写项目记忆、Skill 或长期记忆, 没有独立事务检查目标层、旧版本、来源、冲突和写入权限。
- **可观察表现**: 错误环境观察覆盖仍有效事实; 项目值同时进入项目记忆和流程; 依赖版本变化导致通用流程反复改写; 编译器语法错误被当作流程缺陷; 能力不足被包装成更多规则。
- **当前技术解释**: 产生证据、追求当前任务成功和决定未来状态的目标在同一上下文中耦合。运行记录又没有留下“依赖怎样绑定、哪个步骤消费、哪项检查处于什么状态”的中间证据, 因而更新对象和写操作都难以审计。
- **替代解释**: 独立事务不会自动产生正确归因; 收益也可能只来自多一次模型调用。因此需要在同模型、同证据与同写权限下比较直接写入和独立治理, 并报告归因上限。

#### F5 缺少跨仓库演化记录导致流程误提升与修改振荡

- **发生条件**: 每次流程更新只查看当前任务轨迹或最终分数, 不知道相似修改以前为何提出、在哪些仓库验证、是否被拒绝或是否后来回滚。
- **可观察表现**: 单仓库成功被升格为通用规则; 相同失败补丁被重复提出; 两个局部修复来回覆盖; 已知会造成迁移回归的修改再次进入候选; 更新次数增加但冻结迁移变差。
- **当前技术解释**: 流程进化缺少跨仓库时序证据和版本后果。`Meta` 通过只追加事件保留修改的触发情形、作用域、验证与决定, 使 Supervisor 能在提交前检查复现范围、既往失败方向和回归历史。
- **替代解释**: 简单版本日志、去重缓存或当次跨仓库验证也可能足够; `Meta` 还可能积累噪声并锚定旧决定。只有它在相同证据和预算下独立降低误提升、重复补丁与回归, 才能支持其必要性。

#### 因果链与可反驳关系

```text
作用域未结构化
-> 项目值、流程规则与局部解释同形出现
-> 项目事实污染流程, 或流程为保持通用而重复探索

执行与记忆治理未分离
-> 当前任务目标、自我解释和持久写入共享上下文
-> 错误项目写入、跨层误写或单次经验直接提升

流程修改没有 Meta 时序证据
-> 不知道相似候选过去的作用域、验证、拒绝与回滚结果
-> 重复补丁、修改振荡和已知回归重新出现

三条路径共同作用
-> 局部适应被误当成跨项目程序
-> 跨轮回归和冻结迁移退化
```

本文需要分别通过表示消融、受控记忆读写、上下文噪声扰动、直接写入与治理事务对照、`Meta` 历史消融、跨仓库提升实验和冻结迁移检验这条链。若自由文本在相同条件下不产生更多污染, 直接自修改与独立治理同样可靠, 普通版本日志或无历史条件达到同等误提升与回归水平, 或迁移退化与这些中间错误无关, 则相应技术解释必须收窄。

### 核心认识与完整方法

核心认识是: **可迁移流程不是一份删去局部细节的抽象文本, 而是一段带未解析外部输入、且其持久修改必须经过独立治理的程序。** 结构化表示回答“信息属于哪里”, 生命周期 Supervisor 回答“谁能依据什么证据修改它”, `Meta` 回答“过去为什么改过、验证过什么、后来发生了什么”。三个问题分别对应作用域、权限和时序证据, 但由两个核心模块共同实现。

#### 两个核心模块与状态关系

| 核心模块 | 持久状态 | 主要责任 | 不直接承担的责任 |
|---|---|---|---|
| 模块一: 结构化记忆层 | 流程语义源 `S_t`, 外部项目记忆 `M_project,t`, 目标适配 | 区分跨项目程序、项目值、平台表达及其可写位置 | 决定某次轨迹是否足以提交修改 |
| 模块二: 带 `Meta` 的生命周期 Supervisor | 只追加演化记录 `Meta_t`, 各治理事务的清单与决定 | 创建/版本化、选择、绑定、编译、归因、项目写入、流程提升、验证、提交、拒绝与回滚 | 代替执行 Agent 完成任务; 把全部历史无差别塞进一个永久上下文 |

#### 模块一: 结构化流程记忆与可读写项目状态

流程语义源记为 `S_t = <A, K, P, O, V>`, 具体序列化不是方法身份, 语义责任固定如下:

- **A — 适用与目标契约**: 任务类别、必要输入、允许副作用、权限边界、成功目标和明确不适用条件。
- **K — 知识依赖契约**: 每项依赖的问题、类型与作用域、允许来源、证据要求、时效策略、冲突策略、缺失策略及消费位置。
- **P — 流程与分支契约**: 阶段、步骤、条件分支、输入输出关系和恢复路径; 分支条件引用依赖标识, 不直接内嵌项目值。
- **O — 执行检查契约**: 应被追踪的读取、提问、测试、静态检查、审批和工具动作, 以及其触发条件、预期证据和可判定状态; 它不规定运行时硬拦截。
- **V — 验证与完成契约**: 需求到检查的映射、回归范围、完成条件、失败条件和不可继续条件。

结构约束至少要回答: 某段内容属于哪种责任; 某个项目值从哪里进入; 哪些步骤消费它; 哪种证据允许提出修改; 修改应落到哪个状态层。它提供可检查坐标, 不自动保证分类和更新正确。

外部项目记忆保存 `m = <value, scope, provenance, observed_at, validity, authority, supersedes>`。`authority` 区分可由环境观察的事实、工具结果和只能由人类决定的约束; `supersedes` 保留修正或失效关系。允许的治理写操作包括:

- **新增**: 没有既有记录且证据满足项目作用域要求时追加新事实;
- **修正/取代**: 新证据否定旧值时追加新版本并指向被取代记录, 不静默覆盖来源;
- **失效**: 记录已过期或前提不再成立, 但保留历史值供审计;
- **保留冲突**: 权威或时效无法裁决时并存候选并标记冲突, 不强行合并;
- **不写入**: 观察不稳定、权限不足或证据不能支持持久化时保持原状态。

外部后端可以由事件日志、知识图谱、文件索引或人工记录实现。本文不改造其存储与索引, 但明确使用其读写接口。实验固定后端、初始快照、权限和可用操作, 随后允许各条件在相同能力下演化项目状态。

可追踪检查项记为 `o=<id, trigger, expected_evidence, status, reason>`。状态按 `未触发 -> 待完成 -> {已完成, 失败, 有理由跳过}` 转移; 跳过必须记录理由与允许规则。Harness 可以据此判定“检查完成/检查未完成”, 但该结果只进入任务状态与归因, 不成为工具硬门。

#### 模块二: 生命周期 Supervisor 与 Meta

`Meta_t` 是 Supervisor 的跨仓库、时序、只追加流程演化记录:

```text
Meta_t = {g_1, ..., g_t}
g_i = <event_id, time, repository_scope, task_pattern,
       S_before, trigger_evidence, attribution,
       proposed_delta, validation_set, validation_result,
       decision, S_after, supersedes_event>
```

`decision` 至少区分提出、接受、拒绝、回滚和无更新。流程首次创建被视为 `S_before=∅` 的演化事件; 回滚不是删除旧事件, 而是追加一个指向被回滚版本和原因的新事件。`Meta` 可以引用项目记忆版本和轨迹证据, 但不复制整份仓库或执行对话。它不直接成为执行提示, 也不因记录了某次成功就把候选补丁升级为流程规则。

当 Supervisor 考虑修改 `K/P/O/V` 时, 它按失败模式、流程位置、候选改动和仓库作用域查询相关 `Meta` 事件。查询结果用于回答: 相同问题是否跨仓库复现; 候选是否以前被拒绝; 既往接受版本是否在其他任务造成回归; 当前验证集合是否只覆盖源仓库; 是否存在互相冲突的演化证据。最终提交仍由预注册验证决定, `Meta` 只提供时序证据与可追溯先验, 不替代验证。

#### 独立治理上下文协议

“独立”指每项职责使用新的模型调用和上下文窗口, 不要求更强模型。完整生命周期至少区分以下上下文:

```text
X_bind     = Pack(task_normalized, S_active, Query_K(M_project), probes, T)
X_exec     = Pack(task, C, AcceptedView(B_manifest), O_ledger, tool_interfaces)
X_attr     = Pack(S_version, B_manifest, O_ledger, V_results, artifact_summary, trace_slices)
X_pwrite   = Pack(project_state_versions, attribution, write_evidence, authority, conflicts)
X_evolve   = Pack(S_version, attribution, Relevant(Meta), proposed_delta, cross_repo_evidence)
X_validate = Pack(candidate_version, preregistered_validation_set, results, regression_history)
```

- `X_bind` 只负责选择、依赖解析、项目状态读取与运行制品编译; 不继承执行 Agent 的完整对话或未命中依赖的记忆。
- `X_exec` 只包含编译后的 `C`、已采用绑定的最小视图、检查项和工具接口; 执行 Agent 没有持久写权限。
- `X_attr` 只负责从运行证据判断责任层和是否需要候选写入, 本身不提交修改。
- `X_pwrite` 只处理当前项目的新增、修正、失效、取代、冲突或不写入决定; 它能看见目标记录的版本历史, 但不能修改 `S`。
- `X_evolve` 只生成流程候选并读取相关 `Meta` 与跨仓库证据; 它不能仅以源仓库分数提交版本。
- `X_validate` 依据预注册集合决定接受或拒绝候选, 并在已部署版本出现回归时提出显式回滚; 决定完成后向 `Meta` 追加事件。

这种隔离提供的是对当前记忆决定所需状态的完整可见性, 不是对全部原始上下文的占有。所有比较条件必须使用同一模型、初始记忆、可访问证据、读写能力和输出上限; 独立调用、输入过滤、`Meta` 可见性与跨仓库验证分别消融。有效上下文信噪比、绝对上下文长度、决定正确性和最终任务效果同时报告。

#### 从创建到回滚的完整生命周期

1. **创建与版本化**: 候选流程由既有归纳或优化后端产生; Supervisor 检查结构责任, 验证后形成初始 `S_0`, 并向 `Meta` 追加创建事件。
2. **选择**: 根据规范化任务与 `A` 激活候选流程, 不把流程检索正确率与知识绑定正确率混为一个指标。
3. **绑定与编译**: `X_bind` 按 `K` 读取 `M_project`、核验缺失/过期/冲突信息、选择分支并生成 `C`、完整绑定清单和检查项清单。
4. **最小交付**: Harness 保存完整清单; 执行 Agent 只接收 `X_exec`, 不看到原始候选池和被拒绝内容。
5. **执行与观察**: 执行 Agent 完成任务, Harness 记录绑定值消费位置、检查项状态、工具结果、产物和验证结果, 不直接写持久记忆。
6. **隔离归因**: `X_attr` 判断主要问题属于项目值、知识依赖、通用流程/验证、目标适配、执行模型能力或证据不足。
7. **项目写入**: 若根因与新证据都属于当前项目, `X_pwrite` 对 `M_project` 执行新增、修正、失效、取代、保留冲突或不写入。
8. **流程候选**: 若根因可能跨项目成立, `X_evolve` 结合相关 `Meta` 事件形成 `ΔS`, 但不直接提交。
9. **跨仓库验证**: `X_validate` 在预注册仓库/任务集合上测试候选, 同时检查既往 `Meta` 中的回归模式。知识依赖、流程、检查和验证契约可以使用不同门槛, 但必须事先声明。
10. **提交、拒绝或回滚**: 通过验证的候选形成不可变的新版本 `S_{t+1}`; 未通过则拒绝; 已部署版本出现满足条件的回归时追加回滚版本。三种决定都写入 `Meta`。
11. **冻结部署**: 迁移评测固定 `S` 的版本与哈希, 只允许项目记忆继续在项目作用域受治理地读写, 并使用预注册目标适配。

#### 责任路由与提升规则

| 观察到的主要问题 | 首选目标状态 | 提交所需证据 | 禁止的捷径 |
|---|---|---|---|
| 项目值缺失、过期、错误或冲突 | `M_project` | 当前项目的来源、权限、时效与核验结果 | 把新值写进通用流程; 静默覆盖旧版本 |
| 流程未声明必要知识 | `K` 候选 | 跨仓库复现或预注册迁移验证; 相关 `Meta` 历史 | 仅添加源仓库示例后立即提交 |
| 知识充分但通用步骤/检查/验证不完整 | `P/O/V` 候选 | 多项目失败模式、候选验证和回归集合 | 用单个平台命令替代通用修正 |
| 目标模型/Harness 无法消费已绑定流程 | 目标适配 | 相同 `S` 与绑定结果下的跨目标差异 | 改写源流程迎合单一平台 |
| 模型能力不足、偶发外部错误或证据不充分 | 不写回 | 重复运行、对照执行器或明确缺证 | 把不可修复失败包装成更多规则 |
| 流程候选被接受、拒绝或回滚 | `Meta` 追加事件 | 候选来源、验证集合、结果与决定 | 删除失败历史或把 `Meta` 当作执行规则 |

#### 核心模块一的贡献关系: 结构化作用域约束

- **回应的问题**: 自由文本或一般文件层级仍允许项目值、通用程序、执行检查和平台适配共同成为无差别优化参数。
- **设计选择**: 把未绑定知识依赖设为 `S` 的一等语义, 让分支、检查和验证引用依赖; 项目答案只存在于可读写的 `M_project`, 修改保留版本与来源。
- **运行变化**: 具体项目值不能作为源流程默认答案通过结构校验; 每个候选写入必须声明目标状态和作用域; 项目纠错无需改写流程。
- **预期可观察差异**: 即使关闭生命周期 Supervisor 的额外推理, 仍减少项目常量进入流程、责任位置误写和因项目事实更新引起的流程改动。
- **区分性证据**: 比较自由文本、逐级披露文件层级、一般结构化流程和完整作用域表示; 使用相同轨迹、优化后端、写权限和预算, 分开报告项目写入正确性与流程污染。
- **当前边界**: 结构可能漏表达或错误分类, 也不能自己判断局部经验是否应提升; 若一般结构化流程达到同等效果, `K` 与分层状态的必要性被削弱。

#### 核心模块二的贡献关系: 带 Meta 的独立生命周期治理

- **回应的问题**: 结构说明“写到哪里”后, 执行上下文仍可能直接解释并提交修改; 即使使用独立角色, 没有跨仓库修改历史也会重复局部补丁或错误提升。
- **设计选择**: 执行 Agent 无持久写权限; Supervisor 用 `X_bind/X_attr/X_pwrite/X_evolve/X_validate` 治理完整生命周期, 用只追加 `Meta` 保存流程候选的触发情形、仓库范围、验证和接受/拒绝/回滚结果。
- **运行变化**: 项目事实可以持续读写, 但写入在项目事务中保留旧版本、来源和冲突; 流程候选必须离开源任务上下文, 查询相关 `Meta`, 再通过预注册跨仓库验证提交; `Meta` 不进入普通执行上下文。
- **预期可观察差异**: 独立事务减少错误绑定、错误项目写入和跨层误写; `Meta` 与提升门减少单仓库经验误提升、重复已拒绝补丁、修改振荡和已知回归复现。
- **区分性证据**: 第一组比较执行 Agent 直接写入、共享历史中的 Supervisor 提示、独立但全量上下文和独立最小充分上下文; 第二组在独立治理下比较无修改历史、仅版本号/差异日志、扁平自然语言历史和结构化 `Meta`; 第三组比较无跨仓库门、仅当次跨仓库验证和 `Meta` 加跨仓库验证。所有条件使用相同初始快照、模型、可访问证据、读写能力、验证集合上限和总预算。
- **当前边界**: 隔离本身不保证正确, `Meta` 可能产生锚定和历史噪声, 跨仓库门也可能延迟真实更新。若直接写入同样可靠、简单日志同样有效或收益只来自更多验证, 对应机制主张不成立。

#### 两个模块的联合能力: 冻结迁移

- **回应的问题**: 项目值、局部修改理由和平台表达进入核心流程后, 更换项目、模型或 Harness 往往需要重新进化。
- **联合机制**: 结构化状态把项目差异留在 `M_project`, 生命周期治理阻止未经跨仓库证据的局部补丁提交到 `S`, 目标差异留在预注册适配层。
- **预期可观察差异**: 固定 `S` 的版本和哈希后, 项目记忆仍可在各项目作用域内受治理地更新, 但无需为每个目标改核心流程; 相对源条件的性能下降、人工修订率和重新进化率更低。
- **区分性证据**: 做跨项目、跨模型、跨 Harness 单轴和组合矩阵, 同时报告治理调用、`Meta` 查询、项目读写和执行成本, 并与每目标重新进化上限比较。
- **当前边界**: 若多数目标仍需改写核心流程, 或自由文本/一般结构化基线具有同等冻结迁移, 两个模块的联合必要性不成立。

当前明确主张是: 在目前核对到的相近工作中, 尚未发现一种方法同时把**未绑定项目知识依赖、外部可读写项目状态、执行者无持久写权限、独立生命周期治理事务、跨仓库只追加 Meta 以及项目经验提升门**组织为同一个可检验的流程记忆边界。本文将分别检验结构化作用域、独立读写治理、`Meta` 时序证据和跨仓库提升规则, 再检验它们是否共同改善冻结迁移。由于这次新增的 `Meta` 主张与 Memento-Skills、MUSE-Autoskill、SkillGrad 和优化器历史最接近, 其新颖性目前只应视为待进一步核验, 不能因命名不同直接成立。

当前明确不主张: 不主张首次提出结构化 Skill、分层 Skill、Skill 优化、Skill 生命周期、项目记忆读写、版本日志、修改历史、RAG、上下文编译、依赖图、模型/Harness 编译、角色分工、失败归因或跨模型迁移; 不主张发明项目记忆存储与索引后端; 不主张每种记忆都需要永久独立窗口; 不主张独立上下文天然掌握全部信息、自动解耦或保证不过拟合; 不主张 Supervisor 的额外模型能力或额外调用本身构成贡献; 不主张 `Meta` 取代跨仓库验证; 不主张检查项提供工具硬门或形式正确性; 不在实验前声称达到 SOTA。若已有工作完整覆盖“结构化作用域—执行/写入权限分离—Meta 时序证据—跨仓库提升与回滚”, C1 和整条新颖性叙事必须推翻或收窄。

## 分点核心主张

### C1 现有路线尚未证明结构化作用域与跨仓库写入治理的完整组合

- **分点主张**: 目前核对到的代表性工作已经分别提供受控 Skill 优化、分层更新、结构化执行契约、Skill/长期记忆、项目记忆读写、上下文编译、角色分工和生命周期管理, 但尚未确认一种方法同时满足六个条件: (1) 流程源显式声明尚未知的项目知识; (2) 项目值由外部可读写状态携带来源、版本和有效性; (3) 执行 Agent 无权直接修改持久记忆; (4) 绑定、项目写入、流程进化和验证使用独立职责上下文; (5) `Meta` 跨仓库记录流程修改的触发、接受/拒绝/回滚及后续结果; (6) 这些记录与预注册跨仓库验证共同控制项目经验向流程规则提升。一般结构化、版本历史、角色命名或生命周期框架不能自动推出这六项。
- **叙事作用**: 划定两个核心模块不是已有组件简单相加的前提, 同时暴露 `Meta` 与 Memento-Skills、MUSE-Autoskill、SkillGrad 和优化器历史可能撞车的最高风险。
- **需要的证据**: 对最相近论文及公开实现建立“表示—读权限—写权限—上下文—演化历史—提升门—回滚”矩阵。检查真实持久化对象、执行 Agent 是否能直接写、修改历史是否跨仓库、是否记录拒绝和回滚、历史是否进入提交决定、跨仓库验证是否预注册。必须核验代码与真实制品, 不能只依据字段名或作者自述。
- **削弱或推翻条件**: 若已有工作功能等价地实现上述六项, 整体新颖性被推翻。若已有工作覆盖结构化作用域与独立写入但没有 `Meta`, 本文收窄到演化证据; 若覆盖 `Meta` 与提升门但没有知识依赖表示, 收窄到运行时绑定; 若 `Meta` 只是已有优化日志的改名, 不得以术语维持主张。

### C2 结构化作用域本身减少项目事实污染并支持局部纠错

- **分点主张**: 在相同初始制品、项目记忆快照、任务、轨迹、优化模型、验证数据、迭代次数和 token 预算下, 显式区分未绑定知识依赖、通用流程、项目状态、执行检查与验证契约, 能比自由文本 Skill、逐级披露目录和不含作用域语义的一般结构化流程更准确地路由候选内容: 项目值进入 `M_project`, 跨项目程序才进入 `S`。因此项目事实可以被新增、修正或失效而无需连带改写流程。
- **叙事作用**: 证明模块一对内容边界有独立贡献, 而不是所有收益都来自 Supervisor 的额外推理、`Meta` 或跨仓库验证。
- **需要的证据**: 做四级配对比较: 自由文本; 逐级披露文件层级; schema-only 流程; 完整作用域表示。保持候选生成后端、训练证据和写权限一致, 记录每个内容单元的真实责任层与路由结果。报告项目标识符/路径/版本泄漏、语义污染率、项目写入目标准确率、依赖声明完整度、跨层误写、项目事实纠错后需要改动 `S` 的比例、未见项目效果和表达能力损失。另在关闭独立治理与 `Meta` 的条件下测试, 隔离模块一的作用。
- **削弱或推翻条件**: 若 schema-only 或自由文本达到同等污染率、写入路由和迁移效果, 绑定感知语义的必要性被削弱; 若优势来自人工标注目标层、额外 token 或定制优化器, 比较无效; 若结构阻止有效程序表达或使项目纠错显著延迟, 主张必须报告代价并收窄。

### C3 独立生命周期事务提高项目记忆读写与责任路由正确性

- **分点主张**: 当任务需要读取或修正项目局部状态时, 执行 Agent 无持久写权限, 由独立 Supervisor 在职责过滤的上下文中完成绑定、归因和 `M_project` 写事务, 相比执行 Agent 在原上下文中直接读写、只有角色提示但共享历史、或独立调用却复制全量历史, 能提高判定相关证据密度, 减少错误事实采用、静默覆盖、冲突丢失和跨层误写。该主张不声称隔离自动防止过拟合, 只检验权限与上下文边界对单次读写决定的作用。
- **叙事作用**: 证明模块二首先改变“谁能写”和“写决定看什么”, 并确认外部项目记忆是受治理的可读写状态, 不是只读检索库。
- **需要的证据**: 固定后端实现、初始快照和读写能力, 构造正确、缺失、过期、冲突、越权、错误观察、需要失效、需要取代及应不写入的项目状态。比较四个条件: 执行 Agent 直接读写; 同一历史中的 Supervisor 角色提示; 独立 Supervisor 加全量上下文; 独立 Supervisor 加最小充分上下文。所有条件使用同一模型、证据、工具、输出上限和总预算。测量绑定正确率、不可信值拒绝率、项目写入精确率/召回率、正确操作类型、旧版本保留、冲突保留、静默覆盖率、目标层准确率、应不写入校准、上下文信噪比、任务成功、调用数、时延与成本; 加入 oracle 归因和 oracle 写入上限。
- **削弱或推翻条件**: 若独立事务不优于直接写入, 权限分离主张不成立; 若最小充分上下文不优于独立全量上下文, 信噪比机制被削弱; 若 Supervisor 获得基线不可见证据、更强模型或更高写权限, 比较无效; 若隔离导致关键反证丢失并增加错误写入, 方法需收窄或重新设计。

### C4 Meta 与跨仓库提升门减少流程误提升和演化回归

- **分点主张**: 当多个仓库按时间顺序产生局部成功、真实共性、相互冲突证据和后续回归时, 让流程进化事务读取结构化 `Meta` 并经过预注册跨仓库验证, 相比无修改历史、仅版本号/差异日志或扁平自然语言历史, 能更准确地判断候选应接受、拒绝、延后还是回滚, 从而减少单仓库经验误提升、重复已拒绝补丁、修改振荡和已知回归重新引入。
- **叙事作用**: 承担“之前在什么情况下为什么修改 Skill”这一跨仓库时序记忆的研究责任。删除它后, Supervisor 只是一次性的隔离写入器, 不能利用流程自身的演化后果治理完整生命周期。
- **需要的证据**: 构造跨仓库时序任务流, 同时包含只在单仓库成立的模式、晚到的反证、真正跨仓库复现的流程缺陷、重复候选、先成功后回归和应回滚版本。做二乘二主消融: 修改历史为无/简单日志或结构化 `Meta`, 提升规则为无跨仓库门或预注册跨仓库验证; 再加入扁平自然语言历史和 oracle 历史选择。所有条件访问相同过去证据与验证集合上限, 不允许 `Meta` 条件提前看到未来结果。测量错误提升率、正确提升召回、提升延迟、重复候选率、振荡次数、已拒绝补丁重提率、回滚识别、跨仓库回归、冻结迁移、`Meta` 检索精确率/召回率、存储和 token 成本。
- **削弱或推翻条件**: 若简单版本日志、当次跨仓库验证或无历史条件达到同等误提升与回归水平, `Meta` 的必要性不成立; 若优势来自额外验证数量或未来信息泄漏, 结果无效; 若 `Meta` 锚定旧决定、压制真实新规律并显著降低正确提升召回, 主张必须按更新延迟和适用任务序列收窄。

### C5 结构化作用域与 Meta 治理共同改善冻结迁移

- **分点主张**: 在源任务序列完成受治理进化后, 将项目值排除在 `S` 之外, 把项目差异留给可读写 `M_project`, 把未经跨仓库验证的修改留在候选与 `Meta` 中, 并把模型/Harness 差异留给目标适配, 应使同一份哈希不变的流程版本更稳定地迁移到未见项目、不同模型和不同 Harness。冻结期间项目记忆仍可在项目作用域内新增、修正和失效, 但目标反馈不能修改核心流程。
- **叙事作用**: 把模块一的内容边界和模块二的演化治理转化为最终能力边界; 删除它后, 论文只能证明记忆写入更可审计, 不能证明流程更可复用。
- **需要的证据**: 固定 `S` 版本、哈希、训练序列和 `Meta` 截止时间, 禁止目标反馈修改核心流程; 构造跨项目、跨模型、跨 Harness 的单轴与组合迁移矩阵。允许各目标项目的 `M_project` 在相同治理规则下读写, 并使用预注册目标适配。比较无 Skill、自由文本 Skill、一般结构化 Skill、只有独立治理、只有 `Meta`/提升门、完整方法以及每目标重新进化上限。报告成功率、相对源条件退化、人工修订/重新进化比例、项目记忆写入质量、错误提升历史、各治理事务和执行成本、失败类型及收益随目标距离的变化。
- **削弱或推翻条件**: 若冻结源在多数目标上仍需修改核心流程, 流程可迁移性主张不成立; 若自由文本或一般结构化基线同样稳定, 联合方法没有必要; 若项目反馈被隐式写回 `S` 或 `Meta` 使用了目标未来信息, 不能算冻结证据; 若收益只来自更多跨仓库验证或更强编译模型, 应归因于相应资源而非本文边界。
