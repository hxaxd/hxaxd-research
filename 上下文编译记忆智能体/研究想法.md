# 研究想法

## idea

本研究关注的不是如何再设计一种对 Skill 进行增删改写的优化算法, 而是可进化流程记忆的**内容边界与写入边界**。长程代码任务同时依赖两类性质不同的信息: 一类是可以跨任务、跨项目复用的程序性知识, 例如如何调查代码库、定位根因、控制修改范围和完成验证; 另一类是只在当前项目、版本或用户关系中成立的局部知识, 例如模块边界、依赖版本、接口约定、测试入口、历史决策、用户偏好和实时环境状态。前者可以成为流程不变量, 后者应保留在项目作用域。但仅把两者存进不同文件还不够: 系统还必须决定谁有权读取、修改和提升这些状态, 以及一次局部经验何时才足以改变跨项目流程。

现有工作已经分别证明了这条研究链上的多个环节可以成立。[SkillOpt](https://arxiv.org/abs/2605.23904)将单份自然语言 Skill 视为冻结 Agent 的外部可训练状态, 用受限编辑和验证门稳定优化; [Trace2Skill](https://arxiv.org/abs/2603.25158)从执行轨迹归纳可迁移流程; [SkillGrad](https://arxiv.org/abs/2605.27760)进一步对多层 Skill 包执行分层更新。[Skill-Pro](https://arxiv.org/abs/2602.01869)、[Anything2Skill](https://arxiv.org/abs/2606.09316)和[SkillSmith](https://arxiv.org/abs/2605.15215)分别把激活、执行、终止, 结构化契约与证据, 以及输入输出、依赖和运行边界显式化。[MUSE-Autoskill](https://arxiv.org/abs/2605.27366)还把 Skill 创建、记忆、管理、评估和改进组织为统一生命周期, 同时维护 Skill 级记忆与长期记忆。因此, “结构化 Skill”“分层更新”“项目记忆”“上下文编译”“角色分工”或“Skill 生命周期”本身都不足以单独构成本文的研究增量。

更直接的经验信号来自两个近年的诊断结果。[SWE-Skills-Bench](https://arxiv.org/abs/2603.15401)发现, 与项目上下文不兼容的版本化指导和具体模板可能使原本能完成的代码任务退化; [SkillEvolBench](https://arxiv.org/abs/2605.24117)则发现, 当前 Agent 往往能在获取或回放任务上形成局部适应, 却难以在冻结部署后稳定复用, 原始轨迹还可能优于抽象后的 Skill, 而更多更新或更大的资源包会引入 episode-specific drift 和 procedural clutter。它们没有证明本文提出的技术原因, 但共同说明: 轨迹中“出现过且有用”的内容不等于“应当作为跨项目流程持久化”的内容, 将更多经验写入 Skill 也不等于形成了更好的程序性抽象。

本研究把这个缺口表述为两层责任同时缺失。第一层是**记忆作用域缺失**: 流程没有显式声明自己仍需要哪些项目答案, 项目事实因而可以被误写进通用步骤。第二层是**记忆治理事务缺失**: 任务执行、失败解释和持久记忆修改发生在同一上下文中, 执行 Agent 可以依据单次经历直接改写未来任务会复用的状态。前者造成值与程序耦合, 后者造成执行与学习耦合。即使引入独立角色, 如果它没有版本、来源、跨仓库证据和历史修改结果的可追踪视图, 也只是把同样的局部判断移到了另一个提示词中。

需要特别澄清的是, 本文不再主张“每种记忆天然需要一个永久独立上下文, 因而系统就能把握全部上下文或保证不过拟合”。独立上下文只是一种治理手段: 所有决定持久记忆如何读取、绑定、写入、提升、回滚的事务与任务执行上下文分离, 每次事务获得完整的**记忆相关状态**和最小充分的任务证据, 而不是完整对话、整库文本或全部历史。上下文隔离可以降低无关执行历史的干扰, 但是否过拟合最终取决于作用域规则、跨仓库证据、验证和回滚, 不能由窗口边界单独保证。

完整方法只有两个核心模块。

第一, **结构化记忆层**。其中流程语义源 `S` 保存跨项目程序, 外部项目记忆 `M_project` 保存项目局部状态, 两者都具有明确的读写责任。流程语义源是一份尚未填入具体项目答案的上下文汇编规范:

1. **适用与目标契约 `A`**: 说明流程面对什么任务、接受什么输入、应满足什么结果和权限边界。
2. **知识依赖契约 `K`**: 声明执行前必须知道什么, 知识属于哪个作用域, 允许使用哪些来源, 需要怎样的证据与时效, 缺失、过期或冲突时如何处理。
3. **流程与分支契约 `P`**: 保存跨项目成立的阶段、条件分支以及数据和控制依赖, 不预填某个项目的具体值。
4. **执行检查契约 `O`**: 将读取、提问、测试、静态检查、请求批准或工具动作表示为带触发条件、预期证据和完成状态的检查项, 但不形成运行时硬拦截。
5. **验证与完成契约 `V`**: 将完成条件映射到证据, 区分任务产物、回归检查、失败恢复和不可继续条件。

外部项目记忆不是只读检索库。它通过受治理的读写接口保存带值、作用域、来源、观测时间、有效状态和决策权限的项目知识。Supervisor 可以在任务前读取和核验它, 也可以在任务后依据证据新增事实、修正旧值、将记录标为失效、以新版本取代旧版本, 或在证据冲突时并存多个候选。外部只表示存储与索引责任不由本文重新实现, 不表示内容冻结。实验固定后端能力、初始快照和访问权限, 但允许所有条件在相同规则下读写。

第二, **带 Meta 的生命周期 Supervisor**。Supervisor 不只在任务前展开 Skill, 而是治理流程记忆从创建、选择、绑定、编译、执行观察、失败归因、候选更新、跨仓库验证到提交、拒绝和回滚的完整生命周期。它拥有一份名为 `Meta` 的跨仓库时序演化记录。`Meta` 不是另一个可直接执行的自由文本 Skill, 而是只追加的记忆变更账本:

```text
流程语义源 S_t = <A, K, P, O, V>
项目记忆 M_project,t = 外部、项目作用域、可读写状态
Meta_t = {g_1, ..., g_t}
g_i = <时间, 仓库/任务作用域, 修改前版本, 触发证据, 根因归属,
       候选修改, 验证集合与结果, 接受/拒绝/回滚决定, 修改后版本>
绑定结果 B = Resolve(K, M_project,t, 当前环境 E)
运行制品 C = Lower(S_t, B, 目标配置 T)
```

`Meta` 回答的不是“当前任务怎么做”, 而是“过去在什么情形下为什么想改这份流程记忆、改了哪里、在哪些仓库验证过、后来是否造成回归”。原始事件不被覆盖; 纠错通过追加新的取代或回滚事件完成。它默认不进入每次执行上下文, 只在归因、流程更新和版本治理时按候选失败模式查询。这样可以检查一次修改是否只由单仓库偶然现象触发, 是否重复提出过已被否决的补丁, 以及相同失败是否已经跨仓库复现。

Supervisor 的关键权限边界是: 执行 Agent 可以消费编译后的运行制品并产生证据, 但不能直接修改任何持久记忆。任务前的绑定事务只读取流程语义源、相关项目记忆和必要环境证据; 任务后的归因事务只读取绑定清单、检查项状态、验证结果和必要轨迹切片; 项目记忆写事务只处理项目作用域证据; 流程进化事务则额外读取相关 `Meta` 事件和跨仓库验证结果。每个事务使用新的角色上下文, 但共享可追踪的版本标识、证据引用和决策记录。

写入规则决定局部经验能否升级。单个仓库中新发现或修正的事实默认写入 `M_project`; 一次失败若暴露知识依赖遗漏, 可以提出对 `K` 的候选修改, 但不能仅因源仓库分数改善就提交; 对 `P/O/V` 的跨项目流程修改需要预先定义的跨仓库复现或迁移验证; 平台差异写入目标适配; 证据不足、模型能力不足或偶发外部错误允许不写回。每一次流程候选无论接受、拒绝还是回滚都追加到 `Meta`, 从而保留“为什么改”和“为什么没有改”的演化轨迹。

执行检查只承担可观察性。每项检查记录未触发、待完成、已完成、失败或有理由跳过等状态, Harness 据此给出检查完成与否的确定判断并供归因使用, 但不拦截每次工具调用, 也不保证执行模型遵守。

本研究不替代 SkillOpt、Trace2Skill、SkillGrad 等候选更新后端, 也不重新发明项目记忆存储、检索器、目标编译器或完整开发 Harness。`Meta` 作为修改日志本身也不是新颖性; 它只有在与结构化作用域、独立写事务和跨仓库提升规则共同改变流程记忆的可写条件时才具有研究意义。本文真正改变的是: 未来任务会复用的状态不能由产生该证据的执行上下文直接改写; 项目知识可以持续读写但保持项目作用域; 流程知识只有在 `Meta` 所保留的跨仓库证据和验证支持下才被提升。

研究要检验的中心判断是: **可迁移流程记忆既需要结构化的作用域边界, 也需要与执行分离、由 Meta 支持的生命周期写入治理。** 如果这一判断成立, 那么结构化表示本身应减少项目值污染; 独立治理事务应比执行 Agent 直接读写记忆产生更少的错误绑定和跨层误写; 在相同模型、后端、证据和预算下, 加入结构化 `Meta` 与跨仓库验证应比无历史或扁平修改历史减少单仓库经验被错误提升为流程规则、重复无效补丁和跨轮回归; 冻结同一流程版本后应获得更稳定的跨项目迁移。若收益只来自额外模型调用、更强 Supervisor、不同后端、更多信息, 或简单版本日志与直接自修改达到同等效果, 则本文的隔离治理和 `Meta` 主张必须被削弱或推翻。

## 核心主张

### 共识

长程代码任务的成功是模型、Harness、工具环境、项目状态和人类决策共同作用的结果。跨任务可复用的流程知识与当前项目才成立的事实具有不同的作用域、更新频率、证据来源和失效方式。持久记忆既需要被读取, 也需要随新证据被修正; 但一次任务产生了写入证据, 不等于执行该任务的上下文应直接拥有写入未来复用状态的权限。

### 问题

当前 Skill 表示和进化过程通常同时缺少作用域接口与写入治理。流程没有明确声明尚未绑定的项目知识, 执行 Agent 又可能在同一长上下文中解释失败并直接更新项目记忆或 Skill。系统缺少跨仓库、带验证结果的修改历史来判断一条局部经验是否值得升级。可观察后果包括项目值污染、过期事实采用、错误项目写入、单仓库经验升格、重复提出已失败补丁、跨轮回归和冻结迁移退化。

### 方法

方法由两个核心模块组成。结构化记忆层用 `S=<A,K,P,O,V>` 保存流程不变量, 通过外部但可读写的 `M_project` 保存项目事实。带 `Meta` 的生命周期 Supervisor 使用与执行分离的事务上下文完成创建与版本治理、选择、绑定、编译、归因、项目记忆写入、流程候选生成、跨仓库验证、提交和回滚。执行 Agent 只消费编译结果并产生证据, 不直接修改持久记忆; `Meta` 只追加记录每次流程修改的触发情形、跨仓库证据、验证结果与接受、拒绝或回滚决定。

### 原因

结构化作用域使项目事实不必提前固化进通用步骤; 独立治理事务使产生证据的执行上下文不能直接改写未来任务会复用的状态; 可读写项目记忆允许局部知识持续纠错而不污染流程; `Meta` 保留跨仓库时序证据, 使 Supervisor 能区分单仓库偶然现象、反复出现的流程缺陷和已经失败过的修改方向。过拟合因此不是被上下文隔离自动消除, 而是通过项目默认写入、跨仓库提升门、验证、拒绝与回滚共同受到约束。预期结果是在公平预算下减少错误绑定、错误项目写入、流程误提升、重复补丁和跨轮回归, 提高项目记忆质量、演化可追溯性与冻结迁移稳定性。
