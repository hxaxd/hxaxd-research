# 论文记录字段

论文记录由研究工作台后端的数据契约定义。本文解释字段的研究语义和填写责任；字段是否必填、允许值和类型以运行中的 `GET /api/schema/paper` 响应为唯一权威，不在指导中复制一份可能失效的数据库结构。

## 1. JSONL 交付格式

Agent 为本轮检索得到且完成核验的每篇候选论文生成一个 JSON 对象，每行一篇，形成 JSONL。文件只作为立即提交服务器的临时机械载体，不是候选列表或长期研究文档；成功提交或任务失败处理结束后删除。候选论文不在数据库外等待用户确认，用户必须先能在页面中看到它们。

JSONL 中不填写数据库生成的 `id`、`project_id`、`created_at` 和 `updated_at`。提交时通过命令参数指定目标项目，后端负责生成其余字段。

```json
{"stable_key":"doi:10.xxxx/example","status":"discovered","title_en":"Official English Title","title_zh":"中文标题","authors":["First Author","Second Author"],"organization":"First Author Affiliation","publication_year":2026,"publication_status":"Conference 2026","paper_type":"方法","main_method":"一句话说明主要方法。","contribution":"核心贡献。","selection_reason":"它在当前知识结构中不可替代的作用。","reading_focus":"需要重点阅读的章节、实验或论证。","relations":"与已收录论文的延续、反驳、扩展或评测关系。","stable_url":"https://doi.org/10.xxxx/example","code_url":null,"website_url":null}
```

## 2. 字段语义

- `stable_key`: 项目内稳定去重键。优先使用规范化 DOI，其次 arXiv ID、OpenReview ID，最后才使用稳定 URL。不得使用标题缩写或本地顺序号。
- `status`: 本轮新检索出的候选论文默认填写 `discovered`，提交后立即出现在页面中。用户随后通过对话告诉 Agent 筛选决定，Agent 再把对应记录更新为 `included`、`excluded` 或 `archived`。只有用户在检索前已经明确指定某篇必须收录时，才可以直接填写 `included`。
- `title_en`: 论文官方英文标题，保持原始拼写，不自行改写。
- `title_zh`: 准确、自然且术语一致的中文标题，不为了吸引力扩大原意。
- `authors`: 按论文正式版本中的顺序填写完整作者名数组，不用 `et al.` 代替其余作者。
- `organization`: 第一作者在该论文上的已核验所属组织。无法从原始来源可靠确认时填写 `null`，不得猜测。
- `publication_year`: 首次公开或正式发表年份。版本关系复杂时，以 `publication_status` 进一步说明。
- `publication_status`: 明确写预印本、会议、期刊、track 和年份；不得把投稿中、传闻或搜索摘要写成已发表。
- `paper_type`: 按知识结构中的主要角色选择当前契约允许的类型，如综述、奠基、方法、系统、Benchmark、反例或相邻工作。
- `main_method`: 一句话说明论文实际上采用什么机制解决什么问题，不复述标题。
- `contribution`: 论文相对已有工作的真实增量，区分作者自称与当前核验结论。
- `selection_reason`: 论文为什么值得进入当前项目；必须说明不可替代的知识作用，而不是“高引用”“很经典”或“效果很好”。
- `reading_focus`: 后续阅读最应关注的章节、实验、假设或论证，以及需要验证的疑点。
- `relations`: 与当前项目其他论文的具体关系，如奠定、延续、反驳、扩展、提供评测或暴露边界。孤立论文也要说明它相对目标问题的位置。
- `stable_url`: DOI、arXiv、OpenReview、PubMed 或正式出版页面等稳定入口，不使用搜索结果页。
- `code_url`: 已核验的官方代码仓库；没有则填写 `null`。
- `website_url`: 已核验的官方项目页或 Demo；没有则填写 `null`。

## 3. 完整性要求

- 每一行独立满足当前后端契约，不依赖上一行补充共同字段。
- 所有候选论文的每个必填字段都要在首次提交前完成，不因为状态还是 `discovered` 就只写轻量占位信息，也不把字段补全责任推给数据库或后续 Agent。
- `organization`、`code_url` 和 `website_url` 等可空字段无法核验时使用 `null`，不使用“未知”“待补”等伪值。
- 同批记录中的标题、作者、年份、发表状态和稳定标识符必须互相一致。
- 提交前检查 JSONL 行数等于本轮准备展示给用户的候选论文数；提交后检查服务器返回的创建数与之相等。
