# 学习工作台重构计划

> 状态：已于 2026-07-20 实施并通过验收  
> 范围：只重构学习阶段  
> 数据原则：保留现有服务器数据，任何迁移必须先备份、验证并可回滚

## 1. 目标与边界

工作台只稳定实现两条主流程：

1. 文献检索、核验、登记和用户筛选。
2. 获取 PDF 或 TeX；必要时把 TeX 编译为 PDF，并以 PDF 作为主要阅读格式。

系统需要做到：

- Agent 只受完成任务所需的最小约束。
- 人工文档简洁、无重复、可持续修改。
- 后端负责身份规范化、去重、字段校验、存储和任务状态。
- 平台通过运行时能力告诉 Agent 是否支持并准备好编译、翻译。
- 每个用户语义字段都有明确页面位置或调用方。
- 数据库、备份和 API 能够随版本演进。

本次不做：

- 写作阶段。
- 论文讲解和精读笔记系统。
- 思维导图、知识图谱和领域概览。
- 推荐算法、向量检索和语义搜索。
- 多用户、权限和云端部署。
- 前端视觉风格重做。
- 自动安装工具。
- 未经用户明确要求自动翻译。

TeX 编译属于资源获取的补全动作：没有合适 PDF、已有合法完整 TeX 且编译能力就绪时，Agent 可以提交编译任务；工具缺失时只报告，不自动安装。

## 2. 已核验基线

截至 2026-07-20：

- 后端健康检查正常。
- 服务器有 5 个项目、88 篇论文、264 个 PDF 资源。
- 88 篇论文均有原文、中文和双语 PDF。
- TeX 工具已安装，PDF2zh 未安装。
- 后端 15 项测试通过。
- 前端 2 项测试、类型检查、静态检查和生产构建通过。

现有测试只能证明当前实现内部自洽，尚未覆盖目标产品行为。

已确认的结构问题：

- 论文事实、项目判断和阅读建议混在同一记录。
- 首次发现候选就强制填写大量研究判断。
- `stable_key` 由 Agent 手工生成，而且只在项目内唯一。
- `organization` 难以稳定核验。
- `publication_status` 混合场所、年份、版本和发表状态。
- `paper_type` 是单选且错误地属于全局论文。
- 缺少摘要等基础元数据。
- 资源被固定成三种 PDF，不能表达 TeX 和派生关系。
- TeX 工具存在，但没有编译接口。
- `/api/workspace` 返回全部论文全文字段，当前响应约 214 KB。
- 数据库没有版本化迁移。
- 快照要求 Schema 完全相同，不能自然恢复旧版本再升级。
- 论文详情页没有展示完整元数据和项目判断。
- PDF 阅读器只有单页 Canvas，没有文本层、连续滚动和搜索。

## 3. 核心设计决策

### 3.1 五个正交概念

```text
Project
  └─ ProjectPaper
       └─ Paper
            └─ Resource
                 └─ Job -> output Resource
```

- `Project`：学习范围和边界。
- `Paper`：全局唯一的论文客观事实。
- `ProjectPaper`：论文在某项目中的状态、角色和判断。
- `Resource`：PDF、TeX 及其派生文件。
- `Job`：编译、翻译等转换任务。

同一论文只保存一份事实，可以进入多个项目并拥有不同判断。

### 3.2 渐进完整

- `discovered` 只要求足以识别和展示候选的字段。
- `included` 再要求完整作者、发表信息、稳定入口和项目相关性。
- `excluded`、`archived` 保留身份信息用于去重。
- 无法核验的事实保持为空，不用伪值填满。
- 完整性由后端根据状态校验。

### 3.3 后端负责机械正确性

后端负责：

- 规范化 DOI、arXiv、OpenReview 等标识符。
- 全局去重和跨项目复用。
- 状态相关完整性校验。
- 批量写入的事务性与幂等性。
- 文件格式、散列、路径和安全校验。
- 资源版本、首选资源和派生关系。
- 编译与翻译任务的输入、输出、日志和状态。

Agent 负责：

- 理解用户范围并选择检索策略。
- 判断来源可信度和论文身份一致性。
- 给出有证据的摘要、贡献和项目相关性。
- 在用户授权边界内调用后端并回读结果。

### 3.4 资源不可静默覆盖

- 新文件创建新资源记录。
- 替换首选资源只改变指向，不覆盖历史文件。
- 相同散列可以复用已有 Blob。
- 派生资源必须可追溯输入资源、任务和工具版本。

## 4. 目标领域模型

### 4.1 Project

字段保持最小：

- `id`
- `name`
- `description`
- `created_at`
- `updated_at`

暂不增加项目阶段、负责人和复杂配置。

### 4.2 Paper

保存客观事实：

- `id`
- `title`：官方标题。
- `title_zh`：可选，不参与身份判断。
- `authors`：按正式顺序保存完整作者。
- `abstract`：有可靠来源时保存。
- `publication_year`
- `venue`
- `publication_state`：`preprint`、`accepted`、`published` 或 `unknown`。
- `identifiers`：DOI、arXiv、OpenReview、PMID 等结构化标识符。
- `links`：稳定页面、代码仓库、项目页等带类型链接。
- 系统时间字段。

身份规则：

- 优先按规范化公开标识符判断同一论文。
- 标题和作者只用于冲突检查及弱匹配，存在歧义时不得静默合并。
- `stable_key` 改为后端内部派生值，不再由 Agent 输入。
- 论文身份在整个工作区唯一。

不再进入活动模型：

- 第一作者机构。
- 引用量和实时排名。
- 自由文本论文关系。

### 4.3 ProjectPaper

保存项目判断：

- `project_id`
- `paper_id`
- `status`：暂时保留 `discovered`、`included`、`excluded`、`archived`。
- `roles`：综述、奠基、方法、系统、Benchmark、反例、相邻工作等多选角色。
- `summary`：论文做了什么及主要机制。
- `contributions`：贡献数组。
- `relevance`：为什么进入当前项目；收录时必填。
- `reading_focus`：后续阅读关注点数组。
- 系统时间字段。

只有用户明确决定后才能改变筛选状态。

### 4.4 Resource

资源按正交维度描述：

- `paper_id`
- `format`：`pdf` 或 `tex`。
- `representation`：`original`、`translated` 或 `bilingual`。
- `origin`：`publisher`、`preprint`、`author`、`user` 或 `generated`。
- `source_url`
- `filename`
- `media_type`
- `sha256`
- `size`
- `storage_key`
- `preferred`
- `parent_resource_id`
- `job_id`
- `created_at`

关键约束：

- TeX 按完整源码包保存，首期支持 `.tar`、`.tar.gz`、`.zip`。
- 解包必须阻止路径穿越、绝对路径、符号链接逃逸和异常压缩比。
- PDF 必须实际可解析并具有合理页数，不能只检查文件头和 EOF。
- 官方 PDF 和 TeX 编译 PDF 可以同时存在，由 `preferred` 决定默认阅读版本。

### 4.5 Job

统一编译和翻译任务：

- `operation`：`compile` 或 `translate`。
- `input_resource_id`
- `status`：`queued`、`running`、`succeeded`、`failed`。
- `progress`
- `options`
- `tool`、`tool_version`
- `message`
- `log_excerpt`、`error_summary`
- 时间字段。

任务输出通过新 Resource 的 `job_id` 关联，不在任务中保存不可校验的物理路径。

## 5. 目标流程

### 5.1 检索与登记

1. Agent 读取紧凑工作区和能力状态。
2. 选择现有项目；只有范围明显不同时才创建项目。
3. 自主选择检索式、来源和停止条件。
4. 核验每篇候选的最小身份事实。
5. 批量提交论文事实和项目判断。
6. 后端规范化身份、复用已有论文并保证幂等。
7. 新候选以 `discovered` 展示。
8. Agent 回读并报告创建、复用、冲突和缺口。
9. 用户决定后更新项目关系状态。

### 5.2 获取资源

1. 论文必须已经登记。
2. 读取现有资源和平台能力，已有正确资源时直接复用。
3. 获取合法、完整、身份一致且来源清楚的 PDF 或 TeX。
4. PDF 登记为 `pdf + original`。
5. 找不到合适 PDF 但有完整 TeX 时，登记为 `tex + original`。
6. 编译能力可用时提交任务，生成 `pdf + original` 派生资源。
7. 成功后选择合适的首选阅读 PDF。
8. 回读资源、任务状态和可读取内容。
9. 清理 Agent 创建的网络临时文件。

### 5.3 翻译

1. 由用户明确触发。
2. 后端使用首选 `pdf + original`，无论它来自下载还是 TeX 编译。
3. 成功后登记 `pdf + translated` 和 `pdf + bilingual`。
4. 失败时保留原资源，不产生半完成首选记录。

## 6. 目标 API

### 6.1 启动与能力

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| `GET` | `/api/health` | 健康检查 |
| `GET` | `/api/workspace` | 项目摘要、数量、契约版本和能力状态 |
| `GET` | `/openapi.json` | 完整机器契约 |

`/api/workspace` 不再内嵌全部论文。能力需要区分：

- `supported`：平台实现了该操作。
- `ready`：工具和配置当前可用。

至少报告：

- 支持 `pdf`、`tex`。
- `compile` 接受 TeX、生成 PDF。
- `translate` 接受 PDF、生成中文和双语 PDF。
- 当前工具版本和就绪状态。

### 6.2 项目和论文

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| `GET/POST` | `/api/projects` | 列出或创建项目 |
| `GET/PATCH` | `/api/projects/{project_id}` | 读取或修改项目 |
| `GET` | `/api/projects/{project_id}/papers` | 项目论文组合视图 |
| `POST` | `/api/projects/{project_id}/papers/batch` | 批量创建或复用论文并建立项目关系 |
| `GET/PATCH` | `/api/papers/{paper_id}` | 读取或修改论文事实 |
| `PATCH` | `/api/projects/{project_id}/papers/{paper_id}` | 修改筛选状态和项目判断 |

批量接口一次接收“论文事实 + 项目判断”，逐项报告：

- 新建论文。
- 复用全局论文。
- 新建项目关系。
- 已存在且无变化。
- 身份或字段冲突。

重复提交不得产生重复记录。

### 6.3 资源和任务

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| `GET/POST` | `/api/papers/{paper_id}/resources` | 列出或上传 PDF、TeX |
| `GET` | `/api/resources/{resource_id}/content` | 浏览或下载文件 |
| `PATCH` | `/api/resources/{resource_id}` | 切换首选或补充来源 |
| `POST` | `/api/jobs` | 提交编译或翻译任务 |
| `GET` | `/api/jobs/{job_id}` | 读取任务状态和输出 |

错误体统一包含稳定 `code`、面向用户的 `message` 和可选 `details`。Agent 根据 `code` 处理错误，不解析自然语言消息。

字段类型由 OpenAPI 和运行时 JSON Schema 生成，人工文档不复制一套字段契约。

## 7. 数据迁移与备份

### 7.1 先建立迁移机制

任何业务表修改前完成：

- 新增 `schema_migrations`。
- 使用单调递增的编号迁移。
- 启动时按顺序执行未完成迁移。
- 每次迁移在事务中运行。
- 测试覆盖空库初始化和旧版本升级。

当前规模不要求引入大型迁移框架，编号 SQL/Python 迁移即可。

### 7.2 V1 到 V2 映射

| V1 | V2 |
| --- | --- |
| 项目 | 原样迁移为 Project |
| 标题、作者、年份、链接 | 迁移为 Paper 事实 |
| 状态、类型 | 迁移为 ProjectPaper 状态和角色 |
| `main_method` | `summary` |
| `contribution` | 单元素 `contributions` |
| `selection_reason` | `relevance` |
| `reading_focus` | 单元素 `reading_focus` |
| `relations_text` | 只读 legacy 扩展，不进入新写入契约 |
| `organization` | 只读 legacy 扩展 |
| `stable_key` | 可解析时规范化为 identifier，否则保留 legacy 标识 |
| original/chinese/bilingual Artifact | 对应的 Resource |
| 历史任务 | 能建立关系时迁移，否则标记 legacy |

数据质量处理：

- 含 `et al.` 的作者保留原值并标记不完整，不自动补写。
- `doi:10.48550/arxiv.*` 规范化为 arXiv 标识，同时保留原值用于审计。
- 标题相同但稳定标识冲突时停止合并并生成报告。
- 迁移前后对账项目数、论文数、资源数、文件大小和散列。

### 7.3 快照版本

快照清单分开记录：

- 容器格式版本。
- 数据库 Schema 版本。
- 应用契约版本。

恢复时先在独立暂存目录验证文件和散列，再迁移旧 Schema、运行完整性检查，最后原子切换。不能再只因 Schema 文件散列不同而拒绝所有旧快照。

## 8. 前端最低功能面

本轮只保证功能和信息完整，不重做风格。

### 8.1 首页

- 项目和筛选状态数量。
- PDF、TeX、编译和翻译的 supported/ready 状态。
- 工具安装与备份入口继续保留。

### 8.2 项目页

每篇论文展示：

- 中文与官方标题。
- 作者、发表场所、年份和状态。
- 筛选状态和多个角色。
- 一句话摘要或项目相关性。
- PDF、TeX、中文、双语资源徽标。

支持按状态、角色、年份和资源可用性筛选。

### 8.3 论文页

分成三个区域：

1. 论文事实：标题、完整作者、摘要、发表信息、标识符和链接。
2. 项目判断：状态、角色、摘要、贡献、相关性和阅读重点。
3. 资源与阅读：资源来源、派生关系、任务状态和 PDF 阅读器。

没有页面位置或明确 Agent 消费方的用户语义字段，不应长期保存在活动模型中。

### 8.4 PDF 阅读器

最低要求：

- 连续纵向滚动。
- 适宽、适页和自定义缩放。
- 文本选择、复制和文内搜索。
- 链接与注释层。
- 页码跳转。
- 原文、中文、双语切换。
- 大文件按需渲染。
- 切换资源后不显示上一版本的旧页面。

优先复用 PDF.js 标准 Viewer 能力或完整页面层，不继续扩展单 Canvas 组件。

## 9. 文档架构

目标活动文档：

```text
AGENTS.md
API.md
docs/
├─ domain.md
└─ workflows/
   ├─ literature-search.md
   └─ resource-acquisition.md
```

职责：

- `AGENTS.md`：30–50 行，只写范围、启动顺序、两条路由和硬边界。
- 两份工作流：目标、质量边界、用户权限和完成条件。
- `domain.md`：解释 Paper、ProjectPaper、状态、资源和转换语义。
- `API.md`：服务入口、资源族、典型调用、幂等、错误和回读。
- OpenAPI/JSON Schema：唯一字段级机器契约。
- `README.md`：只面向人类说明当前产品和启动方式。

单向路由：

```text
AGENTS.md -> 一个工作流 -> domain.md / API.md -> 运行中契约
```

现有文档处理：

- 精简根入口和 API 文档。
- 两份学习工作流迁入 `docs/workflows/`。
- 论文字段语义合并进 `domain.md`。
- 领域概览草案移出活动路由。
- `ROUTER.md` 不再承担职责。
- 删除 README 中暂停的写作阶段说明和重复 Agent 指令。

## 10. 测试门槛

### 10.1 必须覆盖的核心场景

1. 同一论文进入两个项目：一个 Paper、两个 ProjectPaper。
2. 同一批次重复提交：结果幂等。
3. discovered 可轻量创建，included 触发完整性校验。
4. 新资源不会覆盖旧文件，首选切换可回滚。
5. PDF 实际解析；TeX 安全解包。
6. TeX 上传后能够编译为可读 PDF，并保留派生链。
7. 翻译可使用下载 PDF 或 TeX 编译 PDF。
8. 任务失败不改变现有首选资源。
9. V1 数据和快照能够升级到 V2。
10. 工作区响应保持紧凑并准确报告能力。
11. 前端展示全部用户语义字段和资源状态。
12. 浏览器端完成“候选登记 -> 收录 -> 获取资源 -> 阅读”主流程。

### 10.2 每阶段质量门槛

- 后端测试和静态检查通过。
- 前端测试、类型检查、静态检查和构建通过。
- 新 API 有契约测试。
- Schema 改动有迁移测试。
- 主页面流程有浏览器验收。
- 写操作后从公开 API 回读真实状态。

## 11. 分阶段实施

### 阶段 0：冻结基线与确认决策

交付：

- 当前数据、工具、测试和 API 基线。
- 可验证的完整备份。
- V1 数据质量报告。
- 确认本文待决策项。

门槛：备份可验证，现有测试通过，V2 核心字段无未解释歧义。

### 阶段 1：数据库演进基础

交付：

- `schema_migrations` 和编号迁移执行器。
- 空库初始化和旧库升级测试。
- 新快照版本字段。
- 暂存迁移后恢复机制。

门槛：迁移失败不破坏旧库，旧快照可进入受支持迁移链。

### 阶段 2：论文事实和项目关系

交付：

- Paper、Identifier、ProjectPaper 数据结构。
- 全局去重、跨项目复用和状态校验。
- 新批量收录接口。
- 紧凑工作区和能力契约骨架。

门槛：跨项目只有一份论文事实，重复提交幂等，Agent 不再提交 `stable_key`。

### 阶段 3：统一资源和任务

交付：

- Resource 与通用 Job。
- PDF/TeX 上传、验证、读取和下载。
- 首选资源和派生关系。
- TeX 编译。
- 翻译迁入通用任务框架。

门槛：完成“TeX -> 编译 -> PDF 阅读”闭环，失败任务不破坏现有资源。

### 阶段 4：迁移现有数据

交付：

- 正式 V1 -> V2 迁移。
- 88 篇论文与 264 个资源的迁移报告。
- 不完整作者、旧标识符和冲突报告。
- 迁移前后数量、大小和散列对账。

门槛：现有数据均能通过新 API 访问，无未解释文件差异，可恢复迁移前备份。

### 阶段 5：补齐功能型前端

交付：

- 首页能力状态。
- 项目页完整字段和资源徽标。
- 论文事实、项目判断和资源详情。
- TeX 上传与编译状态。
- 完整 PDF 阅读器。

门槛：所有用户语义字段有页面位置，两条主流程状态可见，阅读器满足最低要求。

### 阶段 6：精简文档并切换契约

交付：

- 新根入口、API、领域说明和两份工作流。
- 旧文档退出活动路由。
- README 只保留当前能力。
- 清理旧接口和不可达代码。

门槛：新 Agent 只读根入口和一个工作流即可完成任务，人工文档不复制字段契约，全量测试通过。

## 12. 整体完成定义

基础重构完成时，必须能够稳定演示：

1. Agent 选择或创建学习项目。
2. 候选论文轻量登记且重复执行不重复。
3. 用户筛选决定得到正确记录。
4. Agent 能获取 PDF；无 PDF 时能用 TeX 编译得到 PDF。
5. 页面展示完整论文事实、项目判断、资源和任务状态。
6. 用户能连续阅读、搜索和复制 PDF 文本。
7. 用户触发翻译后能阅读中文和双语版本。
8. 所有资源有来源、散列和派生链，历史不会被静默覆盖。
9. 旧数据库和备份可以迁移、验证和恢复。
10. Agent 文档短小，无需理解数据库和文件存储细节。

## 13. 已落实决策

1. 迁移期保留 `archived`，稳定运行后再评估是否精简。
2. TeX 同时支持 tar、tar.gz、zip，并统一执行安全检查。
3. 官方 PDF 默认优先；编译 PDF 不抢占已有原文首选，用户可以切换。
4. 旧 `relations_text` 和 `organization` 进入只读 legacy 扩展表。
5. `summary` 和 `contributions` 允许在收录后继续为空；收录只强制可靠身份和 `relevance`。
