# 仓库开发与维护指南

本文面向维护和迭代本仓库的人, 它不是 Agent 的日常执行入口, Agent 仍从`AGENTS.md` 开始

## 1. 设计约束

- 同一规则只能有一个主要归属, 其他文件只引用或补充, 不能复制出多个略有差异的版本
- 区分阶段和可重复调用的能力, 阶段反映输入输出依赖关系 (后面需要前面的产物), 不是流程门禁, 能力由用户请求独立触发, 做完就结束
- 允许跳过, 返工, 乱序写作和独立任务, 推荐路线不是流程门禁
- 让机械工具处理大文件而不把正文自动塞入 Agent 上下文

## 2. 变更影响表

| 想修改的内容 | 首选文件 | 通常需要联动检查 |
| ---- | ---- | ---- |
| 用户第一次看到的项目定位或示例 | `README.md` | `AGENTS.md` 是否仍一致 |
| Agent 启动顺序, 最高边界, 仓库对象定义 | `AGENTS.md` | `ROUTER.md`, 开发指南 |
| 新增, 删除或重命名阶段 | `ROUTER.md` | 阶段专项指导, README |
| 把某功能从阶段改成能力或反向调整 | `ROUTER.md` | 对应指导标题, 全部旧编号引用 |
| 修改跳过和返工规则 | `ROUTER.md` | `指导/使用说明.md` |
| 修改首次落盘, 输入检查或组合阻塞规则 | `指导/使用说明.md` | `AGENTS.md`, 相关专项指导 |
| 修改工作区, 领域, 论文或项目目录结构 | `AGENTS.md` 与`ROUTER.md` | README, 处理指导, 开发指南 |
| 修改论文归档, PDF 翻译和验收方法 | `指导/学习/论文处理工具使用.md` | 脚本, 配置指导, README |

| 修改论文来源和版本选择规则 | `指导/学习/论文资源获取.md` | |
| 修改 R1 Idea 预审与立项方法 | `指导/研究/研究想法审查与落地.md` | R1 契约, `相近工作与撞车分析.md`, README |

如果一项修改同时落在多层, 先修改最高层概念再向下同步, 例如改变论文目录结构时, 顺序应是`AGENTS.md` →`ROUTER.md` → 通用协议 → 专项指导 → README

## 3. 新增阶段或能力

### 新增阶段

1. 在`ROUTER.md` 定义编号, 触发条件, 不触发条件, 所需输入, 产物和指导路径
2. 创建专项指导, 并说明跳过和返工影响
3. 更新阶段列表
4. 搜索所有编号和旧路线, 避免出现两个权威版本

### 新增能力

1. 确认它可以由用户请求独立触发
2. 创建专项指导
3. 在`ROUTER.md` 能力表中登记
4. 明确触发条件, 不触发条件和指导路径
5. 确保能力输出不会伪造阶段完成

### 新增交接文档

只有当信息需要跨多个阶段长期维护, 且现有文档无法合理承载时才新增, 必须定义谁创建, 谁更新, 稳定编号或状态, 哪些阶段读取, 新证据如何覆盖或保留历史。文献, 引用, 领域共识和相关工作比较默认进入`文献证据库.md`; 核心主张默认进入`叙事定稿.md`, 不新增平行文件

## 4. 修改时的核对方法

每次文档契约修改至少执行:

1. 搜索旧术语, 旧编号和旧路径是否残留
2. 检查 README, AGENTS, ROUTER, 通用协议和专项指导是否出现冲突
3. 检查所有反引号中的`指导/...md` 路径真实存在
4. 用典型场景走查直接入口, 跳过, 返工和独立能力
5. 运行`git diff --check`
6. 确认没有为了说明规则创建空的工作区模板

目录模型变化时额外检查:

- 是否仍有旧目录名残留
- 是否会导致同一材料复制多份
- 新对象的创建权属于 Agent 还是用户
- 已有工作区是否需要迁移, 以及迁移是否必须由用户确认

## 5. 常见设计失败

- **入口过早承诺**: 还没读 ROUTER 就宣布工具, Skill, 目录和方案
- **层级泄漏**: 把专项步骤塞进 AGENTS, 或把状态机细节塞进 README
- **重复权威**: 同一规则在多个文件分别定义, 后续只改了一处
- **阶段膨胀**: 每个工具操作都编号, 导致流程图看似精确但无法灵活调用
- **能力越权**: 翻译, 绘图或审稿完成后宣称阶段完成
- **路径代替语义**: 仅凭目录名字判断状态, 忽略真实内容和用户确认
- **Agent 替用户分类**: 把论文自动归入某个领域, 混淆技术推断与知识组织决定
- **组合任务被静默拆分**: 用户要求完整结果, Agent 留下半成品却宣称已经推进
- **为整洁而复制**: 同一论文, 证据或实验结果在多个目录重复保存, 随后版本漂移
- **工具读取等同模型阅读**: PDF2zh 可在独立进程处理全文, 却把正文回显给 Agent
- **过度严谨化**: 把推荐路线和检查表变成无法跳过的门禁, 牺牲实际研究灵活性

## 6. 维护原则

- 优先修正错误, 再补缺失, 最后才处理轻微不严谨
- 不为了统一格式删除已有有效研究约束
- 规则应足够明确以避免错误, 但不应消灭用户可以主动选择的灵活性
- 所有新增确认点都应回答: 这个决定是否真的只能由用户作出
- 所有新增文档都应回答: 它是否拥有现有文档无法承担的独立责任
- 所有新增目录层级都应回答: 它是否提供真实语义, 而不只是让树看起来整齐
