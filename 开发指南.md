# 仓库开发与维护指南

本文面向维护和迭代本仓库的人，解释设计目的、文档分层、路由机制、目录模型，以及未来修改某类行为时应该改哪个文件。

它不是 Agent 的日常执行入口。Agent 仍从 `AGENTS.md` 开始；普通用户主要阅读
`README.md`。

## 1. 设计目标

本仓库试图解决的不是“如何写一个超长提示词”，而是如何让 Agent 在长期研究任务中：

1. 从真实工作区恢复上下文，而不是依赖聊天记忆。
2. 区分主线阶段和可重复调用的工具能力。
3. 把关键决定留给人类，同时减少无意义的重复确认。
4. 让事实、文献、实验、Claim 和最终稿件保持可追溯。
5. 允许跳过、返工、乱序写作和独立任务，不把推荐路线变成流程门禁。
6. 通过公开 Markdown 规范持续迭代，而不依赖隐藏状态或 Skill 注册。
7. 让机械工具处理大文件而不把正文自动塞入 Agent 上下文，保留上下文给真正的语义
   判断。

设计时优先保持概念边界清楚。详细规则可以多，但同一规则只能有一个主要归属；其他文件只引用或补充它，不能复制出多个略有差异的版本。

## 2. 文档分层

### README.md：给人看的入口

放置：

- 项目是什么、不是什么；
- 用户如何在 Codex 中开始；
- 可以提出哪些典型请求；
- 工作区中会产生什么；
- 最简目录结构和可选外部工具使用方法。

不要放置：

- Agent 内部启动顺序；
- 完整状态判定算法；
- 细粒度研究方法；
- 大量禁止事项和冲突优先级。

判断标准：第一次看到仓库的人能否理解“怎么用”和“会得到什么”。

### AGENTS.md：给 Agent 看的最高入口

放置：

- 仓库定位和规范文档之间的权威关系；
- 仓库固有概念；
- Agent 启动读取顺序；
- 在完成路由前不能预设方案等入口纪律；
- 中文、证据、人类决策、不伪造和访问控制等最高级边界。

不要放置：

- 每个阶段的输入和产物；
- 具体检索、实验、翻译或写作步骤；
- 大量阶段特有的完成条件。

判断标准：Agent 只读本文件后，是否知道“先读什么、不能先做什么、仓库里的对象是什么”，但还不会误以为已经知道具体任务怎么执行。

### WORKFLOW.md：生命周期和路由控制面

放置：

- 生命周期阶段和按需能力的区别；
- 阶段状态、任务整体结果及推导方式；
- 新领域、孤立论文、研究项目和独立能力的初始化规则；
- 用户意图到阶段或能力的路由；
- 推荐转移、跳过、返工、阻塞和返回语义；
- 每个生命周期阶段的目的、输入、产物、确认点和完成条件；
- 能力索引及其主指导路径。

不要放置：

- 某个工具的完整命令说明；
- 某类论文具体怎么检索或怎么写；
- 每个专项任务的详细检查清单。

判断标准：它应像控制面和地图，说明“当前在哪里、调用什么、完成后去哪”，而不是亲自执行每一种工作。

### 指导/使用说明.md：所有行为共同继承的协议

放置：

- 调查、输入检查、确认、首次落盘、回写和完成的共同方法；
- 独立目标与组合目标的阻塞语义；
- 证据标签、有限检索、停止条件和通用禁止事项；
- 阶段与能力都要遵守的产物质量要求。

不要放置：

- 单个阶段独有的文件列表；
- 特定外部工具的参数；
- 某一章节的写法或某种实验的设计细节。

判断标准：如果一条规则对大多数专项指导都成立，就应优先放在这里。

### 指导/ 下的专项文档：具体执行方法

放置：

- 当前阶段或能力特有的输入；
- 专项执行步骤；
- 专项证据标准和禁止事项；
- 专项产物、完成条件和返回位置；
- 需要引用的辅助指导。

生命周期阶段指导可以使用 L1-L2、R1-R8 标题。按需能力不使用阶段编号，避免让它们看起来必须线性经过。

判断标准：读完该文件后，Agent 应知道这件具体事情怎么做，但不能突破上层确认、
证据和目录规则。

### 交接文档指导：跨阶段数据契约

`指导/交接/` 规定 `文献证据库.md`、`相关工作矩阵.md`、`核心主张.md`、
`失败记录.md` 等长期文档的字段和更新规则。

修改表格字段、状态枚举或跨阶段责任时，应首先修改对应交接指导，再检查所有使用该文档的阶段和能力。

### scripts/ 与 PDF2zh：机械能力

仓库不再维护项目级 Python 文档处理代码。`scripts/install-pdf2zh.ps1` 安装固定版本外部工具，`scripts/translate-pdf.ps1` 负责参数、Windows 持久环境变量、临时输出、
文件级验收和中文文件名。

PDF2zh 读取全文与 Agent 模型阅读全文必须保持分离。脚本只输出版本、路径、大小、哈希和错误，不回显论文正文。仓库调用 PDF2zh，不复制、修改或 vendoring 其 AGPL-3.0
源码；工具缺陷应上游报告或等待修复，不能重新长出本地 PDF 解析与翻译框架。

外部能力边界以实际工具为准：PDF2zh 只接受 PDF，输出单语和双语 PDF。PDF→Markdown、
Markdown/HTML 翻译和通用 TeX 翻译不属于论文机械处理。TeX 阅读材料先由原项目编译成PDF；可编辑 TeX 翻译属于写作能力。

修改脚本、固定版本或输出结构时，必须同步检查 README、两份论文处理指导、AGENTS 和本指南。升级 PDF2zh 前先核对官方 CLI、环境变量、许可证和输出命名。

## 3. 工作区目录模型

`workspace/` 顶层保持扁平，但顶层对象分为两类：

```text
workspace/
├─ <中文领域名>/
└─ <中文研究项目名>/
```

不增加 `领域/`、`项目/` 等固定类型目录。对象类型由真实内容判断。

### 领域目录

领域目录负责长期知识组织：

```text
workspace/<中文领域名>/
├─ 学习目标.md
├─ 最终论文列表.md
├─ 领域全景地图.md
├─ 资源获取记录.md
└─ <论文中文短标题>/
```

领域是人类知识分类。Agent 可以根据论文内容提出候选归属，但用户决定加入哪个已有领域或创建哪个新领域。

### 论文目录

一篇论文的材料集中在一个目录：

```text
workspace/<中文领域名>/<论文中文短标题>/
├─ 论文信息.md
├─ 原文.pdf
├─ 中文译文.pdf
├─ 双语对照.pdf
├─ 论文处理记录.md
├─ 阅读笔记.md
└─ 论文评议.md
```

文件按需创建。目录不使用 `论文资料/` 中间层，也不把原文、译文和评议拆到不同位置。

论文目录是档案主位置。同一论文被多个领域或研究项目使用时，其他位置通过路径、K
编号或证据位置引用，不复制整套档案。

### 研究项目目录

研究项目负责从 Idea 到稿件的主线：

```text
workspace/<中文研究项目名>/
├─ 研究想法.md
├─ 叙事定稿.md
├─ 核心主张.md
├─ 文献证据库.md
├─ 相关工作矩阵.md
├─ 实验记录/
└─ 稿件与其他研究产物
```

研究项目的 `文献证据库.md` 应引用领域论文档案中的原始来源和准确位置，不能只引用领域地图的二手总结。

### 实验仓库

实验代码位于被根仓库忽略的 `labs/<实际仓库目录>/`，由用户手动准备。研究项目只记录路径、commit、命令、配置和研究结论，不规定独立仓库内部布局。

## 4. 路由机制

### 第一步：完成入口读取

Agent 首先读取 `AGENTS.md`、`WORKFLOW.md` 和 `指导/使用说明.md`。完成路由前不能宣布具体工具、Skill、目录或执行方案。

这条规则防止 Agent 先形成幻觉式方案，再选择性读取文档为它辩护。

### 第二步：识别对象

判断用户正在处理：

- 新领域或已有领域；
- 领域中的论文档案；
- 新研究 Idea 或已有研究项目；
- 独立的一次性能力。

对象不明确时先检查工作区。只有无法从现有内容确定且选择会改变结果时才询问用户。

### 第三步：区分阶段与能力

生命周期阶段改变任务主线，并有状态。按需能力像函数，可以由阶段调用，也可以独立运行，结束后返回调用位置。

不要因为能力有持久产物，就把它升级成生命周期阶段；也不要因为阶段内部会调用工具，
就把阶段降格为一次工具操作。

### 第四步：检查输入和确认

确认目标、输入、范围、非目标、输出位置、外部服务、首次落盘和用户决策。

孤立论文尤其需要确认领域归属。领域名称不是从论文标题自动生成的技术细节，而是用户的知识组织选择。

### 第五步：执行或阻塞

所有必要条件满足后执行。缺少关键条件时：

- 单一目标：阻塞该目标；
- 多个真正独立目标：继续不受影响的目标；
- 用户提出的组合目标：默认整体阻塞，除非用户明确同意拆分。

### 第六步：回写与返回

阶段更新状态和交接文档。能力记录输出、未决项和返回位置，但不改变生命周期阶段。

## 5. 未来需求应该改哪里

| 想修改的内容 | 首选文件 | 通常需要联动检查 |
|---|---|---|
| 用户第一次看到的项目定位或示例 | `README.md` | `AGENTS.md` 是否仍一致 |
| Agent 启动顺序、最高边界、仓库对象定义 | `AGENTS.md` | `WORKFLOW.md`、开发指南 |
| 新增、删除或重命名生命周期阶段 | `WORKFLOW.md` | 阶段专项指导、README |
| 把某功能从阶段改成能力或反向调整 | `WORKFLOW.md` | 对应指导标题、全部旧编号引用 |
| 修改状态、跳过、终止、返工语义 | `WORKFLOW.md` | `指导/使用说明.md` |
| 修改首次落盘、输入检查或组合阻塞规则 | `指导/使用说明.md` | `AGENTS.md`、相关专项指导 |
| 修改工作区、领域、论文或项目目录结构 | `AGENTS.md` 与 `WORKFLOW.md` | README、处理指导、开发指南 |
| 修改论文归档、PDF 翻译和验收方法 | `指导/学习/论文处理工具使用.md` | 脚本、配置指导、README |
| 修改正文进入 Agent 上下文的条件 | `AGENTS.md` 与 `指导/使用说明.md` | 论文处理指导、README、脚本输出 |
| 修改论文来源和版本选择规则 | `指导/学习/论文资源获取.md` | `合法全文定位.md` |
| 修改 Idea 审查方法 | `指导/研究/研究想法审查与落地.md` | R1 契约、交接文档 |
| 修改实验代码与研究文档的边界 | 对应 `指导/实验/` 文档 | AGENTS 的 `labs/` 契约 |
| 修改论文写作章节方法 | 对应 `指导/写作/*.md` | R7 编排指导 |
| 修改 Claim、证据库或失败记录字段 | 对应 `指导/交接/*.md` | 所有读写该文档的指导 |
| 修改投稿规则、匿名或终稿验收 | 对应 `指导/投稿/`、`指导/写作/` 能力 | R8 契约 |
| 修改 PDF2zh 版本、参数或输出命名 | `scripts/` | README、论文处理指导、开发指南 |

如果一项修改同时落在多层，先修改最高层概念，再向下同步。例如改变论文目录结构时，
顺序应是 `AGENTS.md` → `WORKFLOW.md` → 通用协议 → 专项指导 → README。

## 6. 新增阶段或能力

### 新增生命周期阶段

1. 证明它确实改变主线状态，而不只是一次可重复调用的操作。
2. 在 `WORKFLOW.md` 定义编号、输入、产物、确认点、完成条件和推荐转移。
3. 创建专项指导，并说明跳过和返工影响。
4. 更新 Mermaid 生命周期图和意图路由。
5. 搜索所有编号和旧路线，避免出现两个权威版本。

### 新增按需能力

1. 确认它可以由多个阶段或独立任务调用。
2. 创建无阶段编号的专项指导。
3. 在 `WORKFLOW.md` 能力表和调用图中登记。
4. 明确输入、持久产物、确认点、完成条件和返回位置。
5. 确保能力输出不会伪造生命周期进度。

### 新增交接文档

只有当信息需要跨多个阶段长期维护，且现有文档无法合理承载时才新增。必须定义：

- 谁创建；
- 谁更新；
- 稳定编号或状态；
- 哪些阶段读取；
- 新证据如何覆盖或保留历史。

## 7. 修改时的核对方法

每次文档契约修改至少执行：

1. 搜索旧术语、旧编号和旧路径是否残留。
2. 检查 README、AGENTS、Workflow、通用协议和专项指导是否出现冲突。
3. 检查所有反引号中的 `指导/...md` 路径真实存在。
4. 检查 Mermaid 图与正文路线一致。
5. 用典型场景走查直接入口、跳过、返工、独立能力和阻塞。
6. 运行 `git diff --check`。
7. 确认没有为了说明规则创建空的工作区模板。

目录模型变化时，额外检查：

- 是否仍有旧目录名残留；
- 是否会导致同一材料复制多份；
- 新对象的创建权属于 Agent 还是用户；
- 已有工作区是否需要迁移，以及迁移是否必须由用户确认。

## 8. 常见设计失败

- **入口过早承诺**：还没读 Workflow 就宣布工具、Skill、目录和方案。
- **层级泄漏**：把专项步骤塞进 AGENTS，或把状态机细节塞进 README。
- **重复权威**：同一规则在多个文件分别定义，后续只改了一处。
- **阶段膨胀**：每个工具操作都编号，导致流程图看似精确但无法灵活调用。
- **能力越权**：翻译、绘图或审稿完成后宣称主线阶段完成。
- **路径代替语义**：仅凭目录名字判断状态，忽略真实内容和用户确认。
- **Agent 替用户分类**：把论文自动归入某个领域，混淆技术推断与知识组织决定。
- **组合任务被静默拆分**：用户要求完整结果，Agent 留下半成品却宣称已经推进。
- **为整洁而复制**：同一论文、证据或实验结果在多个目录重复保存，随后版本漂移。
- **工具读取等同模型阅读**：PDF2zh 本可在独立进程处理全文，却把正文回显给 Agent，
  无谓占用上下文并扩大内容暴露。
- **过度严谨化**：把推荐路线和检查表变成无法跳过的门禁，牺牲实际研究灵活性。

## 9. 维护原则

- 优先修正错误，再补缺失，最后才处理轻微不严谨。
- 不为了统一格式删除已有有效研究约束。
- 规则应足够明确以避免错误，但不应消灭用户可以主动选择的灵活性。
- 所有新增确认点都应回答：这个决定是否真的只能由用户作出？
- 所有新增文档都应回答：它是否拥有现有文档无法承担的独立责任？
- 所有新增目录层级都应回答：它是否提供真实语义，而不只是让树看起来整齐？
