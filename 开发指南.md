# 仓库开发与维护指南

本文面向维护和迭代本仓库的人。它不是 Agent 的日常执行入口, Agent 仍从`AGENTS.md`开始。

## 1. 设计约束

- 同一规则只能有一个主要归属, 其他文件只引用或补充, 不能复制出多个略有差异的版本。
- 阶段只保留当前确认的研究主线: R1 Idea、R2 叙事、R3 实验证明、R4 实验执行、R5 写作占位、R6 投稿占位。
- 能力由用户请求独立触发, 做完就结束, 不能伪造成阶段完成。
- 未验证能力不放入正式能力表; 用户明确请求时, 先说明缺少稳定专项指导, 再按通用证据边界自由发挥。
- 机械工具处理大文件, 不把正文自动塞入 Agent 上下文。

## 2. 文档层次与语义

维护时先判断一条规则属于哪一层, 再决定改哪个文件。不要靠影响表穷举, 因为真实变更永远会超过表格。

### README

`README.md`面向第一次进入仓库的人, 解释项目是什么、能做什么、怎么开始、工作区大概长什么样。

它负责公共说明和示例, 不负责约束 Agent 的细节行为。修改 README 时要问: 用户第一次看到这里, 会不会对仓库定位、推荐入口或产物形态产生不同理解。

### AGENTS

`AGENTS.md`是最高级执行入口, 定义仓库对象、启动顺序、人类决策边界、证据边界、工具上下文边界和不可越过的规则。

它负责“Agent 在这个仓库里不能搞错的大前提”。不要把阶段步骤、某个能力的具体执行法或模板字段塞进 AGENTS。只有当变更会影响所有任务的安全边界、目录模型或启动判断时, 才应修改它。

### ROUTER

`ROUTER.md`是路由表, 定义阶段和能力的公开契约: 做什么, 何时触发, 何时不触发, 需要什么输入, 产出什么, 读哪个指导。

它负责回答“用户这句话应该进入哪个阶段或能力”。不要在 ROUTER 中写详细方法。新增、删除、重命名阶段或能力, 或把某件事从阶段改成能力时, 必须改 ROUTER。

### 使用说明

`指导/使用说明.md`是所有正式阶段和能力共同遵守的协作协议, 包括输入检查、确认点、完成条件、停止条件、证据和模型上下文边界。

它负责“无论进入哪个正式阶段或能力都必须这样协作”。如果一条规则只影响某个阶段或能力, 放到专项指导; 如果会影响所有阶段如何确认、如何落盘、如何处理证据或何时停止, 才放到使用说明。

### 专项指导

`指导/`下除使用说明和参考模板外的正式文件, 是某个阶段或能力的执行方法。它们定义该任务如何读取输入、如何行动、何时让用户确认、完成前自检什么、不该做什么。

专项指导负责“这个阶段或能力具体怎么做”。修改某个阶段的方法、输出口径、确认点或失败模式, 优先改对应专项指导; 若改动影响路由契约, 再同步 ROUTER; 若改动影响共享文档格式, 再同步参考模板。

### 参考模板

`指导/参考/`定义长期文档的语义和建议格式, 例如论文列表、领域概览、研究想法、叙事定稿、实验证明和文献证据库。

参考模板负责“跨阶段反复读写的文档应该长什么样”。它不是 Agent 状态文件, 也不是为了整洁预建空文件的理由。只有当信息需要长期维护、被多个阶段读写, 且现有文档无法合理承载时, 才新增参考模板。

参考模板文件本身必须使用统一写法: 一级标题, 一句简单介绍, `## 固定节`, 一句“产出文件必须严格遵循以下骨架, Agent 绝对不能自行增减...”, 然后用一个代码块放完整产物骨架。不要在代码块后追加字段说明、规则或补充段落; 必要说明直接写进代码块。

### 工具脚本

`scripts/`是机械工具入口, 例如 PDF 翻译、安装或校验脚本。脚本可以处理大文件, 但不应把正文回显给 Agent。

修改脚本时要同步检查调用它的专项指导和用户可见说明, 但不要把脚本参数细节复制到多处形成多个权威版本。

### tmp 归档

`tmp/`被仓库忽略, 可保存旧指导、本地参考资料或迁移中间件。移动到`tmp/`的文件不再是正式仓库规范, 不得被 ROUTER 或正式指导引用。

## 3. 变更判断方法

先问这条变更改变了什么语义:

1. 改变仓库是什么、有哪些对象、Agent 启动时必须遵守什么边界: 改`AGENTS.md`
2. 改变用户请求如何被路由、阶段和能力有哪些、产出契约是什么: 改`ROUTER.md`
3. 改变所有任务共同的确认、停止、证据或落盘协议: 改`指导/使用说明.md`
4. 改变某个阶段或能力具体怎么做: 改对应专项指导
5. 改变一个长期文档的字段、状态、编号或读写语义: 改`指导/参考/`中的模板, 再同步所有读写它的专项指导
6. 改变用户第一次看到的说明、示例或推荐入口: 改`README.md`
7. 改变机械工具行为: 改`scripts/`, 再同步调用脚本的专项指导

如果一项修改同时落在多层, 先改最高层语义, 再向下同步。比如某个阶段的产物字段变化, 先改参考模板, 再改读写该文档的专项指导, 必要时同步 ROUTER 的产出描述。

## 4. 新增阶段、能力和交接文档

### 新增阶段

阶段表示研究主线上的依赖节点, 后续工作通常会读取它的产物。新增阶段前先确认它不是一个可独立调用的能力, 也不是某个阶段内部的步骤。

新增阶段至少要做:

1. 在`ROUTER.md`定义编号、触发、不触发、需要、产出和指导路径
2. 创建专项指导, 写清输入检查、执行方法、确认点和完成条件
3. 若阶段产出新的长期文档, 在`指导/参考/`定义模板
4. 检查 README 是否需要更新用户可见路线
5. 搜索旧编号、旧名称和旧路径, 避免出现两个权威版本

### 新增能力

能力表示用户可随时请求的独立操作, 做完就结束, 不应伪造某个阶段完成。

新增能力至少要做:

1. 确认它能由用户独立触发, 不依赖固定流程门禁
2. 确认它已经足够稳定, 不只是临时试验
3. 在`ROUTER.md`能力表登记触发和不触发条件
4. 创建专项指导, 明确输入、输出、证据边界和确认点
5. 检查它是否会读写已有参考文档, 若会, 遵守对应模板
6. 确保能力完成后不会宣称阶段已经完成

### 新增交接文档

交接文档只有在信息需要跨多个阶段长期维护, 且现有文档无法合理承载时才新增。

新增前必须回答:

- 它的独立责任是什么
- 谁创建, 谁更新, 谁读取
- 编号、状态或字段如何保持稳定
- 新证据如何覆盖、修正或保留历史
- 它为什么不能进入已有文档

实际引用的文献身份默认进入`文献证据库.md`; 文献支撑了哪个共识、问题或相关工作比较, 写在使用它的研究文档里。Idea 与四段式核心主张默认进入`研究想法.md`; 论文叙事和分点核心主张默认进入`叙事定稿.md`; 实验目标、判据、结果状态和产物路径默认进入`实验证明.md`。不新增仅用于 Agent 跟踪进度的元文档。

## 5. 修改时的核对方法

每次文档契约修改至少执行:

1. 搜索旧术语、旧编号和旧路径是否残留
2. 检查 README、AGENTS、ROUTER、通用协议和专项指导是否出现冲突
3. 检查所有反引号中的具体指导文件路径真实存在
4. 用典型场景走查直接入口、跳过、返工和独立能力
5. 运行`git diff --check`
6. 确认没有为了说明规则创建空的工作区模板

目录模型变化时额外检查:

- 是否仍有旧目录名残留
- 是否会导致同一材料复制多份
- 新对象的创建权属于 Agent 还是用户
- 已有工作区是否需要迁移, 以及迁移是否必须由用户确认

## 6. 常见设计失败

- **入口过早承诺**: 还没读 ROUTER 就宣布工具、Skill、目录和方案
- **层级泄漏**: 把专项步骤塞进 AGENTS, 或把状态机细节塞进 README
- **重复权威**: 同一规则在多个文件分别定义, 后续只改了一处
- **阶段膨胀**: 每个工具操作都编号, 导致流程图看似精确但无法灵活调用
- **能力越权**: 独立能力完成后宣称阶段完成
- **路径代替语义**: 仅凭目录名字判断状态, 忽略真实内容和用户确认
- **Agent 替用户分类**: 把论文自动归入某个领域, 混淆技术推断与知识组织决定
- **组合任务被静默拆分**: 用户要求完整结果, Agent 留下半成品却宣称已经推进
- **为整洁而复制**: 同一论文、证据或实验结果在多个目录重复保存, 随后版本漂移
- **工具读取等同模型阅读**: PDF2zh 可在独立进程处理全文, 却把正文回显给 Agent
- **过度严谨化**: 把推荐路线和检查表变成无法跳过的门禁, 牺牲实际研究灵活性

## 7. 维护原则

- 优先修正错误, 再补缺失, 最后才处理轻微不严谨
- 不为了统一格式删除已有有效研究约束
- 规则应足够明确以避免错误, 但不应消灭用户可以主动选择的灵活性
- 所有新增确认点都应回答: 这个决定是否真的只能由用户作出
- 所有新增文档都应回答: 它是否拥有现有文档无法承担的独立责任
- 所有新增目录层级都应回答: 它是否提供真实语义, 而不只是让树看起来整齐
