# 指导文件共同执行协议

本文规定 `指导/` 下所有阶段与按需能力共同遵守的执行方法, 全局边界以 `AGENTS.md`为准, 生命周期, 状态, 能力路由, 产物和确认点以 `WORKFLOW.md` 为准, 专项指导负责具体方法

## 1. 执行宗旨

指导文件不是为了生成看起来完整的文字, 而是把研究行为约束为:

1. **真实**: 事实, 论文, 引用, 实验和投稿规则有可定位来源
2. **可证伪**: 假设和 Claim 说明什么证据会支持, 削弱或推翻它
3. **可追踪**: 关键判断, 能力返回值和用户决定写入工作区
4. **克制**: 不扩大结论, 不隐藏冲突, 不用修辞弥补证据缺口
5. **可交接**: 后续执行者无需依赖聊天记忆即可恢复任务
6. **有限与复用**: 检索和执行有停止边界, 优先复用已有指导, 脚本和外部工具
7. **顺序灵活**: 推荐路线服务于判断, 不把不必要的流程约束变成门禁

## 2. 统一执行协议

### 2.1 先调查可发现信息

1. 按 `WORKFLOW.md` 判断当前请求是生命周期推进, 跳过, 返工还是能力调用
2. 若领域, 论文或研究项目目录存在, 先列出文件并通读相关控制文档, 识别已有结论,
   确认, 跳过项, 未决问题, 最近回写和文档冲突, 不得把“通读 Markdown”解释为自动
   加载论文正文, 译文, 稿件或大体量结果
3. 读取当前阶段或能力的专项指导及其明确引用的辅助指导和示例
4. 检查原始材料和前置产物是否真实存在且内容有效, 标题, 占位符, 无依据结论不构成
   有效输入
5. 能从工作区, 代码或可靠来源确认的事项先自行调查, 不把可发现的问题反问给用户

### 2.2 隔离工具处理与模型阅读

文件被本地命令, 脚本或外部处理服务读取, 不代表其内容需要进入 Agent 的模型上下文
执行时按以下三类访问区分:

1. **机械处理**: 归档, 复制, 哈希, PDF 翻译, 编译和格式转换可以让
   工具读取完整输入, Agent 不打开正文, 只接收退出状态, 文件路径, 大小, 哈希, 计数
   和错误摘要
2. **结构检查**: 检查文件头, 大小, 哈希, 引用键, 编译错误或输出是否存在时, 优先
   运行只输出聚合指标的脚本, 不把正文打印到工具结果
3. **语义阅读**: 摘要, 解释, 证据核验, 审稿, 写作, 翻译质量判断等确实依赖内容时,
   才读取明确相关的页, 节, 段落或有限样本, 能用局部内容完成时不得加载全文

论文 PDF, Tex 稿件和大体量实验结果属于内容产物, PDF2zh 的用户级缓存属于外部工具状态, 不复制进论文目录, 也不作为研究证据, 除非当前目标需要正文语义, 不得用文本提取, 截图, 搜索结果上下文或类似方式把这些内容回显到模型上下文

用户要求“归档并翻译”表示允许已确认的工具链处理正文；它不自动授权 Agent 为了观察过程而阅读正文, 向外部 API 传输全文仍须单独遵守传输确认规则

论文机械处理默认采用 `scripts/translate-pdf.ps1` 调用固定版本 PDF2zh, 确认输入,
输出目录和外部传输后, Agent 只启动命令并接收机器状态, 不插入人工抽查或正文修订
管线失败不能转为 Agent 手工处理, 也不得在本仓库复制, 修改或临时补丁 PDF2zh 源码

### 2.3 锁定目标, 输入和边界

执行前明确:

- 本次目标和成功标准；
- 当前生命周期阶段, 状态, 或能力调用的来源与返回位置；
- 已有输入, 缺失的关键输入和允许使用的证据；
- 处理范围和明确不处理的内容；
- 产物名称, 目标领域/论文/研究项目目录或其他输出位置；
- 用户需要作出的决定和预计确认点；
- 检索, 计算, 时间, 数据许可, 外部服务和工具限制

关键输入不完整时, 先说明缺口及影响, 只询问会改变执行结果的问题, 输入齐备前不得假装进入专项执行, 也不得用 Agent 猜测替代用户必须作出的方向, 范围或资源决定

论文归档的领域归属和最终路径属于关键输入, 用户只提供孤立论文时, Agent 可以根据内容提出候选领域, 但必须询问用户加入哪个已有领域或创建哪个新领域, 不能自行把论文标题, 关键词或推测主题升级为目录决定

用户明确要求跳过前序阶段时, 不要求补走前序流程, 应说明跳过造成的证据, 可复现性,
主张强度或投稿风险, 记录决定后继续；只有目标任务本身无法可靠执行时才阻塞

### 2.4 确认后首次落盘

以下行为发生前必须得到用户确认:

- 创建新的 `workspace/<中文领域名>/`, `workspace/<中文研究项目名>/` 或领域下的
  论文目录；
- 首次写入当前生命周期阶段的主要产物；
- 首次写入独立能力的持久产物且用户尚未指定输出方式；
- 确认候选方案, 研究 Idea, Story, 实验范围, 结果口径, 阶段跳过或重大投稿决定；
- 改变已确认目标, 范围, 主要结论, Claim 强度或输出位置

确认时说明将写入什么, 依据是什么, 有哪些取舍, 用户已批准某阶段或能力及其产物后,
批准范围内的增量回写, 证据补充, 运行记录和错误修正不需要逐次请求许可

“先确认再落盘”和“决定必须回写”是连续动作: 先取得确认, 再把决定, 日期, 依据和影响写入对应文档, 只留在对话中的决定不视为阶段完成或能力交接

### 2.5 执行, 记录与纠偏

- 按专项指导执行最小必要工作, 不静默叠加其他任务
- 检索前定义问题, 来源范围和停止边界；找不到资源时记录查询和失败原因, 不无限递归,
  也不把“未发现”写成“不存在”
- 外部事实进入研究项目或论文前写入 `文献证据库.md`；相近工作的本质比较写入
  `相关工作矩阵.md`
- 使用领域地图 K 条目时, 把原始来源, 准确位置和 K 编号同步到项目证据库, 不能把
  地图摘要当作一手引用
- 实验按批准设计和计划运行, 保存目标, 配置, 命令, 代码版本, 结果, 失败原因和结论
  实现发现研究设计不可行时回写研究文档, 不能只改代码
- 新证据或负结果推翻旧判断时, 指出受影响的阶段, 文档和 Claim；改变已确认口径前
  重新对齐
- 能力调用结束时记录产物, 未决项, 影响和返回位置, 不借能力产物修改阶段状态

### 2.6 阻塞与恢复

遇到影响方向, 证据可靠性, 资源或权限的问题时暂停当前目标, 说明:

1. 被阻塞的具体行为；
2. 已完成且仍然有效的部分；
3. 不能继续的技术原因；
4. 需要用户或外部条件提供什么；
5. 恢复后从哪里继续

某项能力不可用时, 可以继续其他真正独立的目标, 若用户将多个动作作为一个组合目标,
例如“归档并翻译”, 不得自行拆分并落盘部分结果；任一必要动作不可用时整体阻塞,
除非用户明确同意分段完成

### 2.7 完成, 返回与转移

阶段或能力结束前:

1. 检查产物是否包含真实内容和可追溯依据
2. 记录未核验信息, 反例, 失败, 限制和不能被后续当作事实的内容
3. 写入用户确认, 跳过或风险接受及其影响
4. 阶段按 `WORKFLOW.md` 标记状态；能力记录返回位置但不改变主线
5. 只有完成条件成立时才标记已完成；跳过, 终止和阻塞不得伪装为完成

## 3. 证据与判断标签

研究文档至少区分:

- **已核验事实**: 可定位到官方页面, 论文页码/章节, 代码或真实实验记录
- **作者声明**: 作者明确声称的贡献, 局限或结论, 不等于独立证明
- **Agent 推断**: 由材料推导的解释, 写明推理链和不确定性
- **待验证假设**: 仍需检索或实验, 不得提前写成论文结论
- **用户决定**: 用户对方向, 范围, Story, 资源, 跳过和投稿目标作出的选择
- **未核验信息**: 暂时无法确认, 保留来源和核验缺口

日期, 发表状态, 会议政策, 软件能力和其他可能变化的信息使用本次执行时的最新可核验来源, 并记录核验日期, 不能仅凭记忆

## 4. 人类协作与确认

- 方向选择, 最终论文列表, Idea 立项, Story, 实验范围, 结果口径, 阶段跳过和重大
  投稿决定必须由用户确认
- 提问聚焦真正影响决策的信息, 附上已知事实, 可选方向和代价
- 同名中文领域, 研究项目或论文目录冲突时询问有意义的限定词, 不自动追加编号
- 未经用户要求, 不为语义任务全文读取 PDF 或正文, 用户批准机械处理时, PDF2zh 可以读取
  全文, 但 Agent 仍不得把全文加载到模型上下文
- 翻译会将正文发送到外部 API 时, 先说明服务和传输范围并得到许可
- 审稿, 翻译, 引用核验等独立能力不要求用户补建无关生命周期文件；但论文归档必须
  有用户确认的领域归属

## 5. 停止条件

以下情况应暂停当前目标并对齐:

1. 关键文档冲突, 无法从证据判断有效版本
2. 关键输入缺失, 继续只能依赖高风险猜测
3. 关键外部事实无法核验, 且会改变方向或核心主张
4. 新论文显示高度撞车, 或新实验破坏核心假设
5. 实验需要显著扩大预算, 改变数据许可, Benchmark, 主要指标或研究目标
6. 任务将越过登录, 验证码, 付费墙, 许可证或其他访问控制
7. 翻译将发送正文到外部 API, 但用户尚未确认
8. 用户要求的表述强度超过现有证据
9. 当前目标必须依赖不存在的实验仓库, 工具链或提交材料

发现前序阶段存在问题不自动阻塞后续工作, 先说明影响, 由用户决定返工, 跳过, 弱化口径或终止；只有所选目标无法技术实现时才阻塞

## 6. 通用禁止事项

- 不伪造论文, 作者, 机构, DOI, 页码, BibTeX, 源码, 实验数据或日志
- 不把搜索摘要, 二手博客, 模型记忆或未读正文伪装成论文证据
- 不把作者的 Future Work 直接改写为已证明新颖, 重要或可行的 Idea
- 不把格式变化, 模块改名, 多智能体包装或提示词重组自动当作机制创新
- 不因结果不理想而事后替换主要指标, 筛掉失败种子或隐藏负结果
- 不用“显著, 全面, 首次”等词掩盖证据缺口
- 不创建整套空白阶段文档, 复杂状态系统或无实际内容的管理文件
- 不因某项能力不可用而否定仍可独立执行的工作
- 不把官方未要求的伦理评价扩展为仓库主动审查；官方强制声明只核验要求和完成状态

## 7. 实验与写作约束

- 论文主要主张对应 `核心主张.md`, 外部事实对应 `文献证据库.md`, 实验事实对应
  `实验记录/`, 无证据主张必须弱化, 标为假设或删除
- Claim 的证据责任必须明确且可追溯, 多个 Claim 可以共享实验, 一个 Claim 也可以由
  多项证据共同支持；不强制每项 Claim 都有独立实验或图表
- 实验结论只来自真实运行记录, 缺失运行, 排除规则, 失败种子和异常结果必须可见
- 发现方法不可实现, 比较不公平或结果破坏 Story 时, 回写受影响的研究, 叙事, 证明
  设计或实施计划
- 写作不得脱离已确认 Story 和 Claim 增加贡献；翻译, 润色, 缩写和去 AI 痕迹不得
  改变技术事实, 因果, 数字, 限定条件和引用
- R7 允许任意章节顺序和反复修订；顺序自由不等于证据自由
- 新证据推翻旧判断时保留必要变化记录, 不覆盖导致研究转向的历史事实

## 8. 产物质量

阶段文档和能力记录应让下一位执行者回答:

1. 本次试图解决什么？
2. 使用了哪些输入, 证据和执行记录？
3. 做出了哪些判断, 判断强度多大？
4. 哪些内容未核验, 存在哪些反例, 失败或替代解释？
5. 用户确认, 跳过或接受了什么风险？
6. 当前主线状态是什么, 能力应返回哪里？
7. 后续可以使用什么, 不能把什么当作既定事实？

状态使用明确枚举, 需要保留转向历史时追加简短变更记录, 不静默覆盖

## 9. 常见失败模式

- **清单完成主义**: 填满模板却没有回答研究问题
- **摘要替代证据**: 只读摘要便判断机制, 实验公平性或作者局限
- **正例偏差**: 只搜索支持 Idea 的工作, 不主动寻找反例和撞车论文
- **阶段门禁错觉**: 把推荐顺序当成强制流程, 拒绝用户可执行的直接入口
- **能力越权**: 用审稿, 绘图或翻译结果宣称未执行阶段已经完成
- **语言增益错觉**: 通过更强修辞让弱 Claim 看起来成立
- **文件存在错觉**: 把空文件, 旧版本或未核验草稿当作有效输入
- **对话依赖**: 关键决定没有回写, 导致后续无法恢复
- **确认泛滥**: 批准范围内每次小改动都重复询问
- **工具读取错觉**: 因为外部工具需要处理全文, 就让命令回显或 Agent 主动打开正文和
  缓存, 浪费上下文并扩大不必要的数据暴露
- **Agent 接管管线**: 机械处理遇到瑕疵后, Agent 开始阅读全文, 手改译文
  或逐页视觉验收, 使自动任务变成昂贵的交互任务

## 10. 完成前自检

- 当前请求是阶段推进, 跳过, 返工还是能力调用？
- 任务结果, 当前阶段和状态是否从真实材料推导？
- 目标, 输入, 范围, 非目标, 输出位置和确认点是否齐备？
- 首次落盘或改变已确认口径前是否得到确认？
- 跳过决定是否记录原因和证据影响？
- 所有强事实是否有可定位来源？
- 推断, 假设, 用户决定和未核验信息是否区分？
- 是否记录反例, 失败, 风险和未决问题？
- 是否只创建当前需要的产物？
- 能力是否正确返回调用位置, 没有伪造阶段进度？
- 机械处理是否保持在工具侧, 模型上下文只接收必要的聚合结果？
- 下一位执行者能否仅凭工作区恢复任务？
