# 不推荐: 在引言中只写抽象化方法描述

`不推荐: 如果方法很简单, 不要在引言中回避具体方法细节, 只写抽象洞见来让它看起来更新颖,`

判断说明:

1. 这种写作模板的技巧在于如何让一个简单的流水线看起来更有新意
2. 注意: 这不是要让洞见看起来更新颖, 而是要让流水线步骤看起来更新颖
3. 大多数情况下并不推荐这样写
4. 更好的目标是在引言中清楚解释核心贡献是如何实现的

```latex
% 为了解决这个问题, 我们提出一种新的 3D GAN 训练方法, 使其能够生成不受观察角度影响的逼真图像

% 介绍关键想法
% 我们的关键想法如下, 为了缓解学习逼真且多视角一致图像合成这一困难问题, 我们把该问题转化为两个子问题, 而这两个子问题都能更容易地被求解

% 解释关键想法为何有效, 但不要具体讨论完整流水线 (或者只讨论抽象收益) 
%% 示例: 具体来说, 我们把问题形式化为两个简单判别问题的组合, 其中一个学习判别合成图像是否看起来真实, 另一个学习判别合成图像是否与相机姿态一致, 与已有方法试图为每个姿态学习真实图像分布或学习姿态估计的形式不同, 我们的子问题要容易得多, 因为它们都类似于一个基础的二分类问题

% 介绍流水线模块, 但不要清楚说明完整流水线 (或者跳过具体流水线细节) 
%% 示例: 基于这个关键想法, 我们提出一种双分支判别器, 它有两个分支, 分别用于学习逼真性和姿态一致性, 由于这些分支都被明确地监督用于各自的目的, 因此可以在每个观察角度生成具有姿态一致性的高质量图像, 进而生成器也能创建高质量的图像和形状,  (这一段并没有清楚说明流水线是如何工作的, ) 

% 介绍另一个贡献
%% 示例: 此外, 我们提出一种姿态匹配损失, 通过为给定图像考虑一个正姿态 (即渲染姿态或真实姿态) 和一个负姿态 (即无关姿态) 来为判别器提供关于姿态一致性的监督,  (这一段并没有清楚说明流水线是如何工作的, ) 

% 解释相较于已有方法的预期收益
%% 示例: 例如, 对于一张侧视图图像, 正面视角就是其中一个无关姿态, 正如实验所报告的那样, 这个损失有助于提升图像和形状质量, 它可以被理解为把一个包含大量类别的分类问题简化为二分类问题, 由正样本对和负样本对组成
```

## 为什么不推荐

- 抽象洞见可能掩盖方法其实与已有流程相同
- 读者无法判断贡献发生在流程哪一步
- 优势缺少机制解释, 只能依赖宣传性语言

## 只有何时可用

当方法细节确实需要后文定义才能理解时, 可以在引言先给高层洞见, 但仍应提供足够具体的机制轮廓, 并在方法段落明确导航, 不能把“暂不展开细节”变成隐藏简单性的策略
