# 引言

## 引言要完成一次受控的认识转变

引言让目标读者理解现有认识在何处不足、本文为什么采取这种改变、后文将用什么证据让它成立。先从全文结论反推读者缺少的前提, 再按阅读顺序正写。

![引言内容拆解](../images/引言内容拆解.png)

可以从这些问题中寻找必要内容:

- 本文真正解决的技术问题是什么
- 为什么这个问题仍存在, 最关键的技术原因是什么
- 哪些已有路线必须介绍, 才能公平到达这个原因
- 本文的认识和设计怎样直接作用于它
- 哪项结果最能证明这种变化值得读者接受

图和问题描述内容关系。一个读者熟悉的任务可以很快进入矛盾; 一个新设置则需要更多定义。

## 选择进入任务的距离

引言开头提供多少背景, 取决于目标读者与研究对象之间的距离。

小众任务可以先用输入、输出和目标定义:

```text
[Task] aims to recover / estimate [output] from [input] for [relevant use].
```

熟悉任务可以直接进入本文设置或失败条件:

```text
Existing [method class] succeeds under [usual condition] but fails when [target condition] because [reason].
```

一般任务与特定设置之间存在重要差异时, 先定位一般目标, 再明确本文研究哪一个设置。应用例子只保留真正说明研究价值或约束的部分, 不用宽泛行业清单制造重要性。

## 公平地收窄到技术问题

介绍已有路线时, 先承认它们解决了什么, 再说明与本文有关的边界。论文可能按技术路线逐步收窄, 也可能直接从一个共同假设切入; 选择哪种路径取决于读者理解当前问题所需的历史。

一项有用的局限描述包含对象、条件、可观察失败和技术原因, 这些信息可以分布在相邻句段中。技术原因要标清证据身份: 是已有文献事实、本文观察, 还是后续实验要检验的解释。

传统路线若已经包含与本文相近的思想, 直接承认它在原场景中的贡献, 再解释为何不能直接满足当前设置。精确继承关系通常比“所有现有方法都忽略了……”更能建立可信差异。

新任务没有直接既有方法时, 从任务内生困难组织问题, 每项困难都推动后文某项设计或证据。

## 让方法从问题自然出现

引言中的方法介绍要让读者理解方法本体和关键差异, 又不抢先承担复现细节。可以根据研究关系采用不同节奏:

- 一个核心设计带来多项后果: 先讲机制, 再讲可验证优势
- 多项设计存在依赖: 说明前一设计产生什么状态、为何还需要后一设计
- 方法由关键观察推出: 先说明观察怎样改变问题理解, 再引出具体设计

写作前先确认“设计改变了什么运行关系, 为什么能作用到刚才的技术原因, 哪项证据验证了这个变化”。如果只能说“我们增加模块 A/B/C”, 说明引言尚未抓住方法的研究含义。

下面只是一个可裁剪的内容草图:

```tex
\section{Introduction}
% 让目标读者进入任务和本文设置
% 公平呈现已有能力, 收窄到精确失败及其原因
% 交付本文的核心认识和方法变化
% 用决定性设计与结果说明这项变化为什么成立
% 在需要时总结贡献与范围
```

## 贡献总结

是否使用项目符号、写几项贡献由目标场所和研究本身决定。每项贡献应交付一项真实增量: 新认识或设计解决什么精确问题, 改变了什么, 哪类证据支持它。普通工程步骤不因被单独列出就成为贡献, 同一贡献也不拆成多种近义说法增加数量。

完成引言后, 读者应能够预测后文为什么需要当前方法与实验。背景、最相近工作、方法与原因或实验概述出现关系错误时, 先重建相应论证关系。

可参考[任务和应用的引言组织](https://arxiv.org/abs/2212.04965)与[从一般任务进入特定设置](https://openaccess.thecvf.com/content/CVPR2021/papers/Martin-Brualla_NeRF_in_the_Wild_Neural_Radiance_Fields_for_Unconstrained_Photo_CVPR_2021_paper.pdf), 重点观察作者如何根据任务熟悉度选择进入距离。
