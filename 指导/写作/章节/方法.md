# 方法

## 方法节要让设计变得可理解、可复现、可质疑

方法部分从`叙事定稿.md`中的完整方法与设计关系、R3 固定的方法身份和真实实现出发。组织顺序服从读者理解机制所需的依赖。

![方法内容拆解](../images/方法内容拆解.png)

一篇方法节通常需要建立问题设置与整体流程, 再展开承担研究增量的机制和必要实现。小节数量由真实依赖决定:

```tex
\section{Method}
% Problem setup and the minimum overview needed to read the method
% Contribution-bearing mechanisms in dependency order
% Necessary objectives, updates, constraints and implementation details
```

## 先给读者一张方法地图

概览说明任务输入输出、必要符号、核心认识、完整方法和主要数据流。它的详略取决于后续结构复杂度: 简单方法保持简洁, 多阶段系统让读者预先知道状态由谁产生、流向哪里、何时更新。

总体方法图适合展示数据流与依赖, 但正文仍要解释关键箭头和状态。图中名称、数学符号、实验配置和正文术语使用同一含义。

## 展开贡献性设计

理解一项设计通常需要知道它为何存在、实际怎样运行、相对直接替代方案改变了什么。原参考材料把这三类内容概括为动机、设计过程和技术优势:

![方法段落三元素](../images/方法段落三元素.png)

三类内容根据实际关系分布。动机已在概览中统一说明时, 后续小节可以直接进入机制; 优势来自两个设计联合时, 在联合关系处解释。

### 动机来自尚未解决的技术责任

说明当前状态缺少什么、前一机制为何不能处理, 以及新设计承担什么必要责任。避免用“为了进一步提升性能”作为通用动机; 它没有指出设计选择。

```text
A remaining difficulty is [problem under a precise condition].
To change [relevant mechanism or state], we introduce [design].
```

### 运行过程要让状态变化可追踪

根据机制复杂度说明表示、输入来源、操作顺序、中间状态、更新条件、输出和消费者。训练、进化与推理阶段只有在行为不同时才分别展开。

```text
We represent [object] as [representation].
Given [input], [operation] produces [state], which is then used by [downstream operation].
```

公式出现前说明它要表达的关系和符号, 公式之后解释它在系统运行中的作用, 使数学表达进入完整机制说明。

### 技术优势是可被证据挑战的关系

与最直接替代设计比较, 说明机制差异预计改变什么可观察性质, 并让 R3/R4 的实验能够支持或削弱这项判断。

```text
Unlike [alternative], the proposed design changes [mechanism], allowing [observable consequence] under [condition].
```

技术优势落到明确对象、条件和证据, 例如灵活性具体改变了哪一种可观察行为。

## 处理实现细节

模型、维度、优化器、轮次、数值处理、推理步骤、硬件和软件环境等信息按复现需要集中或放入附录。会影响主要结论的设置同时出现在 R3 协议和实验设置中; 研究增量决定哪些设计进入贡献论述。

完成方法节后, 目标读者应能还原核心流程, 理解每项关键设计为什么出现及怎样改变运行, 并把声称的优势连接到实验中的直接证据。正文与正式运行的方法身份出现差异时, 以真实方法为依据纠正正文或返回上游确认。
