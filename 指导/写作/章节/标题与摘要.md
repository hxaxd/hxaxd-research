# 标题与摘要

## 1. 内容来源

标题和摘要在方法、实验结果与结论稳定后完成。写作时读取:

- `叙事定稿.md`中的中心叙事、论文事实和分点核心主张
- `实验证明.md`中的正式结果、成本、成立范围和当前判定
- 正文最终采用的术语与方法名称

下面的骨架表示语义职责, 不规定句数。摘要的 TeX 命令服从当前官方模板。

## 2. 标题

先列出任务、精确问题、关键机制和核心认识, 再选择最能区分本文的必要组合。标题应让目标读者判断论文研究什么和凭什么不同。

检查:

- 只有项目名称时, 是否缺少技术含义
- 效果、泛化或最佳水平判断是否得到正式结果支持
- 标题中的核心名词是否与正文稳定术语一致
- 去掉一个修饰词后含义是否完全不变

## 3. 摘要的最小论证

摘要需要独立交付:

1. 任务和精确设置
2. 已有方法在目标条件下的技术问题及原因
3. 本文的核心认识与方法响应
4. 决定性设计怎样产生技术优势
5. 能够直接支持上述判断的真实结果
6. 结果允许得到的认识或范围边界

### 范式一: 技术问题到技术贡献

适用于核心贡献集中、不需要单独展开核心认识的论文。

```tex
\begin{abstract}
% 说明任务、输入输出和必要设置
% 说明已有方法无法解决的技术问题及其原因
% 用一到两句话介绍核心贡献和运行机制
% 说明该机制带来的可检验优势
% 报告正式实验支持的主要结果与范围
\end{abstract}
```

### 范式二: 技术问题到核心认识再到方法

适用于方法由一个容易解释且有价值的核心认识自然推出的论文。

```tex
\begin{abstract}
% 任务和设置
% 已有方法的技术问题与原因
% 一句话说明核心认识
% 说明怎样把该认识实现为方法
% 说明技术优势
% 报告正式结果与边界
\end{abstract}
```

可用于检查信息关系的英文骨架:

```text
This work studies [task] under [setting].
Existing methods rely on [assumption], which causes [failure] when [condition].
Our key observation is that [observation].
Based on this observation, we propose [method], which [core mechanism].
This design enables [measurable advantage].
Experiments on [datasets / settings] show [supported result and boundary].
```

### 范式三: 多项贡献分别交付关系

适用于几项贡献相对独立, 且每项都承担中心叙事中的必要责任。

```tex
\begin{abstract}
% 任务和必要的问题背景
% 贡献一: 设计、机制与优势
% 贡献二: 设计、机制与优势
% 按实际需要继续, 不固定数量
% 正式实验支持的主要结果
\end{abstract}
```

每项贡献都写成完整关系:

```text
We introduce [contribution], which [mechanism and advantage].
```

## 4. 结果句

结果句优先包含决定论文判断的信息:

- 主要任务、数据集或迁移设置
- 与强基线在公平协议下的主要差异
- 与新机制直接相关的现象
- 关键成本、稳定性或适用边界

结果句中的数字、比较对象和统计口径与主表及正文逐项一致。最佳水平判断只在可比协议下成立。

## 5. 自检

- 一遍阅读后能否准确复述任务、问题、方法和结果
- 技术问题是否正是方法实际解决的问题
- 方法名和新术语第一次出现时是否信息自足
- 每个强主张是否能指向正文中的具体证据
- 摘要是否遗漏正文最重要的发现或边界
- 是否有一句同时承担过多职责
- 标题、摘要和结论是否表达同一中心判断
