# 文献检索与收录

## 1. 做什么

为领域名、研究方向、具体问题或单篇论文输入建立可核验、可去重的候选池。Agent 必须为每篇候选补全当前服务器契约要求的全部字段，并立即以 `discovered` 状态提交目标项目，使用户能在页面中看到和比较候选。用户随后通过对话告诉 Agent 哪些保留、排除、归档或修订，Agent 再更新服务器记录。

本能力不生成候选 Markdown、论文列表 Markdown 或其他领域文件，不直接读写 SQLite，也不在提交前把候选藏在对话或本地文件中等待用户确认。

## 2. 输入

输入可以是以下任意形式：

- 一个宽领域名、中等方向、具体研究问题或待验证假设
- 一篇论文的标题、作者或稳定标识符
- 一篇用户提供的 PDF
- 时间范围、目标 venue、任务类型和允许的相邻领域
- 用户要求的侧重，如最新、高影响、奠基、系统、Benchmark、反例或最小知识闭环
- 目标服务器地址以及已有项目，或需要创建的新项目名称

侧重会改变查询、覆盖结构和排序，不降低书目信息、内容判断和来源核验标准。

## 3. 开始前确认

### 3.1 确认研究范围

- 输入是宽领域、具体子方向、某项机制、单篇论文，还是一个待验证假设
- 时间范围、目标 venue、任务类型和允许的相邻领域
- 用户是想建立系统覆盖、补充新近工作，还是定位少量关键论文
- 用户是否指定篇数或特殊侧重

仅当这些差异会显著改变检索结果时询问用户；能从公开输入安全推导时直接说明假设并继续。

### 3.2 确认服务器目标

1. 确认研究工作台后端可访问，默认地址为 `http://127.0.0.1:8000`。
2. 读取 `GET /api/projects`，确认目标项目，避免在近似项目间自行选择。
3. 新目标需要创建项目时，先确认有意义的名称，再调用当前服务器接口创建。
4. 读取目标项目已有论文，使用稳定键和论文身份去重；已有记录进入补充或状态更新，不重复创建。

服务器不可用时可以继续完成检索调查，但不能把对话中的候选声称为已收录；恢复服务器后才能完成本能力。

### 3.3 读取当前字段契约

每次生成提交数据前读取 `GET /api/schema/paper`。字段类型、必填项和允许值以该响应为唯一权威，字段研究语义见 `指导/参考/论文记录字段.md`。不得根据记忆猜测数据库列，也不得直接连接或修改 SQLite。

## 4. 检索方法

### 4.1 构造四组检索问题

不要只把用户原句放进搜索框，先建立四组查询：

1. **问题组**：目标问题、失败现象、常见同义表达和上位概念
2. **机制组**：可能的方法名、机制作用、替代实现和历史术语
3. **任务组**：数据集、Benchmark、评价指标、应用场景和系统名称
4. **反证组**：`limitation`、`failure`、`negative result`、`benchmark leakage`、`revisit`、`does not improve` 等可能揭示反例的表达

查询应随着新术语、引用链和反例增量展开，不预先穷举所有组合，也不因单篇综述的分类方式固定整个领域边界。

### 4.2 来源与核验顺序

1. 会议、期刊、出版社、arXiv、OpenReview、PubMed 等稳定学术页面
2. Crossref、OpenAlex、Semantic Scholar 等聚合服务，用于发现、补全和交叉核验
3. 作者主页、项目页和官方代码仓库，用于确认版本关系、组织、代码与补充材料
4. 博客、媒体、排行榜和非官方列表只能提供线索，不能作为最终书目信息来源

每篇候选至少核验：

- 官方英文标题和完整作者顺序
- 首次公开年份或正式发表年份
- 预印本、venue、track 和版本关系
- DOI、arXiv、OpenReview 等稳定标识符
- 官方或稳定论文入口
- 主要方法、真实贡献和与当前项目的关系
- 第一作者组织、代码仓库和项目页；无法可靠确认的可空字段写 `null`

### 4.3 候选池覆盖

候选池应根据目标覆盖以下角色，而不是只按引用量或榜单排序：

- 建立术语和问题背景的入口材料
- 奠定主要技术路线的工作
- 代表当前能力边界的系统或方法
- 定义任务、数据集和评价方式的 Benchmark
- 暴露限制、争议、失败或失效条件的反例
- 与用户目标直接相关的前沿和相邻工作

宽领域通常检索 12 至 25 篇，中等方向 8 至 18 篇，具体问题 6 至 12 篇，最小闭环 5 至 8 篇。用户明确指定数量时按指定数量交付，每一篇都必须是完整记录，不能用轻量占位行凑数。

### 4.4 停止规则

- 已覆盖与任务相称的入口、奠基路线、代表方法、关键 Benchmark、主要前沿和明显反例
- 新增结果连续若干轮只重复已有路线，不再带来新的机制、分类或证据
- 已达到用户指定或当前目标所需的候选规模
- 某篇资源或事实难以核验时记录真实缺口，不为补齐数量无限递归搜索

停止检索不表示领域已经穷尽。向用户说明本轮时间边界、来源范围、主要查询方向和未覆盖缺口，但这些说明不替代数据库中的逐篇完整记录。

## 5. 立即提交候选

### 5.1 生成临时 JSONL

检索和逐篇核验完成后，立即为本轮所有候选生成 JSONL：

- 默认状态统一为 `discovered`
- 一行对应一篇完整论文记录
- 每行独立满足当前 `GET /api/schema/paper` 契约
- 临时文件放在仓库已忽略的 `tmp/`，不放进 `workspace/` 或领域目录
- JSONL 只承担一次批量传输，不是候选清单，不要求用户先阅读或确认

### 5.2 批量提交

在后端目录使用当前提交脚本：

```powershell
uv run python scripts/submit_papers.py ..\tmp\<本轮文件>.jsonl --project-id <项目ID>
```

需要连接非默认服务器时显式增加 `--base-url`。提交由后端执行契约校验和整批事务；任何一行失败时不得声称其余论文已经收录，应根据错误修正整批或明确缩小重试范围。

提交成功后：

1. 检查返回创建数等于 JSONL 有效行数
2. 读取项目论文接口，确认本轮记录均已出现且状态为 `discovered`
3. 删除临时 JSONL
4. 告诉用户候选已经可以在页面中查看，并简要说明检索覆盖和已知缺口

### 5.3 用户筛选后的状态更新

当前页面只负责展示，用户通过对话告诉 Agent 筛选决定。Agent 根据用户明确指令逐条调用：

```text
PATCH /api/papers/{paper_id}
```

状态含义：

- `discovered`：已发现、字段完整，等待用户判断
- `included`：用户决定保留在当前项目
- `excluded`：用户明确不保留为当前入选论文，但记录仍留在数据库中避免重复发现
- `archived`：曾经有用，当前不参与主要视图或工作流

“不留”默认解释为更新为 `excluded`，不是删除数据库记录。用户同时修订中文标题、类型、阅读重点或其他允许修改的字段时一并 PATCH；未获授权时不替用户改变筛选状态。

## 6. 单篇论文与用户提供的 PDF

- 单篇标题或稳定标识符仍按同一字段契约创建完整记录；用户已明确要求收录时可直接使用 `included`，否则使用 `discovered`。
- 用户提供 PDF 时，先从 PDF 和稳定来源核验论文身份并创建或复用服务器记录。
- PDF 本身不在本能力中复制到 `workspace/`。记录就绪后调用 `论文资源获取`，由该能力把用户文件上传到服务器论文资源接口。

## 7. 产出

- 目标项目数据库中新增的完整论文记录，默认状态为 `discovered`
- 用户筛选后按明确指令更新的 `included`、`excluded` 或 `archived` 状态
- 对话中的检索覆盖、来源边界、已知缺口和服务器提交结果摘要

不生成服务器之外的候选清单、论文索引、学习领域目录或永久 JSONL 文件。

## 8. 完成条件

- 本轮计划展示的每篇候选都已完成全部必填字段和必要来源核验
- 全部候选已提交目标项目，用户能够在页面中看到
- 服务器记录数、临时 JSONL 行数和提交返回创建数一致
- 临时 JSONL 已删除，不存在数据库外的第二份候选清单
- 用户已经给出筛选决定时，对应记录状态已更新并回读验证

## 9. 不应做

- 不在用户可见之前把候选留在 Markdown、对话草稿或未提交 JSONL 中等待确认
- 不直接读写 SQLite，不要求 Agent 理解数据库表结构
- 不生成或维护旧的论文列表、候选列表和领域目录
- 不因候选状态是 `discovered` 就省略方法、贡献、选入原因、阅读重点或关系字段
- 不把引用量、上传时间、高关注度或大机构作者当作质量或相关性的直接指标
- 不因论文支持当前 Idea 就提高可信度，也不隐藏否定性结果
- 不把未核验的作者单位、发表状态或投稿传闻写成事实
- 不为达到篇数加入角色重复、质量可疑或仅标题相关的论文
- 不在没有用户筛选指令时擅自把 `discovered` 改为 `included` 或 `excluded`

## 10. 常见失败

- 仍沿用“先写候选文件、用户确认、再入库”的旧流程，导致用户在页面里看不到候选
- 只提交标题和链接，把数据库当成待办清单而不是完整论文记录
- 搜索范围过窄，只使用用户原句而忽略同义表达、上位概念和反证查询
- 只选成功方法，没有 Benchmark、失败分析和争议工作
- 未检查项目已有稳定键，导致整批提交因重复记录失败
- 提交成功后未回读服务器，实际项目或记录数错误却报告完成
