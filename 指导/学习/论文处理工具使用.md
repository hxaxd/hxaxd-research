# 论文处理工具使用

## 能力目标

把用户批准的论文正文机械地解析、翻译、重排和渲染为中文阅读材料。该过程
不判断论文可信度和研究价值，也不通过翻译或重排改写作者语义。

## 前置

- 已选定合法全文。
- 用户明确要求处理全文。
- 翻译前确认用户允许将正文发送给配置的 API。
- 目标目录由用户指定；生命周期任务默认使用
  `workspace/<中文任务名>/论文资料/<论文中文短标题>/`。

## 命令路线

```text
uv run python -m src.hxr.cli doctor
uv run python -m src.hxr.cli parse <pdf> --out <论文目录>
uv run python -m src.hxr.cli translate <输入> --out <输出> [--format auto|markdown|html|tex] [--target zh-CN]
uv run python -m src.hxr.cli reflow <输入> --out <输出> [--format auto|markdown|html|tex]
uv run python -m src.hxr.cli render <输入> --out <pdf> [--format auto|markdown|html]
```

Tex 由论文项目自己的 Tex 工具链编译，不交给 `render` 子命令。

## 执行流程

1. 按需调用 `指导/学习/论文处理工具配置.md` 并运行
   `uv run python -m src.hxr.cli doctor`。
2. 记录输入文件哈希、目标目录和本次处理范围。
3. PDF 使用 `parse` 子命令，检查 `原文.pdf`、`原文.md`、`图片/` 和 `.hxr.json`。
4. 对解析结果做抽样验收：标题层级、段落顺序、公式、表格、图片路径和分页锚点。
5. Markdown、HTML 或 Tex 使用 `translate` 子命令。扩展名不明确时显式指定
   `--format`，输出格式必须与输入一致。
6. 翻译后核对片段数量、标题、公式、代码、链接、图片、引用、标签和资源路径。
7. 使用 `reflow` 子命令进行确定性版式整理；重复执行应得到相同结果。
8. Markdown 或 HTML 使用 `render` 子命令生成中文 PDF，并抽查字体、分页、表格和图片。
9. 每一步验收通过后才进入下一步；失败时保留 `.hxr.json` 和日志中的失败信息，
   不伪造后续输出。

## 缓存与失败恢复

- `.hxr.json` 是机器状态文件，不作为研究结论来源。
- 同一操作的缓存应由操作、格式、输入哈希、配置哈希和输出路径区分。
- 命中缓存时仍需确认目标输出存在且可读。
- 结构校验失败时不得把部分译文覆盖为成功输出。
- parse 使用临时目录和原子替换；失败后检查是否遗留半成品，并按工具行为处理。
- 人工修订过输出后，重新运行可能覆盖内容，执行前必须确认目标路径。

## 翻译边界

- 翻译前明确告知正文会发送到配置的 API，并取得用户同意。
- API 凭据只从环境变量读取，不写入文档。
- 公式、代码、命令、引用键、标签和链接目标不得翻译。
- 对无法稳定解析的结构停止并记录，不用“看起来差不多”的文本覆盖原文。

## 常见陷阱

- OCR 成功不代表阅读顺序正确。
- PDF 能渲染不代表公式、表格和图片引用完整。
- reflow 只能整理结构，不能承担润色和语义修复。
- 缓存命中不代表当前配置和输出路径符合本次目标。
- 批量任务中单篇失败不应被汇总表掩盖。

## 输出

更新 `论文处理记录.md`，记录输入、格式、命令、各阶段输出、缓存命中、失败原因
和人工修正。标准论文目录包含 `原文.pdf`、`原文.md`、`中文译文.md`、
`中文重排.md`、`中文译文.pdf`、`图片/` 和 `.hxr.json`。

## 完成条件

每个标为成功的阶段都有实际输出和抽样验收；失败有可复现命令与原因；原文始终保留；
翻译和重排没有改变受保护结构或资源路径。能力完成后返回原调用位置，不改变生命周期
阶段。
