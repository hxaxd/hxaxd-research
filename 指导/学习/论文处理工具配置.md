# 论文处理工具配置

## 能力目标

检查并逐步补齐论文处理 CLI 的最小运行环境，不一次性把所有环境问题丢给用户。
本能力只解决当前处理目标需要的环境，不把“安装所有可选依赖”当作成功标准。

## 输入

- 当前操作系统、Python 与硬件环境。
- 根目录 `config.toml`（若存在）。
- `uv run python -m src.hxr.cli doctor` 的检查结果。
- 用户当前要执行的命令及输入格式。

## 检查顺序

1. Python、uv 和项目虚拟环境。
2. `src.hxr.cli` 源码模块是否可执行。
3. PaddlePaddle、PaddleOCR 和配置设备。
4. Playwright 与 Chromium。
5. 根目录 `config.toml`。
6. 翻译端点、模型和 API Key 环境变量。
7. 字体、Tex 或其他仅在目标任务需要时才检查的外部工具。

## 执行流程

1. 先确认当前任务只需要 parse、translate、reflow、render 中的哪些能力。
2. 运行 `uv run python -m src.hxr.cli doctor`，保留检查结果，但不暴露任何密钥值。
3. 将结果分成“可用、当前阻塞、当前不需要、需用户决定”四类。
4. 对第一个阻塞项给出最小修复命令和预期验证方式。
5. 修复后重新运行 doctor 或对应的无付费验证命令。
6. 每次只推进一个依赖层，避免用户无法判断哪一步解决了问题。

## 行为边界

- API Key 只从环境变量读取，不写入仓库或运行记录。
- `doctor` 不调用付费 API、不下载模型、不显示密钥。
- GPU 不可用时不得悄悄回退 CPU；由用户决定修改配置还是安装环境。
- 不主动下载 OCR 模型或大型运行时；需要下载时先说明体积、用途和替代方案。
- 不因 render 不可用而阻塞 parse，也不因翻译配置缺失而否定 reflow。
- 某个工具不可用时明确标记，不阻塞其他可独立执行的阶段或能力。

## 输出

给出当前可用能力、阻塞项、最小修复命令和验证命令。涉及任务处理时，将环境问题
写入 `论文处理记录.md`。

推荐记录：

| 能力 | 状态 | 检查依据 | 阻塞原因 | 下一步 | 是否需用户决定 |
|---|---|---|---|---|---|

## 常见陷阱

- 把“Python 包可导入”误认为 OCR 模型和设备一定可运行。
- 配置写了 GPU，但实际设备不可用时静默改用 CPU，导致时间和成本失控。
- 只检查 Playwright 包，未检查 Chromium 浏览器。
- 为测试 API Key 发起付费翻译请求。
- 把密钥、完整环境变量或含凭据的配置粘贴进日志。

## 完成条件

当前目标命令的必要依赖均可验证；仍不可用的能力有明确原因和最小下一步；任何设备、
大型安装和 API 配置决策已经交由用户确认。
