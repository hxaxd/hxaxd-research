# 论文资源获取

## 1. 做什么

为服务器中已有完整记录、但尚无原文资源的论文寻找或接收合法、完整、身份明确的 PDF。核验论文身份、版本和文件完整性后，通过研究工作台后端接口上传为该论文的 `original` 资源。

本能力不在 `workspace/`、领域目录或任何数据处理目录中保存长期 PDF，不直接写 `backend/data/`，也不负责翻译。服务器是论文资源的唯一正式落点。

## 2. 输入

- 目标服务器地址，默认 `http://127.0.0.1:8000`
- 服务器中已经存在的目标论文记录及其 `paper_id`
- 需要获取全文的具体论文范围
- 每篇论文已核验的标题、作者、稳定标识符和版本信息
- 可选的用户本地 PDF

论文尚未存在于服务器时先调用文献检索与收录，完成完整记录提交后再返回本能力。用户提供本地 PDF 时本能力直接接收，不再把它路由回文献检索。

## 3. 开始前检查

### 3.1 确认论文记录

1. 读取 `GET /api/papers/{paper_id}`，确认记录存在且身份与请求一致。
2. 读取 `GET /api/papers/{paper_id}/artifacts`，确认是否已经存在 `original`。
3. 已有原文时默认复用，不重复下载或覆盖。用户明确要求更换版本时，先说明现有版本与候选版本差异并取得确认。

### 3.2 确认资源范围

- 用户明确指定的单篇或一批论文才进入获取
- 不因为项目里存在其他缺少原文的论文就自动全部抓取
- 多版本可用时比较正式版本、作者接受稿和预印本的差异，由用户决定有实质差异时采用哪一版

## 4. 获取来源

### 4.1 来源顺序

按以下顺序寻找公开、合法 PDF：

1. arXiv 等正式预印本平台的官方 PDF
2. 出版方、会议、期刊或 OpenReview 的开放全文
3. 作者主页、机构仓储公开的作者版本

非官方镜像、搜索缓存和内容聚合站只能提供线索。若最终只能取得来源存疑的材料，说明风险并由用户决定，不把便利性当作合法性或版本可靠性的证据。

### 4.2 最终资源限定

- 正式资源必须是完整主论文 PDF
- supplement 不能代替主论文
- 只能取得 TeX、HTML、摘要、幻灯片或介绍页时，明确报告未获得可上传的完整 PDF
- 不把其他格式转换后冒充作者或出版方提供的 PDF

## 5. 身份、版本和文件核验

下载或接收用户文件后，在服务器上传前完成以下检查：

### 5.1 文件完整性

- 文件以 `%PDF` 开头
- 文件末尾包含 `%%EOF`
- 文件大小合理，非异常小
- 文件能够被 PDF 工具正常读取，不是 HTML 错误页或登录页改名

### 5.2 论文身份

- 标题与服务器记录一致，允许正式版本与预印本的标点差异
- 作者集合和顺序不存在无法解释的冲突
- DOI、arXiv ID、OpenReview ID 或正文元数据能与目标记录对应
- 不能只凭文件名判断论文身份

### 5.3 版本判断

- 多版本内容等价时优先稳定、公开、来源清楚的版本
- 正式版本与预印本存在实质章节、实验或结论差异时告诉用户
- 目标服务器已有不同版本时不静默替换

## 6. 临时文件与服务器上传

### 6.1 临时文件

- 网络下载使用操作系统临时目录，只为核验和上传存在
- 不在 `workspace/`、研究项目、论文目录或长期数据目录中建立副本
- 用户提供本地 PDF 时不移动、删除或覆盖源文件，可在完成核验后直接读取上传
- 下载失败或上传结束后清理本轮创建的临时文件

### 6.2 上传接口

通过当前后端的 multipart 文件接口上传：

```text
POST /api/papers/{paper_id}/artifacts/original
字段名: upload
内容类型: application/pdf
```

Agent 只调用服务器接口，不推断服务器内部目录，不直接向 `backend/data/artifacts/` 复制文件。后端负责 PDF 二次校验、稳定命名、散列计算、原子替换和数据库登记。

### 6.3 上传后验证

1. 检查接口返回状态成功，返回资源类型为 `original`
2. 回读 `GET /api/papers/{paper_id}/artifacts`，确认存在对应记录
3. 通过原文读取接口确认资源可访问且内容类型为 `application/pdf`
4. 多篇任务逐篇记录成功、失败和原因，不把部分成功报告为全部完成

## 7. 失败处理

- 只有摘要、介绍页、TeX 或 HTML：报告“未获得合法完整 PDF”
- 所有公开来源不可用：请用户提供本地 PDF；收到后仍由本能力核验并上传
- 论文身份无法确认：停止该篇上传，说明冲突字段和需要的证据
- 服务器不可用：可以保留用户原文件，但不把临时下载当成正式产物；恢复服务器后重新核验并上传
- 上传接口拒绝文件：保留必要错误摘要，清理 Agent 创建的临时文件，不直接写服务器目录绕过校验

## 8. 产出

- 目标论文在服务器中的 `original` PDF 资源记录
- 对话中的来源、版本、上传状态和失败原因摘要

不生成学习论文目录或任何长期本地副本，正式产物只存在于服务器论文资源中。

## 9. 完成条件

- 每篇请求论文的合法完整 PDF 已核验并通过服务器接口登记为 `original`，或者已经准确报告无法获取的原因
- 服务器资源列表和读取接口能够确认已上传文件
- 用户源文件未被移动、删除或覆盖
- Agent 创建的临时下载已清理
- 未触发任何 Agent 翻译步骤；翻译由用户在页面中启动

## 10. 不应做

- 不批量抓取用户未指定的全部论文
- 不把非官方来源材料无条件当作正式版本
- 不把 supplement、TeX、HTML、摘要或介绍页作为原文资源上传
- 不在尚未确认论文身份和版本时上传
- 不静默覆盖服务器已有原文
- 不在 `workspace/` 中创建学习领域目录或 PDF 档案
- 不直接操作 SQLite 或服务器文件目录
- 不在资源获取完成后替用户启动翻译

## 11. 常见失败

- 延续旧流程，把 PDF 放进 `workspace` 的论文目录，导致页面和数据库看不到资源
- 文件名相同就认为论文相同，忽略作者主页旧稿和正式版本差异
- 搜索结果标注“PDF”就认为链接公开合法
- 上传成功后不回读资源接口，数据库记录或文件实际不存在却报告完成
- 找不到全文时无限递归搜索，没有及时说明真实缺口
