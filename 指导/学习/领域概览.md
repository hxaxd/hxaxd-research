# 领域概览

## 1. 做什么

解释领域如何组织, 为什么选择这些论文, 论文之间是什么关系, 以及当前真正值得关注的研究入口, 概览是持续增长的知识索引, 不是逐篇精读记录

## 2. 输入

- 可用的 `论文列表.md`
- 已核验的摘要, 综述结构, 论文证据和处理材料

直接进入领域概览时, 用户也可以提供等价的领域范围, 知识问题和已知来源, 不要求为了进入本阶段补建前置文档

## 3. 执行

### 开始前: 网络搜索与缺口分析

论文列表提供了领域基础, 但列表中的论文发表时间有先后, 较新的子方向, Benchmark 或共识变化可能未被覆盖. 论文本身偏严肃保守, 而技术媒体, 行业报告和开发者社区往往更早捕捉到趋势转向. 在撰写任何内容之前, Agent 必须:

1. 检查论文列表中最新的论文发表年份, 判断覆盖的时间窗口
2. 同时从两类来源搜索最新进展:
   - **论文**: 近 1-2 年新出现的子方向, 新 Benchmark, 论文列表发表后的重要后续工作或反驳
   - **媒体**: 技术博客, 行业报告, 头部实验室的技术发布, 开发者社区讨论——这些更早暴露趋势转向, 新术语和未成形但正在凝聚的范式
3. 搜索结果用于补充论文列表未覆盖的缺口, 写入对应节
4. 论文列表中的论文是领域基础, 搜索结果补充时效性——两者结合, 不替代

### 0. 判断是否已有概览

先检查领域目录下是否已有 `领域概览.md`: 若无则从头构建; 若已有则以增量方式将新论文融入现有结构, 不推翻已有子方向边界和论文关系重建

### 1. 这个领域在研究什么

直接描述领域本身——一句话定义, 研究对象, 典型输入输出, 明确排除的相邻问题及排除原因. 若术语存在多个含义, 声明本领域采用哪一种

禁止出现 "本概览将 X 定义为..." "本版依据..." "本概览采用..." 等自我指涉——内容只描述研究领域, 不描述文档本身

### 2. 为什么这个领域重要

从能力落差说明: 缺少这个领域时哪些问题无法解决, 哪些能力无法获得. 不以论文数量或热度论证重要性

### 3. 主要子方向与技术路线

将论文按子方向分组, 每个子方向内描述技术演化——子方向和演化是同一件事: 子方向由问题定义, 演化由论文之间的推进关系构成. 严格按以下格式组织每个子方向:

```md
### 3.X <子方向名>

**定义**: <一句话定义>

**技术演化**:
- <旧限制> → <论文A> 提出 <新路线> → <论文B> 在此基础上 <延续/扩展/反驳/提供评测>
- ...

**边界**: <什么不算在内>
```

划分依据来自问题, 假设, 机制而非论文自称的方法名. 演化关系词限用: 奠定/延续/反驳/扩展/提供评测/暴露边界. 每条演化箭头说清什么旧限制促成了什么新路线. 同一论文可出现在多个子方向中, 须说明原因

### 4. 当前发展状态与关键问题

将领域当前状态按三类组织, 每类说明对应的 Benchmark 和关键问题:

- **4.1 已解决的问题**: 方案成熟, 哪些 Benchmark 已饱和或接近饱和, 在什么设定和证据范围内可以认为已解决
- **4.2 稳步推进的问题**: 范式明确, 在哪些活跃 Benchmark 上持续改进, 当前瓶颈在哪里
- **4.3 仍在探索的问题**: 尚无公认范式或评价方式本身还在形成中, 对应哪些早期 Benchmark 或评测尝试, 为什么现有方法不够

三类之间不是进度条——一个子方向可能同时有已解决和还在探索的子问题. Benchmark 是状态的证据: 已解决问题对应饱和 Benchmark, 推进中对应活跃 Benchmark, 探索中对应仍在定义的评测方式

## 4. 产出

生成`领域概览.md`, 格式详见`指导/参考/领域概览格式.md`

领域概览定稿后, 检查领域目录下论文子目录是否已有 PDF: 若多数论文尚无 PDF, 向用户说明当前论文档案覆盖不足, 询问是否需要调用 `论文资源获取`

## 5. 确认点

- 无

## 6. 完成条件

概览形成可解释的知识结构, 不是逐篇论文摘要拼接, 用户能据此说明主要路线, 代表论文, 评价方式和至少一个尚未解决的真实问题

## 7. 不应做

- 不在正文中出现 "本概览" "本版" "本文" "本综述" 等自我指涉——内容只描述研究领域本身, 不描述文档自身
- 不在写作领域概览时仅依赖论文列表而不搜索网络补充最新进展——论文列表提供基础, 网络搜索补充时效性缺口
- 不把综述的分类原样复制而不检查其时效与偏见
- 不把作者自称 SOTA 当作领域共识
- 不把一个 Benchmark 上的领先结果推广为问题已经解决
- 不为生成选题而夸大开放问题

## 8. 常见失败

- 推荐顺序和论文列表顺序完全相同, 没有根据概览结构重新设计阅读路径
- 推荐顺序和论文列表顺序完全相同, 却没有明确的领域边界和问题意识
