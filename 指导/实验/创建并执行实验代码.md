# R5 实验执行与记录

## 1. 做什么

按已经确认的实验计划实现代码并产生真实执行记录

编码服务于研究证明, 不是独立的软件重构任务, 实现发现研究设计不可行时, 必须回写研究文档, 不能通过静默改变指标, 数据或机制让代码“跑起来”

## 2. 输入

- `研究想法.md`
- `叙事定稿.md`
- `核心主张.md`
- `实验证明设计.md`
- `实验编程计划.md`
- 计划指定的 `labs/<仓库目录>/` 中的 `AGENTS.md` 和工程规范

目标仓库必须已经由用户手动拉取到 `labs/`, 仓库缺失时停止并请用户准备, 不得执行clone, git init 或创建替代仓库

## 3. 执行

### 1. 建立工程基线

核对 `实验编程计划.md` 记录的上游 URL, 分支和基线 commit, 并检查独立仓库自己的执行规范, 随后检查现有代码, 依赖, 数据, 测试和运行环境, 运行最小已有测试或样例,
确认修改前基线, 仓库可能包含用户修改, 不得回退无关变更

### 2. 实现最小闭环

先实现数据读取, 单个方法/Baseline, 指标, 日志和结构化输出, 使用小规模数据验证:

- 输入输出形状和样本身份
- 指标方向和单位
- 随机种子是否生效
- 日志与配置是否完整
- 结果文件是否不会覆盖
- Claim 所需中间行为是否可观察

### 3. 实现正式实验

按计划逐步加入方法, Baseline, 配置, 评测和恢复机制, 对共享行为添加与风险匹配的测试, 每次范围变化都要确认是否仍符合 R3/R4（确认点）

### 4. 建立单次实验记录

每次真实运行创建 `实验记录/<实验编号>-<中文短标题>.md`（确认点）

同一实验的重跑使用可辨识记录, 不覆盖失败或旧配置

代码, 配置, 日志, 原始结果和工程产物保留在独立实验仓库中, 研究项目目录只保存上述研究级记录和交接文档, 不复制或重组实验仓库内部产物, 也不要求其采用固定布局

### 5. 处理失败与设计冲突

真实失败同时更新 `失败记录.md`, 若发现指标无效, 数据不允许, 机制不可观测,
Baseline 无法公平比较或预算显著变化, 停止扩展, 回写 `实验证明设计.md` 或
`实验编程计划.md`, 并在需要时请求用户确认（确认点）

### 6. 聚合与导出

聚合脚本从原始运行文件读取, 不手工抄表, 表格和图表数据应保留实验编号, 种子,
单位和筛选规则, 排除运行必须有预先规定的理由并写入记录

## 4. 产出

- `实验记录/<实验编号>-<中文短标题>.md`, 至少记录:
    - 目标 Claim 和实验目的
    - 实验仓库路径, 上游 URL, 分支, 实际 commit
    - 环境, 数据与模型版本
    - 完整配置, 随机种子和实际命令
    - 开始/结束时间, 设备和资源消耗
    - 日志, 原始结果和结构化输出在实验仓库中的具体路径
    - 运行状态: 成功 / 失败 / 中断 / 部分完成
    - 直接观察, 初步解释和不能得出的结论
    - 下一步决定

## 5. 确认点

- 仓库缺失时停止并请用户准备
- 每次范围变化都要确认是否仍符合 R3/R4
- 创建实验记录后确认记录完整性
- 设计冲突时并在需要时请求用户确认

## 6. 完成条件

结果来自真实执行, 可复现设置完整, 原始结果可追溯, 失败原因被保留, 代码, 配置和研究文档保持一致, 任务内实验记录能够通过仓库路径, commit, 命令和内部产物位置定位真实结果, 所有计划内实验要么完成, 要么有用户可理解的停止或变更记录

## 7. 不应做

- 不替用户创建, 初始化或拉取实验仓库
- 不把独立实验仓库的内部结构强制改造成研究项目目录结构
- 不补造未运行的结果, 不把 dry run 当作正式实验
- 不只保存最好种子或最好 checkpoint
- 不在看到结果后无记录地更改超参数, 指标或数据拆分
- 不把环境/实现错误直接解释为假设失败, 也不反过来
- 不为了赶进度跳过 Baseline, 日志和原始结果保存
- 不只改代码而不更新受影响的研究文档

## 8. 常见失败

- **命令记录不一致**: 命令记录与实际执行配置不一致
- **日志覆盖**: 日志路径正确, 但每次运行覆盖同一文件
- **聚合乐观**: 聚合脚本默认忽略失败运行, 导致结果偏乐观
- **跳过验证**: 最小闭环通过后直接扩大运行, 没有先验证指标与对照公平性
- **机制漂移**: 代码实现改变了机制, 却仍沿用旧 Claim
