# R4 实验执行与结果交付

## R4 的核心边界

R4 把已经确认的证明方案变成真实、可追溯的运行与结果。代码结构、依赖、缓存、调度和并行方式由 R4 根据`labs/<仓库目录>/`的真实工程决定; 完整方法、基线、数据、模型、执行框架、预算、主要指标、重试、统计与判据来自 R3。这个分工同时支持工程迭代和稳定的受测对象。

实验仓库由用户手动准备。目录缺失时暂停并说明所需仓库; 进入已有仓库后先读它自己的`AGENTS.md`和工程规范, 沿用现有测试、日志与结果模式。

## 从证明方案编译运行计划

读取`实验证明.md`中的方法身份、共同协议、实验编号、比较逻辑和结果责任, 再按需要参考`叙事定稿.md`中的论文事实与设计关系。把每个实验编号映射到实际配置和共享运行, 避免同一结果重复执行或保存。

若缺少的是会改变研究结论的变量, 例如基线身份、数据版本、准确模型、预算、主要指标或排除规则, 返回 R3 与用户补齐。低风险工程选择由 R4 自主决定。

## 从可运行到正式证据

下面五个环节说明实验如何从调试进入可用证据。工程可以在现有目录和调度体系中实现这些责任。

### 冒烟测试

用最小数据和预算确认数据、模型、方法、评价、日志与汇总能够闭环。它只回答系统能否运行, 不承担论文结论。

### 小规模预跑

在受控规模下暴露实现错误、指标错误、不可行配置和预算估计偏差。预跑用于修正工程与协议; 除非 R3 事先说明可纳入, 调试期间接触和选择过的结果不进入正式统计。

### 协议冻结

正式运行前核对受测方法、实验范围、研究变量、失败处理和统计方式已经足以复现。若预跑暴露了需要改变论文含义的问题, 先更新 R3 并让用户确认。冻结为后续每个正式结果提供可识别的协议版本。

### 正式运行

运行冻结后的配置。保持实验含义的工程缺陷可以修复; 若修复影响已有输出, 将受影响运行标为协议无效并重跑。数据、预算、模型、主要指标或统计含义改变时结束当前批次, 新协议使用独立批次。

### 汇总验证

从原始结果和运行记录自动生成统计汇总, 抽查数字能否追溯到样本、种子与配置。论文图件使用验证后的结构化结果和确定性绘图脚本; 需要实际图件时调用`指导/能力/科研配图.md`。

## 让正式运行可追溯

实验仓库为正式运行保存机器可读的 JSON 或 YAML 记录。字段和文件布局服从项目, 但记录应足以回答四类问题:

- **运行了什么**: 运行与实验编号、方法配置、任务、数据划分、模型、执行框架、种子和预算
- **基于哪一版**: 代码提交、工作区状态, 以及迁移制品、编译上下文或项目记忆的必要哈希
- **发生了什么**: 开始与结束时间、调用、令牌、时间和成本中适用的部分, 以及成功、模型失败、环境失败或协议无效
- **证据在哪里**: 日志、配置、原始结果和汇总的真实路径

记录当前项目实际适用的字段, 一个批次清单可以包含多次运行。跨模型或跨执行框架实验同时记录语义源制品与目标侧编译产物, 让读者能够区分冻结迁移和重新优化。

## 失败与协议变化

接口瞬时故障、模型失败、格式失败、预算耗尽、方法失败和环境失败按 R3 预先约定处理。正式统计保留协议规定的完整运行集合, 环境错误按其真实身份记录。

单个负结果可能是方差, 也可能只削弱一个条件。只要协议仍有效、预算允许且剩余核心运行仍有辨识力, 就完成预定核心证据; 不因为结果不利而立即停止, 也不临时追加只对本文有利的配置。

以下变化需要暂停相关批次并与用户回到 R3: 方法或评价实现使当前结果失效, 协议本身存在缺陷, 预算或访问条件需要新决策, 或继续运行已经失去对原主张的辨识力。有效实验得到负结论时按真实观察和边界交付。

## 把运行还原成论文事实

每个已执行实验在`实验证明.md`的`实际交付`中写入:

- 直接观察到的数值、趋势、成本、机制现象或失败
- 这些观察对应的任务、方法、模型、框架、预算、重复与统计条件
- 当前对照允许的解释、仍未排除的替代解释和不能推出的结论
- 对相关分点主张的支持、削弱、推翻作用与成立边界
- 共享运行、清单、原始结果、日志、配置和汇总的真实路径
- R5 制作图表和结果段落所需的结构化材料

随后综合多个实验更新分点主张的当前判定。共享运行服务多处时只引用同一份真实产物。探索发现与预定核心证据分开解释。

若结果改变了任务边界、技术原因、核心认识或贡献性设计关系, 先完成有效证据交付, 再由用户决定返回 R2/R3 如何修改论文认识; R4 不在结果摘要中静默重写 Story。

当预定核心证据已经执行或被明确终止、指标可从原始结果复算、实际交付足以回答各实验问题, 并且主张判定能追溯到真实运行时, R4 才完成。
