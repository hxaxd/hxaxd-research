# R4 实验执行与结果交付

## 1. 做什么

R4 让 coding agent 在用户手动准备的`labs/<仓库目录>/`中, 按 R3 已确认的方法身份、实验组合和共同实验协议实现、运行实验, 保存真实产物, 并回填`实验证明.md`。

本阶段的边界是:

> 工程自由, 实验协议不可自由。

R4 可以自由决定代码结构、依赖库、脚本、目录、缓存、调度和并行方式; 不能临场改变会影响研究结论的完整方法、基线、数据划分、模型、执行框架、方法变体、预算、冻结策略、主要指标、重试规则、统计口径或判定标准。

## 2. 输入

- `研究想法.md`
- `叙事定稿.md`
- `实验证明.md`
- 用户手动准备的`labs/<仓库目录>/`

目标仓库必须已经存在。若不存在, 停止并请用户手动准备; Agent 不执行 clone、git init 或创建替代仓库。

若 R3 的完整方法、比较基线、核心实验或共同实验协议仍存在会改变研究结论的空缺, 不进入正式运行。低风险工程细节可以由 R4 自主决定。

## 3. 执行

### 1. 读取证明责任与实验协议

只读取本次执行必要的研究文档:

- 从`研究想法.md`理解原始 Idea 和核心主张
- 从`叙事定稿.md`理解中心叙事、论文事实、贡献性设计关系与分点核心主张
- 从`实验证明.md`读取方法与比较边界、共同实验协议、实验组合、分点判据和预定交付

先把每个实验编号及其共享运行映射到实际运行配置。若完整方法或基线身份不清, 或缺少模型准确标识、数据版本、预算、主要指标、重试或排除规则等研究变量, 回到 R3 与用户补齐, 不由 R4 猜测。

### 2. 进入独立实验仓库

确认`labs/<仓库目录>/`真实存在。进入后先读取该仓库自己的执行规范, 再检查代码结构、依赖、数据入口、测试、日志和结果格式。

实验仓库是独立 Git 工作区。R4 优先沿用其已有模式, 不把研究工作区的文档结构强加给代码仓库。

### 3. 按五个阶段推进运行

#### 冒烟测试

用最小数据和预算确认数据、模型、方法、评测、日志和结果汇总能够形成闭环。冒烟测试只证明系统能运行, 不进入正式结果。

#### 小规模预跑

用受控的小规模条件发现工程错误、指标错误、预算估计错误和明显不可行的配置。预跑可以帮助修复实现, 但默认不能进入正式主结果; 只有 R3 预先明确允许时才可纳入。

#### 协议冻结

正式运行前确认:

- 完整方法、方法变体和比较基线已经固定
- 核心、强化和探索实验的范围及预算已经确认
- R3 中所有研究变量已经确定
- 重试、超时、排除和失败分类已经固定
- 运行清单可以完整记录所需字段
- 指标能够从原始结果自动复算
- 预跑期间的调参不会与正式结果混合

满足这些条件后进入正式运行; 若需修改研究变量, 回到 R3 更新并由用户确认。

#### 正式运行

只运行协议冻结后的配置。正式运行期间可以修复不改变实验含义的工程缺陷; 修复若会影响已有输出, 受影响运行必须标为协议无效并重跑。任何会改变研究变量、数据、预算、重试或统计口径的修改都必须停止当前正式批次, 回到 R3 重新确认, 旧批次不得与新协议混合。

#### 汇总验证

从原始结果和机器可读运行清单自动生成指标与统计汇总。抽查汇总能否追溯到真实样本、种子和运行, 再把结论回填到`实验证明.md`。

若需要把正式结果生成论文图件, 调用`指导/能力/科研配图.md`。定量图必须从已经验证的原始结果或结构化汇总通过绘图脚本生成, 不能手工抄数或使用图像生成模型模拟。

### 4. 保存正式运行清单

每次正式运行在实验仓库中保存一份 JSON 或 YAML 机器可读清单。字段名和目录由实验仓库决定, 但至少能够记录:

- 运行编号以及服务的分点主张
- 对应的实验编号及证据等级
- 代码提交标识与工作区是否存在未提交修改
- 数据集、任务集合、划分及版本
- 模型的准确标识与角色
- 执行框架及版本
- 流程记忆源文件哈希
- 编译后上下文哈希
- 项目记忆快照哈希, 若该运行使用项目记忆
- 方法配置、随机种子、预算与超时
- 开始时间、结束时间、调用量、令牌量、运行时间和实际成本中适用的部分
- 成功、模型失败、环境失败或协议无效等运行状态
- 日志、原始结果和汇总结果的真实路径

不适用的字段可以省略或显式标为不适用, 不能伪造值。实验仓库可以在同一批次清单中记录多次运行, 无须机械地为每次调用创建文件。

对于跨模型和跨执行框架实验, 应用源文件哈希和必要的编译产物哈希证明迁移的究竟是哪一份流程记忆。若目标环境必须重新编译上下文, 同时记录相同的语义源文件与不同的编译产物, 不把重新优化后的版本称为直接迁移。

### 5. 遵守重试、排除与失败规则

正式运行只按 R3 预定规则重试和排除:

- 接口超时或环境瞬时故障是否重试、最多几次
- 模型执行失败、格式失败和预算耗尽如何计入
- 环境失败、模型失败、方法失败和协议无效如何区分
- 哪些结果可以排除, 哪些必须保留在统计中

不得只保留成功运行、最好种子或最好 checkpoint。遇到未预见但会改变统计含义的情况, 暂停相关批次并让用户决定是否修改协议。

### 6. 回填实验结果与主张判定

先按`实验证明.md`中每个已经执行的实验编号回填`实际交付`:

- 直接观察到的关键指标、趋势、成本、机制现象或失败类型
- 观察对应的任务、方法、模型、执行框架、预算、重复和统计条件
- 当前对照允许的解释、仍未排除的替代解释以及不能推出的结论
- 对相应分点主张的支持、削弱或推翻作用及成立边界
- 共享运行编号、机器可读清单、原始结果、日志、配置和汇总结果的真实路径
- 与`重点观察`和`预定交付`逐项对应的完成情况
- 可供 R5 生成主表、曲线、消融表、案例图或结果段落的结构化材料

随后按`分点主张判定`中的证据组合汇总多个实验, 回填每条主张的`当前判定`: 支持、削弱、推翻或尚不能判定, 并明确成立范围。探索实验产生的新现象与预定核心结论分开解释, 不能替代缺失的核心证据。

同一份共享运行服务多个实验或多条主张时, 引用同一真实产物即可。研究文档保存结果摘要和真实产物路径。

回填使用`叙事定稿.md`和 R3 已固定的术语。若正式结果表明任务边界、技术原因、核心认识或贡献性设计关系本身需要改变, 不在结果摘要中悄悄重述 Story; 完成有效证据交付后返回 R2/R3 由用户决定怎样修改。

回填必须保持`指导/参考/实验证明.md`的固定一级节。R4 可以在现有三级和四级标题内补充必要内容, 但不能把文件改写成实验日志或工程记录。

### 7. 处理负结果与停止条件

单个负结果可能来自方差, 也可能只削弱某个条件下的主张。只要预算允许、协议仍然有效且剩余核心运行仍能回答问题, 就完成 R3 预先确认的核心证据组合, 不因首个不利结果立即停止, 也不为了挽救主张临时追加只对本文有利的配置。强化实验按用户确认的预算完成; 探索实验不能挤占尚未完成的核心实验。

只有以下情况停止并与用户对齐:

- 方法或评测实现已经不可用, 当前结果不再有效
- 继续运行不再能够回答任何预定主张
- 预算、硬件、许可、访问控制或外部条件需要用户重新决策
- 实验协议被证明存在缺陷, 继续运行只会制造不可比较结果
- 需要改变 R3 的完整方法、Benchmark、Baseline、数据划分、模型、执行框架、预算、主要指标或判据

完成证据组合后, 即使结果推翻主张, 也如实回填并交由用户决定返回 R1、R2 或 R3。

## 4. 产出

正式 R4 产出是:

- `labs/<仓库目录>/`中的代码、配置、机器可读运行清单、日志、原始结果和结构化汇总
- 已回填实验实际交付和分点主张判定的`实验证明.md`

## 5. 确认点

- 目标仓库不存在时, 由用户手动准备
- R3 的方法身份、核心实验或共同协议缺少会影响结论的条件时, 由用户补充确认
- 需要改变 R3 的研究变量、重试规则、统计口径或判据时, 由用户确认
- 协议缺陷、预算阻断或完整证据组合完成后的重大负结果, 由用户决定下一步

## 6. 完成条件

- 已完成或明确终止 R3 预定的核心实验和对应共享运行
- 强化实验已按用户确认的范围执行, 探索实验与核心结论分开
- 每次正式运行都有可追溯的机器可读清单和真实产物
- 正式结果与冒烟测试、预跑结果没有混合
- 指标和汇总可从原始结果自动复算
- 跨模型或跨执行框架迁移所用流程记忆可由哈希核对
- 每个已执行实验都有实际交付
- 实际交付包含可直接用于论文写作的观察、条件、不确定性、解释边界和结构化图表材料
- 每条需要经验支持的分点主张都有基于完整证据组合的当前判定
- 回填后的`实验证明.md`仍遵守固定一级节
- 未改写 R3 的方法身份、研究目标、实验组合、共同协议和判据

## 7. 不应做

- 不替用户创建、初始化或拉取实验仓库
- 不把独立实验仓库强制改造成研究项目目录结构
- 不把 dry run、冒烟测试或未获允许的预跑当作正式结果
- 不在正式运行后无记录地调参并混入同一主表
- 不只保存最好种子、最好 checkpoint 或成功运行
- 不把环境错误直接解释为研究主张失败
- 不把研究失败包装成环境问题或积极措辞
- 不因单个负结果立即放弃其余仍有判别力的预定运行

## 8. 常见失败

- **协议漂移**: 为了让结果更好, 在正式运行中改变预算、模型或重试规则
- **预跑污染**: 调参用过的结果被混入正式统计
- **迁移失真**: 跨模型时重新优化了流程记忆, 却声称直接迁移
- **成本失配**: 只比较轮数, 没有比较实际调用、令牌或时间
- **路径不可追溯**: 汇总数字无法定位到运行清单和原始结果
- **过早停止**: 首个负结果出现后没有完成原定消融或重复运行
- **工程洁癖**: 为重构仓库拖慢研究验证
