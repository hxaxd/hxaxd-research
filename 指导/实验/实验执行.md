# R4 实验执行与结果交付

## 1. 做什么

R4 让 coding agent 在用户手动准备的`labs/<仓库目录>/`中, 以`实验证明.md`的证明目标为最终目标, 自由实现、运行、保存结果并回填交付状态。

R4 不重新设计研究目标, 不创建新的实验计划文档, 不把工程挫折反向改写成新的证明判据。R3 已经声明证明方案和交付要求, R4 负责把它落实为真实代码、日志、结果和路径。

## 2. 输入

- `研究想法.md`
- `叙事定稿.md`
- `实验证明.md`
- 用户手动准备的`labs/<仓库目录>/`

目标仓库必须已经存在。若不存在, 停止并请用户手动准备; Agent 不执行 clone、git init 或创建替代仓库。

## 3. 执行

### 1. 读取研究目标

只读取本次实验必要的研究文档:

- 从`研究想法.md`理解原始 Idea 和核心主张
- 从`叙事定稿.md`理解叙事与分点核心主张
- 从`实验证明.md`读取每条分点主张的指标、判据和结果交付要求

不要为了执行实验而重写 R1 到 R3 的研究判断。若发现 R3 缺少关键交付要求, 先停下让用户回到 R3 补齐。

### 2. 进入独立实验仓库

确认`labs/<仓库目录>/`真实存在。进入后先读取该仓库自己的执行规范, 再检查代码结构、依赖、数据入口、测试、日志和结果格式。

实验仓库是独立 Git 工作区。R4 可以在其中编辑代码、运行实验和保存产物, 但不规定其内部布局; 优先沿用该仓库已有模式。

### 3. 以研究为目的实现

技术取舍以研究验证为目的, 不是以生产系统为目的。实现应优先满足:

- 能快速形成可运行闭环
- 能准确计算 R3 指标
- 能保存 R3 要求的原始结果、日志、配置和结构化汇总
- 能定位代码版本、实际命令和产物路径
- 能在失败时保留足够信息判断是工程问题、资源问题还是研究主张受损

R4 可以自由选择具体框架、脚本、目录和运行方式, 只要不改变 R3 已确认的证明目标和判据。

### 4. 运行与保存结果

每次正式运行按 R3 的交付目标保留足够材料。可以包括实际执行入口、代码状态、配置、随机种子、数据版本、模型版本、运行日志、原始预测、指标文件、中间产物、结构化汇总、运行状态、资源消耗、运行时间或成本等, 但具体保存什么由实验本身和 R3 交付要求决定。

具体保存位置由实验仓库决定, 但必须能从`实验证明.md`回填路径追溯到真实文件。

### 5. 回填交付状态

按`实验证明.md`的“结果交付要求”回填。回填内容由实验本身决定, 可以包括实际状态、关键指标或摘要、原始结果路径、日志和配置路径、结构化汇总路径、与支持/削弱/推翻判据的关系、当前不能得出的结论等。

回填只记录真实产物和直接判断, 不另建独立结果判断文档。

回填 R3 时必须严格参照`指导/参考/实验证明.md`的固定节格式, 不得自行增减一级节, 不得把`实验证明.md`改写成结果报告、失败记录或实验日志。

### 6. 处理挫折

遇到工程挫折时, 先尝试在不改变 R3 目标的前提下解决。以下情况必须停止并与用户对齐:

- 需要更换 Benchmark、Baseline、指标或主要判据
- R3 要求的结果无法通过当前数据或接口获得
- 预算、硬件、许可或访问控制阻断实验
- 实现发现方法逻辑不可行
- 结果直接触发 R3 的推翻判据, 或需要用户决定是否继续投入

不要因为实现困难就悄悄降低主张、替换指标或删除对照。

## 4. 产出

正式 R4 产出是:

- `labs/<仓库目录>/`中的代码、配置、日志、原始结果和结构化汇总
- 回填后的`实验证明.md`

不创建独立实施计划、独立失败文档或独立结果判断文档。

## 5. 确认点

- 目标仓库不存在时, 由用户手动准备
- 需要改变 R3 的 Benchmark、Baseline、指标、判据或结果交付要求时, 由用户确认
- 结果触发推翻判据、转向或重大范围变化时, 由用户决定下一步

## 6. 完成条件

- R3 列出的每条分点主张都有实际状态
- 回填后的`实验证明.md`仍严格遵守`指导/参考/实验证明.md`的固定节格式
- 已完成项可通过回填路径追溯到实验仓库中的真实产物
- 失败、中断或部分完成项有可定位日志或错误摘要
- 关键指标与 R3 支持、削弱、推翻判据对应
- 未改写 R3 的研究目标和判据
- 没有创建被新版流程禁止的独立实验阶段文档

## 7. 不应做

- 不替用户创建、初始化或拉取实验仓库
- 不把独立实验仓库的内部结构强制改造成研究项目目录结构
- 不补造未运行的结果, 不把 dry run 当作正式结果
- 不只保存最好种子或最好 checkpoint
- 不在看到结果后无记录地更改指标、数据拆分或对照
- 不把环境错误直接解释为研究主张失败, 也不把研究失败掩盖成环境问题

## 8. 常见失败

- **目标漂移**: 为了让代码跑通, 偷换 R3 的证明目标
- **路径不可追溯**: 聊天中说有结果, 文档里没有真实产物路径
- **只保留汇总**: 保存了平均分, 丢失原始预测、配置或日志
- **工程洁癖**: 为了重构仓库拖慢研究验证
- **判断外包给语言**: 结果没有达到判据, 却用积极表述包装成支持
