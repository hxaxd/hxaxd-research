# R3 实验证明

## 1. 做什么

R3 根据`叙事定稿.md`中的分点核心主张, 声明需要经验支持的主张要如何被实验支持或推翻, 并作为 R4 回填真实结果和状态的同一份证明文件。

本阶段只写证明责任和交付要求, 不写技术栈、框架、代码目录、脚本入口或实现流程。实现自由交给 R4 的 coding agent。

## 2. 输入

- 用户已经确认的`叙事定稿.md`
- `叙事定稿.md`中的分点核心主张
- 已知的最相近工作、可用 Benchmark、Baseline 候选、指标和大致资源边界

若分点主张仍不清楚, 先回到 R2 修正, 不用实验证明替代主张澄清。

R3 只处理需要经验支持的分点主张。已经能由文献引用和论述完成的共识类主张, 不必进入`实验证明.md`; 问题类主张只有在需要实验或现象观察时才进入 R3; 方法类主张通常需要实验; 原因类主张通常对应可解释性分析、机制观察、消融、案例或中间过程证据。

若 R3 过程中发现必须回写 R2, 必须严格参照`指导/参考/叙事定稿.md`的固定节格式更新`叙事定稿.md`, 不得把返工说明、实验判断或临时记录写进 R2 产物。

## 3. 执行

### 1. 按分点主张写证明方案

对每个需要经验支持的分点主张, 沿用`叙事定稿.md`中的主张编号和标题。每条主张本身就是一个实验项, 不再另设实验编号。

每条分点证明方案只保留两个固定四级标题, 标题下的内容自由组织:

- `#### 实验设计`: 可以写 Benchmark 或数据、指标或中间现象、Baseline 或对照等。
- `#### 预期证据`: 可以写什么结果会支持、削弱或推翻该主张。

如果多条主张使用同一次运行或同一组结果, 不新增独立实验项; 在各自主张条目中写清相同的 Benchmark、数据、指标或对照即可。

### 2. Benchmark 与指标

Benchmark 选择要服务主张, 不是追求数量。描述时可以说明:

- 测量的能力或现象
- 与分点主张的关系
- 已知限制或偏差
- 指标方向、单位和聚合方式
- 支持/削弱/推翻判据

若使用自建数据, 只说明数据应满足的声明式条件, 例如覆盖哪些场景、需要哪些标签、如何避免泄漏。不要在 R3 写具体构建脚本或技术实现。

### 3. Baseline 与对照

Baseline 和对照要能服务对应主张。描述时可以考虑:

- 最相近工作或当前强方法
- 简单但有竞争力的替代方案
- 去除关键机制的本文变体
- 控制计算量、数据量、模型能力或调用预算的对照

若某个 Baseline 暂时不可获得, 标为待确认并说明它会影响哪条主张的可信度。

### 4. 结果交付要求

R3 需要提前让 R4 知道大概要交付什么, 使 coding agent 能按目标填充结果。结果交付要求沿用分点证明方案中的主张编号和标题, 不另起实验编号。内容可以包括主要指标、原始结果、日志和配置、种子级/样本级/子集级结果、资源消耗、运行时间或成本等, 但只在相关时写, 不预设固定条目。

结果交付要求只定义“可能需要拿到什么证据”, 不规定固定字段、固定顺序、固定小标题, 也不规定 R4 必须采用什么框架、目录或命令实现。

## 4. 产出

生成或更新`实验证明.md`。结构必须严格参照`指导/参考/实验证明.md`的固定节格式, 只包含:

- 分点证明方案
- 结果交付要求

不得使用表格, 不得新增其他一级节。需要用户决定的风险或返工事项在对话中说明, 不作为`实验证明.md`的固定结构。

## 5. 确认点

- 用户确认 Benchmark、Baseline、指标、实验范围和资源边界
- 用户确认每个主张的支持、削弱和推翻判据
- 用户确认结果交付要求足以交给 R4 执行

## 6. 完成条件

- 已生成`实验证明.md`
- `实验证明.md`严格遵守`指导/参考/实验证明.md`的固定节格式
- 每个需要经验支持的分点主张都有对应证明方案
- 每条进入 R3 的分点主张都有支持、削弱和推翻判据
- 已用自由结构说明 R4 可能需要交付的指标、原始结果、日志、配置、结构化汇总或其他必要证据
- 文档不包含具体技术栈、代码路径、脚本命令或工程实现方案
- 用户确认可以进入 R4

## 7. 不应做

- 不把一张主表当作所有主张的充分证据
- 不选择只容易显示优势、但不测量主张的 Benchmark
- 不用弱实现制造优势
- 不在看到结果后更换主要指标或重新定义成功
- 不在 R3 规定实验仓库内部布局、框架或命令
- 不把“多做一些实验”写成没有目标的扩张

## 8. 常见失败

- **Benchmark 错位**: Benchmark 测量总体能力, 却被用来证明特定机制
- **判据缺失**: 只写预期提升, 没有削弱或推翻条件
- **证明混乱**: 一个实验结果被拿来支持它并不能支持的主张
- **交付不清**: R4 不知道最后应该保存哪些结果和日志
- **实现越界**: 在 R3 里提前设计代码架构, 限制后续 coding agent 的自由
