# R8 投稿准备

## 1. 做什么

把稳定稿件和全部提交材料收束为可提交版本, 编排终稿能力, 清除硬性阻断项, 并记录用户对剩余研究风险和最终投稿版本的决定

## 2. 输入

- 稳定稿件和论文项目路径
- 目标 venue, 年份, track, 投稿类型和预计提交时间
- 当前附录, 补充材料, 代码与数据链接
- R7 前期的 `投稿规范检查.md` (若已执行)
- 可用的审稿意见, 引用核验, 编译和匿名检查记录

直接进入 R8 时, 不要求补建完整阶段系列, 但目标稿件, 提交规则和实际材料必须足以完成终稿检查

## 3. 执行

R8 不是再次线性重做整篇论文, 也不把每项建议都升级为投稿门禁, 它负责区分官方与工程硬要求, 可修复内容问题和可由用户明确接受的非阻断风险

### 1. 锁定候选提交版本

记录主 Tex 入口, 当前 commit, 候选 PDF, 补充材料和提交文件路径, 候选版本变化后,
受影响的工程和合规检查必须重跑

### 2. 执行终稿能力

按实际依赖调用:

1. 投稿规则核验的 R8 终稿模式
2. 参考文献与引用核验
3. Tex 成稿工程验收
4. 稿件匿名化检查
5. 综合论文审稿

顺序可按项目情况调整, 也可在修复后重复调用, 能力结果不自行改变阶段

### 3. 分类问题

- **硬性阻断**: 官方强制要求不符合, 编译错误, 缺失或未定义引用, 必要匿名暴露,
  最终 PDF 无法重建
- **内容返工**: Claim, Story, 实验或引用证据存在必须修正的问题, 需先解决 R7 或更早阶段的问题
  受影响阶段
- **非阻断风险**: 不会直接违反官方要求或破坏工程完整性, 但可能影响论文质量,
  说服力或录用概率

硬性阻断必须清零, 非阻断风险可以修复, 也可以由用户在理解影响后明确接受（确认点）

### 4. 形成投稿准备记录

生成 `投稿准备记录.md`, 至少包含:

| 项目 | 内容 |
|---|---|
| 最终稿 | Tex 项目, commit, PDF 和提交材料路径 |
| 规则核验 | 终稿模式结论和未核验项 |
| 引用核验 | BibTeX, 缺失引用和论断证据状态 |
| Tex 验收 | 构建命令, 最终 PDF 和工程问题 |
| 匿名检查 | 暴露, 修复和剩余风险 |
| 综合审稿 | 主要问题, 修复状态和返回阶段 |
| 未解决风险 | 严重性, 影响, 是否阻断 |
| 用户决定 | 接受项, 最终版本和日期 |

### 5. 用户最终确认

向用户展示硬性项目清零证据, 仍存在的非阻断风险和候选提交文件, 用户确认最终版本,
接受项和投稿决定后（确认点）, R8 才能完成

## 4. 产出

- `投稿准备记录.md`: 包含最终稿信息、规则核验结论、引用核验状态、Tex 验收结果、匿名检查结果、综合审稿结论、未解决风险和用户最终决定
- 最终候选 PDF 和提交材料（路径记录在投稿准备记录中）

## 5. 确认点

- 用户接受非阻断风险（§3 分类问题步骤）
- 用户确认最终版本、接受项和投稿决定（§3 用户最终确认步骤）

## 6. 完成条件

官方硬性要求, 编译错误, 缺失引用和必要匿名暴露已经清零, 最终稿和提交材料可定位,
可重建, 非阻断风险已修复或由用户明确接受, `投稿准备记录.md` 包含最终决定, 发现新的硬性问题时需先解决 R7 的问题, 不以”接近截止时间”为由跳过核验

## 7. 不应做

- 不把”编译成功”当作内容和引用已经正确
- 不把审稿建议全部视为硬性门禁
- 不在未重跑检查时沿用旧版本的通过结论
- 不替用户操作外部投稿系统或作出重大投稿决定, 除非用户明确要求且允许
- 不把未核验规则, 匿名暴露或缺失引用标为已解决

## 8. 常见失败

无
