# 相近工作与撞车分析

## 1. 做什么

寻找真正共享问题假设或机制作用的工作, 而不是只搜索表面关键词

撞车分析的目标不是证明“没有人做过”, 而是界定已有工作已经覆盖什么, 当前 Idea
还剩下什么, 以及剩余空间是否足以形成可验证贡献

## 检索策略

1. 为每项本质假设建立问题词, 同义表达, 历史术语, 机制作用, 任务, 数据集和实验
 设置查询
2. 分别检索“同问题不同方法”“不同问题同机制”“同场景同评价”和“直接反例”
3. 从最相近工作向前追溯其引用, 向后检查引用它的改进, 复现和批评工作
4. 对声称“首次”或“不同于所有已有方法”的位置进行反向核验
5. 记录检索时间, 数据库, 查询式和停止边界, 未发现不等于不存在

## 比较框架

对每篇高相关工作至少比较:

- 它认为问题是什么
- 它依赖哪些本质假设
- 方法实际改变了系统中的什么
- 训练, 推理或交互时的作用位置
- 使用什么实验来证明
- 作者明确声明的贡献与局限
- 当前 Idea 与它在问题, 机制, 场景和证据上的差异

按以下五类情形分类:

- 表面相似但本质假设不同
- 方法形式不同但本质假设相同
- 本质假设与方法都接近
- 本质假设不同但实验场景高度重叠
- 同一问题但未覆盖当前关键机制

## 撞车风险判定

- **完全撞车**: 核心问题, 机制作用和主要证明方式均已覆盖, 仅实现细节不同
- **高度重合**: 本质假设相同, 当前 Idea 只在组件, 规模或应用上变化
- **部分覆盖**: 关键链条已有工作覆盖, 但仍存在可明确验证的未覆盖环节
- **场景重叠**: 任务或 Benchmark 相同, 但问题和机制不同
- **相邻参考**: 可提供术语, 组件或实验借鉴, 不构成核心覆盖
- **未发现直接覆盖**: 仅表示在已记录边界内未发现, 不能写成“首次”

## 反向攻击

在得出“未撞车”前, 主动回答:

1. 如果审稿人只允许引用一篇最接近工作, 会选择哪篇
2. 删除方法名称和架构图后, 两者的作用是否仍有本质区别
3. 当前差异是否只是更多数据, 更大模型, 不同提示词或不同模块边界
4. 能否设计一个实验专门区分两种机制, 若不能, 差异可能不可证

## 证据回写

更新`相关工作矩阵.md`, 把每项关键判断的原文位置写入`文献证据库.md`
作者没有明确说出的局限只能标为 Agent 推断

## 8. 常见失败

- 只搜索 Idea 使用的最新术语, 漏掉早期同机制工作
- 把不同模型架构误判为不同研究机制
- 为了保留 Idea, 把最相近工作降格为“相关但不同”却不给可证区别
- 只看摘要和贡献列表, 不读方法与实验设置
- 把“没有找到”写成“从未有人研究”

## 4. 产出

`相近工作与撞车分析.md` 必须写明检索边界, 最相近工作, 本质比较, 未覆盖空间,
不确定点和最终撞车结论, 结论应说明继续, 修改或停止 Idea 的具体原因, 而不是只给风险标签
