# R1 Idea 预审与立项

## 1. 做什么

R1 负责把用户随口描述的研究 Idea 对齐、正式化并落盘为`研究想法.md`。落盘后, 推荐用户继续做两类对话式审查: 撞车分析和硬度审查。两类审查统一遵守`指导/能力/学术审阅与反向论证.md`, 尤其是先固定审阅对象和判据、完成足够的反向调查, 以及只根据新证据更新原判断。

R1 的正式产物是`研究想法.md`。撞车分析和硬度审查在对话中给出判断和风险; 只有撞车非常接近、主张被击穿或用户明确要求时, 才给修改建议。

## 2. 输入

- 用户原始 Idea, 包括想解决什么、打算怎么做、为什么可能有效、适用场景和不确定点

## 3. 执行

### 1. 对齐用户 Idea

先通过对话确认自己没有误解用户。用户原话通常粗糙、跳跃或混合了方法、直觉、场景和预期效果, Agent 要做的是整理, 不是替用户发明一个更容易成立的新题目。

对齐时至少说清:

- 这个 Idea 想解决的场景和对象
- 用户真正关心的问题
- 用户提出的方法或机制
- 用户相信它会有效的原因
- 当前明确不处理的范围

如果用户指出理解偏了, 立即按用户修正重写, 不在旧理解上继续审查。

### 2. 写入研究想法

创建或更新`研究想法.md`, 参考`指导/参考/研究想法.md`。文档只包含两部分:

1. **研究想法**: 用正式、准确、自然的中文描述用户的 Idea, 保留用户意图, 去掉随口表达中的歧义和包装。
2. **核心主张**: 把 Idea 拆成四块:
   - 共识: 当前研究或工程中已经成立、可作为出发点的背景判断
   - 问题: 在什么场景或条件下出现什么问题, 导致什么效果不好
   - 方法: 用户主张做什么改进或采用什么机制
   - 原因: 为什么该方法应当改善问题, 预期改变什么行为或结果

核心主张不是贡献文案, 而是后续检索、审查、叙事和实验都要围绕攻击或支持的研究判断。

### 3. 推荐撞车分析

`研究想法.md`落盘后, 立即建议用户进行撞车分析。撞车分析使用核心主张里的词和等价表达反复检索最相近论文, 重点找最可能否定新颖性的工作。

撞车分析默认直接进行网络搜索, 面向外部公开论文和最新相关工作。不要先研究本地已有论文、领域目录或论文列表; 本地材料只能作为用户额外指定的线索, 不能替代网络检索。

检索范围、充分性和用户反驳的处理遵守`指导/能力/学术审阅与反向论证.md`。不得看到少量表面相近论文便提前否定, 也不得在用户表示不同意后没有新证据便反转结论。

撞车分析必须先抽主张, 再做对比。对每篇候选论文, 先用自己的话识别它的核心主张: 它认为存在什么问题、提出什么方法、为什么有效、证明了什么。然后逐项对比当前 Idea 的核心主张: 共识、问题、方法、原因分别被覆盖到什么程度。没有完成“对方核心主张 → 当前核心主张 → 覆盖关系 → 撞车判断”这条链路之前, 不得判断“完全一致”“非常接近”或“不算接近”。

对话结论默认只展开最关键的一至两篇论文:

- 对方工作的核心主张是什么
- 它覆盖了当前核心主张的哪一部分
- 二者是完全一致、非常接近, 还是不算接近
- 若完全一致, 明确建议停止
- 若非常接近, 明确建议修改 Idea
- 若不算接近, 说明当前未发现直接覆盖, 但不能写成“首次”

不要用大量论文名制造压迫感。背景论文可以作为线索, 但判断必须落在核心主张比较上。最终对话只展开真正参与撞车判断的关键论文; 其他检索线索不列清单, 不用“还看到了 A/B/C”制造虚假的全面感。不要在结论里堆一串“相关但没有直接吃掉”的工作, 也不要用“没有被单篇工作直接覆盖”来包装新颖性; 如果某篇论文不能决定撞车判断, 通常不需要展开。

### 4. 推荐硬度审查

撞车分析后, 推荐用户进行硬度审查。硬度审查围绕核心主张四块逐项攻击:

- 共识是否真实存在, 是否被证据支持
- 问题是否真实存在, 是否重要, 是否不是伪问题
- 方法是否逻辑可行, 数学、代码、数据或系统上是否能实现
- 原因是否成立, 即方法即使能做出来, 是否真的会让目标问题变好

硬度审查先形成当前 Idea 的最强成立版本和最强反向论证, 再报告能够改变研究去留的一至三个决定性问题。用户辩护时逐条检查它是否真正解决原问题; 用户的信心和继续意愿都不是审查通过的证据。

硬度审查前必须重新抓住用户的方法本体, 区分方法本体、载体示例和实现细节。不要把用户随口举的格式、文件类型、模块名或工程实现当成方法本身来攻击; 如果方法本体不清楚, 先问用户或回到`研究想法.md`修正, 不要审查自己猜出来的方法。

硬度审查不是要求研究已经被证明。原因部分只检查“方法为什么可能有效”的机制链条是否合理、是否能被后续实验观察; “尚未被证明”是研究的正常状态, 不能单独作为风险或否定理由。不要把“合理但未证明”写成负面审查结论; 可写“机制链条目前合理, 后续需要实验检验”。若原因已经被充分证明, 反而说明研究空间可能变小。

硬度审查不是替用户和一个稻草人争论。不要主动反驳用户没有写下或没有明确主张的绝对化说法, 例如“不能说所有现有工作都错”“不能泛泛否定别人工作”。只有当`研究想法.md`或用户原话真的包含这种过度表述时, 才指出并要求收窄。正常情况下, 审查应直接检查当前核心主张是否真实、重要、可行、合理, 不要用常识性提醒填充篇幅。

硬度审查也不是主动替用户改题。结论只在对话中给出: 通过、需要修改、暂缓或不建议继续。只有出现明显撞车、共识不存在、问题不存在、方法不可行或原因链条断裂等严重情况时, 才主动建议修改; 否则只说明风险和不确定性, 不主动推荐新题目、新标题或新表述重心。若用户要求修改建议, 再给可选方向。

### 5. 立项与目录

对于尚未落地的新 Idea, 在对齐研究想法时创建`workspace/<中文研究项目名>/研究想法.md`。后续产物在相应阶段开始时创建。

同名或近似项目已存在时, 先提示用户选择继续旧项目还是使用有意义的限定词命名新项目, 不自动追加不透明编号。

## 4. 产出

正式 R1 产出只有:

- `研究想法.md`

若对话审查中发现后续会使用的文献, 需要写入`研究想法.md`时, 按通用协议在使用位置直接引用; 暂无网页来源时写明论文标题并标记`链接待补`。

## 5. 确认点

- 用户指出 Idea 理解错误时, 先修正`研究想法.md`, 再继续审查
- 准备给出完全撞车、非常接近或不建议继续结论前, 必须确认该结论基于已对齐的核心主张
- 准备批评方法不可行前, 必须确认自己批评的是用户的方法本体, 不是载体示例或实现细节
- 需要改变核心研究方向、目标场景或主要方法时, 必须由用户确认
- 同名或近似研究项目存在时, 由用户决定目录名和是否复用旧项目

## 6. 完成条件

- `研究想法.md`只包含研究想法和核心主张两部分
- 核心主张已明确拆成共识、问题、方法、原因
- 用户已确认当前版本没有误解原意
- 已向用户推荐撞车分析和硬度审查, 并在对话中完成用户选择的审查
- 项目目录包含当前阶段所需的真实产物

## 7. 不应做

- 不把用户随口说法改成另一个更容易成立的问题
- 不在 R1 写论文叙事、贡献点或实验证明
- 不把用户希望继续当作审查通过的证据
- 不在检索前声称首次、全新或未被研究
- 不把本地已有论文库当作撞车分析的主要检索范围
- 不用论文数量替代核心主张比较
- 不在没有抽取对方核心主张时做撞车结论
- 不跳过“对方核心主张 → 当前核心主张 → 覆盖关系 → 撞车判断”的对比链路
- 不用“没有被单篇工作直接覆盖”这种说法替代严肃的新颖性判断
- 不把“原因尚未证明”当作硬度审查的负面结论
- 不把载体示例或实现细节误当成用户的方法本体
- 不主动反驳用户没有主张的绝对化说法
- 不在没有严重错误时主动替用户推荐改题、改名或调整重心

## 8. 常见失败

- **悄悄换题**: 为了让 Idea 更强, 把用户原本想研究的问题换掉
- **先审后写**: 还没把核心主张写清楚, 就急着判断撞车或硬度
- **只比外观**: 只比较方法名、模块名或任务名, 不比较核心主张
- **论文堆砌**: 明明只需要一两篇关键工作, 却列出一串论文来制造“没有被单篇吃掉”的假安全感
- **方法误读**: 把用户举例用的 YAML、schema、文件名或实现方式当作方法本体来批评
- **原因苛责**: 把“原因还没有被证明”当成缺陷, 忘了 R1 只需要检查原因是否合理且可验证
- **过度建议**: 用户只需要审查, Agent 却主动包装新题目、新重心和修改路线
- **硬度审查过软**: 只给优化建议, 不指出哪一块主张可能不成立
