# 论文架构图设计

## 任务目标

根据摘要和方法设计专业、清晰的论文架构图方案或生成提示词。

架构图不是方法章节的装饰。它应让读者在阅读正文前理解输入、核心机制、信息流、
输出和创新位置，并与方法章节使用完全一致的术语与方向。

## 输入

- `叙事定稿.md` 中的核心问题与机制。
- `核心主张.md` 中该图服务的 Claim。
- 方法概述、模块输入输出、训练/推理差异。
- 论文版面、单栏/双栏宽度和目标图号。
- 用户希望生成的是结构方案、提示词、可编辑矢量草案还是最终图。

输入不足时先列缺口，不凭摘要猜出方法中不存在的模块。

## 设计流程

### 1. 确定图的单一叙事

先完成一句话：

> 读者看完这张图，应理解 ______，并能区分本文与 ______ 的关键差异。

一张图不应同时承担领域背景、完整训练流程、全部实验结果和所有实现细节。若目标过多，
拆成总览图与局部机制图。

### 2. 提炼必要对象

列出：

- 输入、输出和外部条件；
- 核心状态或表示；
- 必要模块及其职责；
- 数据流、控制流、反馈或时间顺序；
- 训练专用与推理专用步骤；
- 本文创新和已有组件。

删除仅为实现存在但不影响理解的模块。不要把所有框画成同等权重。

### 3. 选择图式

- 线性流水线：适合单向处理。
- 分阶段面板：适合训练/推理或多个时间阶段。
- 中心机制加周边接口：适合一个核心模块与多类输入输出。
- 双栏对照：适合 Baseline 与本文机制差异。
- 循环/反馈图：只在真实存在迭代或闭环时使用。
- 层级系统图：适合控制、执行和存储有明确层次的系统。

图式必须服从方法，不为视觉效果制造不存在的回路或并行。

### 4. 规划视觉层级

- 创新模块使用稳定强调色，已有组件使用中性色。
- 同类对象形状一致，不用颜色同时表达多种含义。
- 箭头方向统一，反馈边与主流程可用不同线型。
- 面板顺序与正文叙述顺序一致。
- 图中文字使用简短英文标签；缩写应在图注或正文定义。
- 复杂公式放正文，图中只保留必要符号或操作。

默认采用白底、扁平矢量、干净线条和色盲友好配色。避免照片感、卡通感、廉价 3D
阴影、装饰图标堆积和过小文字。

### 5. 生成图示规格

先输出结构规格，再生成提示词。结构规格至少包含：

| 面板/区域 | 对象 | 输入 | 操作 | 输出 | 与 Claim 的关系 | 视觉强调 |
|---|---|---|---|---|---|---|

提示词应明确画布比例、面板、模块、连接、标签、颜色、风格和禁止项。生成式图像工具只适合概念草案或视觉探索；默认任务只交付高质量提示词，不主动调用生图工具。只有用户明确要求生成图像时才调用相应工具。最终科学架构图优先使用可编辑矢量工具重绘并人工校对文字。

## 提示词构成

高质量提示词应按以下顺序描述：

1. 图的用途和版式，例如双栏论文中的横向方法总览。
2. 面板数量、排列和每个面板的职责。
3. 模块、输入输出和箭头关系。
4. 哪个模块是创新点，如何强调。
5. 字体、配色、线条、背景和留白。
6. 明确禁止照片、3D、装饰噪声、伪文字和错误箭头。

不要只写“生成一个专业的论文架构图”，这无法约束科学内容。

## 常见陷阱

- 图好看，但读者不知道输入从哪里来、输出到哪里去。
- 图中流程与方法正文顺序或训练代码不一致。
- 所有模块都高亮，无法识别创新。
- 使用大量图标代替准确技术对象。
- 用生成模型输出的伪英文标签直接进入论文。
- Baseline 与本文对照时故意把 Baseline 画得简单或混乱。
- 把实验结果、推测性效果或未实现模块画成方法事实。

## 简短示例

若核心机制是“在任务切换时隔离上下文”，图的主线可以是：

> 完整轨迹输入 → 任务边界检测 → 按子任务划分的上下文存储 → 当前子任务检索
> → Agent 决策。

创新模块只强调“边界检测 + 隔离存储 + 受控检索”。模型调用、日志和通用解析器可以弱化或省略，除非它们承担 Claim。

## 输出

1. 图的叙事目标和 Claim。
2. 必要对象、模块与连接关系表。
3. 推荐图式、面板布局、视觉层级和配色。
4. 可交给图像生成工具的中英文草案提示词，或可编辑绘图实现计划。
5. 图注要表达的关键信息。
6. 需要人工核对的标签、箭头、训练/推理边界和方法事实。

## 完成检查

- 不看正文能否说出输入、机制和输出？
- 图中每条箭头是否有真实语义？
- 创新点是否清楚但未被夸大？
- 图与方法正文、Claim 和实现是否一致？
- 缩小到论文实际宽度后文字是否仍可读？
- 生成式草案中的所有文字和连接是否已经人工重绘或核验？
