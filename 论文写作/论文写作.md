## 流程

- 想一个 idea -> 画出技术流程图
- 写引言 + 方法 + 实验方案
- 做实验
- 写实验 + 相关工作 + 结论 -> 画各种图表
- 写摘要 -> 写标题

## 关键点

### 表达原则

- 一段文字讲且只讲一个信息
    - 一段开头的第一句直接说明这段在讨论什么
    - 每个后续句子都应该解释, 证明, 细化或限制首句
- 同一个概念在全文使用同一个名称和符号
    - 一句话出现的新名词不能依赖作者脑中的隐藏背景
- 不依赖堆积衔接词制造表面连贯, 先保证句子之间真的存在逻辑关系
- 句间关系通常包括:
    - 原因和结果
    - 问题和解决方案
    - 一般情况和具体情况
    - 观点和证据
    - 对比和让步
    - 结论和限制条件

### 检查表达

- 假装自己是第一次看到论文的读者
- 检查读者是否能理解词汇, 术语和每一步推理
- 对已经写好的内容反向列提纲
- 写出总主张, 每段首句和每段证据
- 检查每段首句是否服务于总主张
- 检查每条证据是否服务于对应段落
- 只有关系明确时才添加表示因果, 对比, 举例或总结的衔接词

### 自我评审

1. 贡献是否充分
    - 论文是否只针对一个早已被充分认识的失效情形, 没有发现新的关键技术原因或问题认知
    - 提出的技术已经被充分探索
    - 该技术带来的性能提升是可以预见或已经广为人知的
2. 写作是否清楚
    - 缺少技术细节, 导致方法不可复现
    - 某个方法模块缺少动机
3. 实验效果是否足够好
    - 只比已有方法好一点
    - 虽然比已有方法效果好, 但效果仍然不够好
4. 实验测试是否充分
    - 缺少消融实验
    - 缺少重要基线
    - 缺少重要评价指标
    - 数据太简单, 无法证明方法是否真正有效
5. 方法设计是否合理
    - 实验设置不实际
    - 方法存在技术缺陷, 看起来不合理
    - 方法不鲁棒, 需要针对每个场景调整超参数
    - 新方法设计在带来收益的同时引入了更强的局限, 导致总体收益为负

## 结构

### 标题

- 标题可能吸引不同领域的审稿人
- 先写下任务, 技术, 问题和核心认知等关键词
- 标题和方法短语需要有具体含义
    - 使用的技术
    - 论文任务
    - 解决的问题

### 摘要

- 我们解决的技术问题是什么
- 为什么已有方法没有成熟有效的解法
- 技术贡献是什么
- 方法本质上为什么能够工作
- 技术优势和新认知是什么
- 哪些实验结果可以支持上述主张
- 摘要中的每项主张必须准确, 并有实验支持

```tex
\section{Abstract}
% 任务
% 已有方法的技术问题和技术原因
% 解决问题的核心认知或技术贡献
% 技术优势
% 有证据支持的实验结论
```

摘要可以采用三种组织方式:

- 技术问题 -> 技术贡献
- 技术问题 -> 核心认知 -> 技术贡献
- 多项技术贡献 -> 分别说明每项贡献的技术优势

详见 [摘要示例](示例/摘要.md)

### 引言

倒推:

- 我们解决什么技术问题, 为什么这个问题还没有成熟有效的解法
- 技术流程包含哪些贡献, 例如新的有价值的任务, 技术指标, 技术问题或技术
- 这些贡献为什么能够解决目标技术问题
- 这些贡献带来什么技术优势和新认知
- 应该怎样介绍已有方法, 才能自然引出目标技术问题和新认知

正推:

- 介绍任务和应用
- 通过已有方法引出目标技术问题和技术原因
- 介绍解决问题的方法和贡献
- 说明技术优势和新认知
- 用实验概述和贡献列表收束

![引言内容拆解](images/引言内容拆解.png)

```tex
\section{Introduction}
% 任务和应用
% 已有方法的技术问题
% 产生问题的技术原因
% 解决问题的方法和技术流程
% 方法为什么有效以及技术优势
% 实验概述
% 贡献列表
```

#### 通过已有方法引出技术问题

对于已有任务:

1. 本文的技术流程解决了什么技术问题
2. 哪一类近期方法存在这个技术问题
3. 这类近期方法是为了处理哪一类更早方法的什么问题而出现的
4. 更早的方法又是为了处理传统方法的什么问题而出现的

不要先虚构一个简单方案, 再把本文写成对它逐项添加改进, 这种写法虽然容易理解, 但会让方法显得像可以直接想到的补丁组合, 削弱读者对目标技术问题和核心贡献的兴趣

#### 介绍解决问题的方法

对于已有任务:

1. 本文的技术流程解决了什么技术问题
2. 本文的技术贡献是什么
3. 方法本质上为什么能够工作
4. 相对于已有方法具有什么技术优势

对于新任务:

1. 本文的技术流程解决了什么技术问题
2. 本文的技术贡献是什么
3. 方法本质上为什么能够工作

详见 [引言示例](示例/引言.md)

### 相关工作

- 先列出与本文直接竞争或最相关的方法
- 根据研究问题和技术路线划分几个主题
- 每个主题: 任务或技术路线 -> 代表性方法 -> 仍未解决的问题 -> 本文区别

详见 [相关工作示例](示例/相关工作.md)

### 方法

```tex
\section{Method}
% Overview
% Module 1
% Module 2
% Module 3
```

![方法内容拆解](images/方法内容拆解.png)

流程:

1. 画流程图
2. 组织子模块
3. 每个子模块
    - 模块设计: 工作流程
    - 模块动机: 为什么需要
    - 模块优势: 为什么能行
4. 先写模块设计
5. 再写模块动机和模块优势

#### 模块设计

- 描述特定数据结构 / 网络结构 / 方法结构
- 描述数据流和计算流程

#### 模块动机

- 问题驱动的, 先说明问题, 再说明为什么需要这个模块
- 确保讲解设计时, 每件事都能让读者知道为什么需要它

详见 [方法示例](示例/方法.md)

#### 流程图

- 技术流程图需要体现与已有方法的差异!!!
    - 如果流程不新, 应当突出创新的模块
    - 不为了一张图看懂
    - 技术流程图最重要的是负责突出创新, 方法的文字部分负责让读者理解具体做法
- 也可以不画完整的大图, 只画几个模块小图, 但整体视觉效果可能较弱

### 实验

- 怎么证明我们 SOTA -> 对比实验
- 怎么证明我们方法的每个模块都有贡献 -> 消融实验
- 怎么证明我们方法的上限 -> 更具有挑战性的实验

![实验规划与内容拆解](images/实验规划与内容拆解.png)

- 表注与图注要清楚说明实验设置与符号定义
    - 没啥好说的就简要概括结果

建议结构:

```tex
\section{Experiments}
% 实验设置
% 对比实验
% 消融实验
```

对比实验:

- 选择相关且较新的强基线
- 对新任务没有直接基线时, 构造合理的方法变体

消融实验:

- 目的是论证模块有效 + 一定程度量化模块效果
- 使用一个总表展示所有核心贡献 + 重要组成的影响
- 使用若干小表分别验证模块内部的设计选择
    - 包括对超参数的敏感度
    - 输入数据质量的敏感度
    - 放弃某个设计的影响

结果分析:

- 先说明要验证的问题和比较对象
- 再指出关键实验现象, 解释技术原因和能够支持的主张
- 结果存在代价或适用边界时直接说明

详见 [实验示例](示例/实验.md)

#### 图表

- 预告图应该展示整篇论文最有故事, 最值得首先让读者看到的部分
    - 方法创新最突出时, 可以展示核心机制
    - 论文带来了有趣的认知或现象时, 可以展示核心认知
    - 论文偏向工具或系统时, 可以展示完整的 pipeline
    - 实验效果最有吸引力时, 可以直接展示代表性结果
- 单栏的图表, 放在论文的右栏比较好看
    - 因为人的阅读习惯会从左上角找第一行文字

详见 [图表示例](示例/图表.md)

### 结论

- 除了常规结论, 还要写局限性
    - 技术缺陷: 在关键指标上弱于强基线, 或引入不可接受的代价 (不可以写!!!)
    - 范围局限: 受当前任务设置约束, 但在相同设置下仍然具有竞争力
